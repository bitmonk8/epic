# B7: Recovery and fix architecture

## Summary
The five failure-handling layers compose with mostly sound guard rails, but several cross-module interaction patterns allow unexpected amplification of retry work or bypass of intended limits.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Leaf fix loop runs unchecked on fix subtasks created by branch fix loop.** Branch fix subtasks are created with `mark_fix: true` (orchestrator.rs:741), which blocks branch fix loop (orchestrator.rs:500) and recovery (orchestrator.rs:1036) for those subtasks. However, `leaf_fix_loop` (orchestrator.rs:519) has no `is_fix_task` guard. A fix subtask assessed as a leaf gets the full 9-attempt fix loop (3 tiers x 3 retries) before failing, which is then repeated for each of up to 3 branch fix rounds. In a worst-case branch fix loop at a single non-root node: 3 rounds x N fix subtasks x 9 leaf fix attempts = potentially dozens of agent calls with no circuit breaker distinguishing fix-of-fix from a normal leaf fix. The comment at orchestrator.rs:501 says "prevents recursive fix chains" but only for the branch path, not the leaf path. | orchestrator.rs:500-504 (branch guard) vs orchestrator.rs:519 (no guard) | Either skip the leaf fix loop for `is_fix_task` leaves (fail immediately on verification failure) or apply a reduced retry budget (e.g., single tier, no escalation) for fix subtask leaves. |
| 2 | major | **Recovery subtasks get their own full recovery budget, enabling multiplicative depth.** Recovery subtasks are created with `is_fix_task: false` (orchestrator.rs:1149-1155), giving each one its own `recovery_rounds: 0`. A recovery subtask assessed as a branch can itself have children that fail, triggering up to `MAX_RECOVERY_ROUNDS` (2) recovery rounds at that level, whose recovery subtasks can themselves branch and recover. The stated bound is `MAX_DEPTH` (8), but the total recovery work is O(2^depth) in the worst case. At depth 8 with 2 recovery rounds per level, the theoretical maximum is enormous. The comment at line 1151-1152 acknowledges this but the practical bound may be weaker than intended since `MAX_DEPTH` only limits tree depth, not the fan-out of recovery subtasks at each level. | orchestrator.rs:1148-1152 (recovery subtask creation) + orchestrator.rs:1042-1043 (round budget is per-task) | Consider a global recovery budget (e.g., total recovery subtasks created across the entire run), or propagate remaining recovery rounds from parent to child so nested recovery draws from the same pool. |
| 3 | minor | **Scope circuit breaker absent from recovery and branch execution paths.** `check_scope_circuit_breaker` is called inside `leaf_fix_loop` (orchestrator.rs:563) and `branch_fix_loop` (orchestrator.rs:713), but not in `execute_branch` (orchestrator.rs:891), `attempt_recovery` (orchestrator.rs:1025), or the branch child iteration loop (orchestrator.rs:919). Recovery subtasks are executed via `execute_task` which eventually reaches leaf/branch paths that do check scope, but the recovery decision-making itself (assess_recovery, design_recovery_subtasks) proceeds without a scope check. A scope-exceeded condition discovered mid-recovery would not be caught until the recovery subtask itself starts executing. | orchestrator.rs:891-1019 (execute_branch) vs orchestrator.rs:563, 713 (fix loops with scope check) | Add a scope circuit breaker check at the top of `execute_branch`'s outer loop or before `attempt_recovery` creates new subtasks. |
| 4 | minor | **Leaf fix loop and leaf execution retry use separate attempt vectors with independent budgets.** `execute_leaf` tracks retries in `task.attempts` (orchestrator.rs:834) with RETRIES_PER_TIER per model tier. `leaf_fix_loop` tracks retries in `task.fix_attempts` (orchestrator.rs:600) with the same RETRIES_PER_TIER per tier. A leaf task that exhausts all 9 execution retries (3 tiers x 3), then succeeds on the last attempt, fails verification, and enters the fix loop gets another 9 fix attempts -- 18 total agent calls for a single leaf. This is by design per the `AgentService` trait having separate `execute_leaf` and `fix_leaf` methods, but the combined budget (up to 18 calls per leaf) is not documented or constrained as a unit, and interacts with the branch fix loop to allow even more if the parent retries. | orchestrator.rs:776-887 (execute_leaf) + orchestrator.rs:519-671 (leaf_fix_loop) + agent/mod.rs:48-52, 74-80 (separate trait methods) | Document the maximum combined leaf budget explicitly. Consider whether the fix loop should inherit the model tier reached during execution rather than restarting from the assessed model. |
| 5 | note | **Recovery round counter lives on the parent, not the failing child.** `attempt_recovery` increments `recovery_rounds` on the parent task (orchestrator.rs:1085), meaning that if two different children fail, they consume from the same parent's budget of 2 rounds. This is a reasonable design choice but means the first child failure to trigger recovery effectively reduces the budget available for handling subsequent child failures. Combined with the fact that recovery subtasks are NOT fix tasks and get their own budgets, there is an asymmetry: recovery of child failures is tightly budgeted at the parent level (2 rounds total) but unbounded in depth once recovery subtasks begin executing. | orchestrator.rs:1042-1051 (parent budget) + orchestrator.rs:1148-1152 (recovery children get fresh budgets) | This is an intentional design tradeoff. Just noting the asymmetry for awareness. |

## Notes
- The `is_fix_task` flag is the primary mechanism preventing recursive fix/recovery chains. It is correctly applied to branch fix subtasks (`mark_fix: true`) and correctly omitted from recovery subtasks (`mark_fix: false`). The gap is that leaf fix loops do not consult this flag at all (finding #1).
- The five layers in order: (1) `execute_leaf` retry+escalation, (2) `leaf_fix_loop` retry+escalation after verification failure, (3) `branch_fix_loop` creating fix subtasks, (4) `attempt_recovery` creating recovery subtasks, (5) checkpoint escalation triggering recovery. A single task can traverse layers 1-2 (as a leaf) or 3-5 (as a branch parent), but not all five on the same task. However, a branch's fix subtask (layer 3) that is assessed as a leaf will go through layers 1-2, and that branch itself may later trigger layer 4-5. So all five layers can fire in one run, just on different nodes of the task tree.
- No deadlock or infinite recursion risk was identified: `MAX_DEPTH` prevents unbounded tree growth, `is_fix_task` blocks branch fix and recovery for fix subtasks, and `MAX_RECOVERY_ROUNDS` caps recovery per parent. The concern is combinatorial cost amplification rather than termination failure.
