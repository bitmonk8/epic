# B6: Agent contract coherence

## Summary
The AgentService trait is largely coherent, with a clear pattern of prompt-build / config-build / run / wire-convert per method. However, cross-module examination reveals several inconsistencies in how method pairs share (or diverge in) their config-gen and signature conventions.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `execute_leaf` and `fix_leaf` have nearly identical implementation bodies (same config builder `build_execute_leaf_config`, same wire type `TaskOutcomeWire`, same `ToolGrant`, same `run_with_tools` path) but are separate trait methods. The only difference is the prompt and extra parameters (`failure_reason`, `attempt`). Similarly, `design_and_decompose` and `design_fix_subtasks` share `build_decompose_config`, `DecompositionWire`, and `run_with_tools` — they differ only in prompt and extra parameters (`verification_issues`, `round`). This suggests the trait surface could be reduced by parameterising these pairs, e.g. a single `execute_leaf(ctx, model, fix_context: Option<FixContext>)` and `decompose(ctx, model, fix_context: Option<BranchFixContext>)`. | `mod.rs` lines 48-52 vs 73-80; `flick.rs` lines 208-248 vs 250-289 | Consider unifying each pair behind a single trait method that accepts an optional "retry/fix context" struct. This halves the trait surface for leaf-execute and decompose paths, and removes the implicit coupling where the two methods must stay in sync on config builder + wire type. |
| 2 | minor | `assess` in `mod.rs` takes only `&TaskContext` (no `Model` param) — the implementation in `flick.rs` hardcodes `Model::Sonnet`. `design_and_decompose` similarly hardcodes `Model::Sonnet`, and `verify` also hardcodes `Model::Sonnet`. Meanwhile `checkpoint` hardcodes `Model::Haiku` and `design_recovery_subtasks` hardcodes `Model::Opus`. The model tier is an invisible implementation detail baked into `flick.rs`, not visible or overridable through the trait contract. Any alternative `AgentService` implementation must independently rediscover or re-hardcode these model choices with no trait-level guidance. | `mod.rs` (trait signatures) vs `flick.rs` lines 194-206, 250-267, 291-305, 307-321, 339-360 | Either (a) add a `model` parameter to the methods that currently hardcode it, making the contract explicit, or (b) document the expected model tier per method in the trait doc-comments so alternative implementations have clear guidance. |
| 3 | minor | `assess_recovery` in `mod.rs` returns `Option<String>` (raw strategy text), while every other trait method returns a domain-typed result (`AssessmentResult`, `DecompositionResult`, `VerificationResult`, etc.). In `config_gen.rs`, the `RecoveryWire` type exists but its conversion is a bare `into_strategy()` method rather than a `TryFrom` impl like every other wire type. This breaks the otherwise consistent wire-type-to-domain-type pattern. | `mod.rs` line 96; `config_gen.rs` lines 164-174 vs all other `TryFrom` impls | Introduce a `RecoveryAssessment { recoverable: bool, strategy: Option<String> }` domain type and a `TryFrom<RecoveryWire>` impl to match the pattern used by every other method. |
| 4 | note | `build_checkpoint_config` and `build_recovery_config` in `config_gen.rs` take no `ToolGrant` parameter (tool-less structured calls), while all other `build_*_config` functions do. This is correct behavior (these methods use `run_structured` not `run_with_tools`), but the two calling conventions (tools vs no-tools) are not reflected in any shared type or naming convention — callers must know which config builder pairs with which runner. | `config_gen.rs` lines 419-435 vs `flick.rs` lines 307-321, 323-337 | Consider a naming or type-level distinction (e.g., `build_*_structured_config` vs `build_*_tools_config`) so the pairing with `run_structured` vs `run_with_tools` is self-documenting. |

## Notes
The trait has 9 methods (counting the trait definition). The overall design is well-structured: every method follows a `prompts::build_* -> config_gen::build_*_config -> run_{structured,with_tools} -> Wire::try_from` pipeline. The issues above are cross-cutting concerns visible only when examining how `mod.rs`, `flick.rs`, `config_gen.rs`, and `prompts.rs` interact. No single-file bugs were flagged per the audit scope.
