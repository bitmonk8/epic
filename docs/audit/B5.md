# B5: Orchestrator responsibility boundaries

## Summary

The orchestrator (~1,170 lines of non-test implementation) owns task execution, state mutation, checkpointing, fix loops, recovery, event emission, and agent dispatch. Most of these are inherent to orchestration. Three cross-module issues emerge where responsibility placement creates duplication or hidden coupling.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Retry/escalation logic duplicated across `execute_leaf` and `leaf_fix_loop` with divergent semantics.** Both methods independently implement the same tier-counting and escalation pattern (count trailing attempts at current tier, drain stale exhaustion, loop with `RETRIES_PER_TIER`, call `Model::escalate`). The retry policy depends on `Task.attempts` vs `Task.fix_attempts` and `Task.current_model` -- fields owned by `task::Task` -- but the counting/escalation logic lives entirely in the orchestrator. This means `Task` exposes two parallel attempt vectors (`attempts`, `fix_attempts`) with no method to query "retries remaining at current tier", forcing the orchestrator to re-derive this from raw data in two places. If either the tier constants or the counting convention changes, both call sites must be updated in lockstep. | `orchestrator.rs` lines 776-888 (`execute_leaf`) and 519-671 (`leaf_fix_loop`); `task/mod.rs` `Task` struct fields `attempts`, `fix_attempts`, `current_model` | Add a method on `Task` (or a small helper type) that encapsulates "retries consumed at the current tier" and "next tier or exhausted" given an attempt slice and a model. Both orchestrator loops would call this instead of re-implementing the counting. This keeps retry *policy* (constants, loop structure) in the orchestrator while moving retry *accounting* closer to the data it reads. |
| 2 | minor | **`build_context` assembles `TaskContext` by walking `EpicState` internals, but `TaskContext` is defined in `agent::mod.rs`.** The `TaskContext` struct (parent goal, ancestor chain, sibling summaries, checkpoint guidance) is the agent module's input contract, yet the assembly logic that populates it lives in the orchestrator and reaches directly into `EpicState` via `get()` calls on parent/sibling IDs. Meanwhile `EpicState` (in `state.rs`) already has tree-walking capability (`dfs_order`). If a new agent call needed additional tree context (e.g., cousin tasks, depth-aggregated discoveries), the orchestrator -- not the state or agent module -- would need to change. This splits context-assembly knowledge across the orchestrator and the `TaskContext` definition. | `orchestrator.rs` lines 240-324 (`build_context`); `agent/mod.rs` lines 26-34 (`TaskContext`); `state.rs` `EpicState` | Move context assembly to `EpicState` as `fn build_task_context(&self, id: TaskId) -> TaskContext` (or a standalone function that takes `&EpicState`). The orchestrator would still call it, but the tree-walking detail would live next to the tree data. |
| 3 | minor | **`evaluate_scope` parses `git diff --numstat` output and compares against `Magnitude`, but `Magnitude` is defined in `task/mod.rs` with no connection to scope-checking.** The `Magnitude` struct carries `max_lines_added/modified/deleted` and is populated by the agent during assessment, stored on `Task`, then consumed exclusively by the free function `evaluate_scope` in `orchestrator.rs`. The multiplier (`3`) that converts magnitude limits into circuit-breaker thresholds is an orchestrator constant invisible to anyone reading `Magnitude`. This creates a hidden contract: `Magnitude` fields look like absolute limits but are silently tripled. | `orchestrator.rs` lines 35-78 (`evaluate_scope`, `ScopeCheck`); `task/mod.rs` lines 68-74 (`Magnitude`) | Either document the multiplier contract on `Magnitude` itself, or add an `fn exceeds(&self, added: u64, modified: u64, deleted: u64) -> Option<&str>` method on `Magnitude` that encodes the threshold policy. This keeps the "what counts as exceeded" logic next to the data definition and makes the 3x multiplier explicit. |

## Notes

- The orchestrator's core responsibilities -- DFS task execution, phase transitions, agent dispatch sequencing, checkpoint timing, recovery/fix loop control flow -- are genuinely orchestration concerns. The fix/recovery loop *structure* (round budgets, when to escalate vs. fail) is inherently cross-cutting and belongs in the orchestrator.
- Finding 1 is the most actionable: the two retry loops are already a maintenance burden visible in the `#[allow(clippy::too_many_lines)]` annotations on both methods, and the duplication is only detectable by reading both methods side by side.
- `EpicState` is intentionally thin (a `HashMap` wrapper with persistence). The audit did not find evidence that it should absorb orchestration logic beyond context assembly (finding 2).
- The test suite (3,300+ lines) mirrors the implementation structure faithfully; the mock `AgentService` has 10 response queues corresponding to 10 trait methods, which is proportional to the trait surface and not itself a problem.
