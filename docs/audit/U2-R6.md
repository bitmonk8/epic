# U2-R6: agent/flick / Testability

## Summary

The helper functions (`check_error`, `extract_text`, `extract_tool_calls`) have solid unit test coverage, but the core async methods (`run_structured`, `run_with_tools`) and the entire `AgentService` implementation are untestable in isolation because `FlickAgent` directly constructs a `flick::FlickClient` with no seam for injecting a fake provider or client.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `build_client` hard-codes `flick::resolve_provider` and `flick::FlickClient::new` with no injection point -- impossible to unit-test `run_structured` or `run_with_tools` without a live provider | `flick.rs:47-55` | Accept a provider factory trait/closure in `FlickAgent`, or extract a `trait ClientBuilder` that can be replaced in tests with a fake returning canned `FlickResult` values |
| 2 | major | `run_with_tools` tool-loop logic (round counting, break-on-complete, bail-on-exceed) is only exercisable through a real Flick round-trip; no unit test covers the loop or the `MAX_TOOL_ROUNDS` bail path | `flick.rs:88-147` | Refactor the loop to operate on an abstract `trait AgentClient { async fn run(...); async fn resume(...); }` so tests can inject a mock that returns `ToolCallsPending` N times then `Complete` |
| 3 | major | `tools::execute_tool` is called directly inside the tool loop with no indirection -- test isolation from real filesystem side-effects (file reads, bash execution) is impossible | `flick.rs:114-121` | Inject tool execution via a trait or closure (e.g., `ToolExecutor`) so tests can supply a stub returning fixed `ToolResult` values |
| 4 | minor | No test covers the `tokio::time::timeout` wrapping in `run_structured` or `run_with_tools`; timeout behaviour is unverified | `flick.rs:69-75`, `flick.rs:98-104` | With a mock client, add a test where the client future sleeps past `call_timeout` to confirm the timeout error message is produced |
| 5 | minor | `MAX_TOOL_ROUNDS` is a module-level `const` with no override mechanism; testing boundary behaviour (e.g., exactly 50 rounds) requires either the mock-client refactor or modifying the constant | `flick.rs:21` | Make the limit configurable on `FlickAgent` (with a default) so tests can set it to a small value like 2 |
| 6 | minor | `AgentService` methods in `FlickAgent` are thin wrappers (prompt + config + run), but none have any unit-level test -- all coverage relies on the orchestrator-level `MockAgentService` which bypasses this module entirely | `flick.rs:194-360` | After introducing a mock client boundary, add at least one per-method test that verifies prompt assembly feeds through to the client and the wire-to-domain conversion is applied |
| 7 | note | The existing tests (lines 409-533) cover the three free-standing helpers thoroughly -- `extract_text`, `extract_tool_calls`, `check_error` -- which is good. No gaps there. | `flick.rs:409-533` | None needed |
| 8 | note | `explore_for_init` lives outside the `AgentService` trait and has no mock path in the orchestrator tests; it is completely untested | `flick.rs:156-187` | Once a client abstraction exists, add a dedicated test for the init-exploration path |

## Notes

- The orchestrator test suite (`src/orchestrator.rs:1171+`) already defines a `MockAgentService` that replaces the entire `FlickAgent` at the trait boundary. This validates orchestration logic but leaves all code inside `flick.rs` -- client construction, timeout handling, tool looping, wire deserialization -- completely untested.
- The most impactful single change would be extracting `build_client` into a trait (or making `FlickAgent` generic over a client factory). This would unlock unit tests for both `run_structured` and `run_with_tools` without network access, and would naturally enable testing the timeout, round-limit, and tool-dispatch paths.
- No integration test directory (`tests/`) exists in the project, so there is no secondary coverage path for this module.
