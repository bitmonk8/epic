# U4-R7: agent/prompts / Design intent

## Summary

The prompt templates faithfully implement the core assess/execute/verify/checkpoint/recovery lifecycle from EPIC_DESIGN2 and carry rich structural context (siblings, ancestors, checkpoint guidance). However, several design-doc requirements are absent from prompt text: cost-control guardrails, verification-model policy, tie-breaking bias, depth-cap enforcement cues, and the decomposition-rationale context channel.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **No cost/scope guardrails in any prompt.** TASK_MODEL specifies a 3x magnitude circuit breaker and retry budgets. No prompt mentions token budgets, magnitude limits, or any cost-awareness language. Agents have no incentive to stay within scope. | `prompts.rs` (all builders) | Add a cost-awareness section to `format_context` that includes the task's magnitude estimate and remaining retry/escalation budget so the agent can self-regulate. |
| 2 | major | **Assessment prompt omits tie-breaking bias.** TASK_MODEL states "Tie-breaking bias: branch. Recovery from wrong-branch is cheaper than wrong-leaf." The assess prompt (lines 80-93) gives no such guidance; the agent may default to leaf when uncertain, which the design explicitly discourages. | `prompts.rs:80-93` | Add a guideline: "When uncertain between leaf and branch, prefer branch -- recovery from an incorrect branch decision is cheaper than from an incorrect leaf." |
| 3 | major | **Assessment prompt omits root-is-always-branch rule.** TASK_MODEL: "Root task is always forced to branch." The prompt gives the agent full discretion even at depth 0. If this is enforced only in orchestrator code it is fine, but the prompt should not contradict or omit it since a model could return leaf for root. | `prompts.rs:80-93` | Either inject a hard constraint ("depth 0 tasks MUST be branch") into the assess prompt when depth == 0, or document that the orchestrator overrides assessment for root. |
| 4 | major | **Verification prompt has no model-policy awareness.** AGENT_DESIGN specifies leaf verification model = max(Haiku, implementing model) capped at Sonnet, branch verification = Sonnet. The verify prompt (lines 231-257) is model-agnostic and identical for both leaf and branch paths, with no distinction in scope or expectations. | `prompts.rs:231-257` | Split into `build_verify_leaf` and `build_verify_branch` with differentiated instructions (branch verifier should check cross-subtask integration and aggregate simplification per TASK_MODEL). |
| 5 | minor | **Decomposition rationale not injected into context.** TASK_MODEL's structural-map spec says parent context should include "goal, decomposition rationale, discoveries." `format_context` includes parent goal but not `decomposition_rationale`, so child agents lack the strategic reasoning behind their existence. | `prompts.rs:12-77` | Add `decomposition_rationale: Option<String>` to `TaskContext` and render it alongside the parent goal in `format_context`. |
| 6 | minor | **Depth-cap not surfaced in prompts.** TASK_MODEL mandates depth 8 cap with forced-leaf at max depth. Depth is included numerically but no prompt warns the agent of the cap or that it should avoid proposing further decomposition near the limit. | `prompts.rs:69-77` | In `build_design_and_decompose`, when depth is near the cap (e.g., >= 6), inject a warning that subtasks at max depth will be forced to leaf. |
| 7 | minor | **Leaf executor prompt does not mention Research Service.** AGENT_DESIGN and TASK_MODEL both specify that Research Service is available as a tool during implementation and design+decompose phases. The execute-leaf prompt (lines 108-134) and decompose prompt (lines 136-163) do not mention it, which may cause agents to under-utilize the service. | `prompts.rs:108-134` | Add a guideline noting that Research Service / DocumentStore is available for querying prior findings and storing new analysis. |
| 8 | minor | **No simplification/review phase prompt.** TASK_MODEL leaf path includes file-level review and local simplification review steps (steps 3-4). No corresponding prompt builder exists, suggesting these phases are unimplemented or handled elsewhere. | `prompts.rs` (absent) | Either add `build_file_review` and `build_simplification_review` prompt builders, or document that these are deferred to a later milestone. |
| 9 | note | **Fix-leaf prompt says "passed but failed verification" which is slightly misleading.** The prompt at line 169 says "A previous execution of this task passed but failed verification" -- the execution did not "pass"; it completed but did not meet verification criteria. Minor wording issue. | `prompts.rs:169` | Reword to "A previous execution of this task completed but failed independent verification." |
| 10 | note | **Recovery assessment prompt is sparse.** `build_assess_recovery` (lines 336-356) has a minimal system prompt compared to the design's specification that this should be an Opus-level call requiring strongest reasoning. The prompt gives little strategic guidance on what makes a failure recoverable vs terminal. | `prompts.rs:336-356` | Expand the system prompt with guidelines on evaluating recoverability: partial progress salvageability, whether the failure is environmental vs logical, retry budget remaining, etc. |

## Notes

- The prompt assembly correctly implements the structural-map injection pattern (own task, parent, ancestors, siblings) specified in TASK_MODEL's Context Propagation section, minus the decomposition rationale gap noted in finding #5.
- Model selection and tool scoping happen outside `prompts.rs` (in the orchestrator and agent service layer), so the prompts themselves are not responsible for enforcing model policy. However, the verify prompt should still differentiate leaf vs branch expectations (finding #4).
- The checkpoint and recovery prompts are well-aligned with the design's discovery flow and recovery machinery.
- Test coverage is adequate for structural correctness but does not assert on design-intent properties (e.g., that certain keywords like "branch bias" appear).
