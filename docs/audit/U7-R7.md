# U7-R7: task / Design intent

## Summary

The task module types faithfully model the EPIC_DESIGN2 lifecycle (assess/execute/verify) with proper cost-control structures and verification enforcement. However, the orchestrator deviates from the design document on the assessment model tier and lacks enforcement of the 2-5 subtask count bound, and the file-level and simplification review steps described in the design are absent from the implementation.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Assessment uses Sonnet instead of Haiku. TASK_MODEL.md specifies "Single Haiku call" for assessment, but `FlickAgent::assess` hardcodes `Model::Sonnet`. This is a meaningful cost increase since assessment runs for every non-root, non-max-depth task. | `/opt/UnitySrc/epic/src/agent/flick.rs:199` | Change to `Model::Haiku` or document the deliberate deviation. Consider conditional escalation to Sonnet as the design mentions. |
| 2 | major | Recovery assessment uses Sonnet instead of Opus. TASK_MODEL.md specifies "Parent Opus recovery assessment" and the `RecoveryPlan` doc-comment says "Opus recovery agent", but `FlickAgent::assess_recovery` uses `Model::Sonnet`. The subsequent `design_recovery_subtasks` correctly uses Opus. | `/opt/UnitySrc/epic/src/agent/flick.rs:331` | Change to `Model::Opus` to match design intent, or split into a lighter triage (Sonnet) followed by Opus planning, and document. |
| 3 | major | File-level review and simplification review are not implemented. TASK_MODEL.md Leaf Path steps 3-4 describe a file-level review (capped at Sonnet) and a local simplification review after verification gates. Neither the task types nor the orchestrator implement these review stages. | `/opt/UnitySrc/epic/src/task/verify.rs`, `/opt/UnitySrc/epic/src/orchestrator.rs` (leaf path) | Add review sub-phases to the leaf path, or document deferral as a deliberate scope cut with a tracking item. |
| 4 | minor | No enforcement of 2-5 subtask count bound. TASK_MODEL.md says "2-5 subtasks" for decomposition. The prompt in `build_decompose` mentions "Aim for 2-5 subtasks" but no runtime validation rejects decompositions outside this range. An agent could return 1 or 20 subtasks. | `/opt/UnitySrc/epic/src/orchestrator.rs` (`execute_branch`, `create_subtasks`) | Add a bounds check on `decomposition.subtasks.len()` with a configurable range, rejecting or warning on out-of-range counts. |
| 5 | minor | `AssessmentResult` does not capture `MagnitudeEstimate`. The design specifies magnitude estimation during assessment, but `AssessmentResult` carries only `Magnitude` (absolute line counts). The `MagnitudeEstimate` enum (Small/Medium/Large) is only set on child tasks via `SubtaskSpec` from the parent's decomposition. Assessment-produced leaves have no estimate, so scope circuit breaker only fires if the agent returns magnitude line counts (which is optional). | `/opt/UnitySrc/epic/src/task/assess.rs:8-12` | Consider requiring magnitude (or at least an estimate) from assessment for leaves, so the scope circuit breaker is reliably armed. |
| 6 | minor | Depth-cap forced leaf hardcodes `Model::Sonnet` without assessment. When `depth >= MAX_DEPTH`, the task is forced to leaf with `Model::Sonnet` bypassing the agent entirely. The design says depth-capped tasks are "forced to leaf path" but does not specify skipping model selection. A simple task at max depth still pays Sonnet cost. | `/opt/UnitySrc/epic/src/orchestrator.rs:426-432` | Consider using the parent's magnitude estimate to inform model selection, or default to Haiku for small estimates. |
| 7 | note | Verification is correctly mandatory. `finalize_task` always runs verification for successful outcomes and enters the appropriate fix loop on failure. Verification cannot be bypassed -- this is a strength of the design. | `/opt/UnitySrc/epic/src/orchestrator.rs:471-516` | No change needed. |
| 8 | note | Cost control mechanisms are well-implemented. The 3-retries-per-tier escalation, scope circuit breaker (3x magnitude), MAX_RECOVERY_ROUNDS=2, and branch fix round budget (3 Sonnet + 1 Opus for root) all match design intent and are crash-safe via checkpoint persistence. | `/opt/UnitySrc/epic/src/orchestrator.rs` (constants, `execute_leaf`, `leaf_fix_loop`, `branch_fix_loop`) | No change needed. |
| 9 | note | Checkpoint classification (proceed/adjust/escalate) is correctly modeled and uses Haiku as specified. | `/opt/UnitySrc/epic/src/task/branch.rs:19-24`, `/opt/UnitySrc/epic/src/agent/flick.rs:315` | No change needed. |

## Notes

- The task module files themselves (`mod.rs`, `assess.rs`, `leaf.rs`, `branch.rs`, `verify.rs`) are pure data definitions with no logic -- all behavioral concerns live in the orchestrator and agent layers. This is a clean separation but means the task types cannot self-enforce invariants (e.g., subtask count bounds, mandatory magnitude on leaves).
- The design document references EPIC_DESIGN2 as the source of truth. Findings #1 and #2 may reflect deliberate implementation decisions that were not back-ported to the design doc. If so, the docs should be updated to reflect the actual model tier choices and their rationale.
- The recovery flow is split into two agent calls (assess_recovery at Sonnet, design_recovery_subtasks at Opus). This is a reasonable cost optimization -- cheap triage before expensive planning -- but it deviates from the design's single "Opus recovery assessment" description.
