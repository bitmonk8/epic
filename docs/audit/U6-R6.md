# U6-R6: agent/models / Testability

## Summary

The `AgentService` trait provides a clean mock boundary and the existing `MockAgentService` in `orchestrator.rs` demonstrates it works well for integration-level tests. However, the mock is private to `orchestrator::tests`, the trait uses RPITIT (return-position `impl Trait`) which prevents `dyn` dispatch and complicates mock reuse, and `FlickAgent` has no unit-level testing of its `AgentService` methods because the Flick client dependency is not injectable.

## Findings

## Notes

- The trait design (generic/monomorphic, RPITIT) is a reasonable Rust-idiomatic choice that avoids `async_trait` overhead. The trade-off is that `mockall` cannot auto-generate mocks for it, so the hand-written mock must be maintained manually as the trait evolves.
- The mock's queue-based approach (push expected responses, pop on call) is simple and effective for linear orchestrator flows but becomes fragile for tests with branching or conditional agent calls -- a callback-based or scenario-based mock would be more robust for complex multi-child test cases.
- `TaskContext` and `SiblingSummary` are plain data structs with `Clone + Debug`, which is ideal for test construction. No builder or factory is needed.
