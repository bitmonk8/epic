# U3-R6: agent/config_gen / Testability

## Summary

The wire-type conversions are well-tested with good coverage of happy paths and error branches. The main testability concern is the `build_config` function, which couples JSON assembly to `flick::Config::from_str` parsing with no seam between them, making it impossible to verify the generated JSON independently of the external crate.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `build_config` is a monolith that serializes JSON and immediately parses it through `flick::Config::from_str`. Tests cannot verify the intermediate JSON structure without calling the real `flick` parser, coupling unit tests to an external crate. | `config_gen.rs:328-373` | Split into a pure `build_config_json(...)-> JsonValue` that tests can inspect, and a thin `parse_flick_config(json)-> flick::Config` wrapper. |
| 2 | minor | `build_init_config` hard-codes `Model::Sonnet` (line 529), but no test covers it, so a future change to the model constant could silently break init without any test feedback. | `config_gen.rs:522-530` | Add a test for `build_init_config` that asserts the expected model string appears in the generated config. |
| 3 | minor | `DetectedStepWire::from` conversion (lines 461-469) has zero test coverage. Its default timeout of 300 and the field mapping are untested. | `config_gen.rs:461-469` | Add roundtrip tests for `DetectedStepWire` into `VerificationStep`, including the default-timeout branch. |
| 4 | minor | `InitFindingsWire` is never deserialized or tested anywhere in this file despite being a public type used in the init flow. | `config_gen.rs:472-477` | Add a `serde_json::from_value` roundtrip test to ensure the schema and the struct stay in sync. |
| 5 | minor | Schema functions (`assessment_schema`, etc.) are only tested indirectly via `build_*` calls. If a schema drifts out of sync with its wire type (e.g., a new required field added to the struct but not the schema), no test will catch it. | `config_gen.rs:208-321` | Add property tests or at minimum a test per schema that serializes a valid wire struct, validates it against the schema, and confirms round-trip deserialization. |
| 6 | note | The existing `config_builds_with_correct_content` test (lines 685-731) manually reconstructs the expected JSON rather than calling `build_config` and inspecting its output, so it does not actually assert that `build_config` produces the expected structure -- it only proves `flick` can parse *something*. | `config_gen.rs:685-731` | Refactor after splitting `build_config_json` (see finding 1) so the test directly asserts on the builder's output. |
| 7 | note | All `build_*_config` public functions share an identical pattern (get tools, get schema, delegate). There is no negative test for what happens when `flick::Config::from_str` rejects malformed input, so error-path behaviour is unverified. | `config_gen.rs:375-446` | Add at least one test that forces a parse failure (e.g., inject an invalid model provider value) and asserts the error message. |

## Notes

- The wire-type `TryFrom` impls are the strongest part of the test suite: every variant and most edge cases (partial magnitude, missing reason, empty discoveries) are covered.
- The `flick::Config` dependency is opaque from this module's perspective. Because `build_config` returns the parsed config directly, tests have no way to assert on the JSON that was actually sent to `flick`, making it hard to catch regressions in config shape without integration tests.
- `parse_model_name` is tested indirectly through `AssessmentWire` conversion but has no direct unit test for the error branch (invalid model name). This is low risk since the `assessment_wire_invalid_path` test covers the analogous pattern.
