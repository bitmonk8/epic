# U13-R6: tui / Testability

## Summary

The TUI module has zero test coverage and tightly couples application state logic with terminal I/O, making the core event-handling and rendering logic difficult to test in isolation. Pure-logic helpers like `render_tree_lines` are well-separated but untested, while `TuiApp` bundles state mutation, time, and terminal concerns into a single monolith.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Zero test coverage across all four TUI files -- no `#[cfg(test)]` modules exist anywhere in the `tui/` directory. | `src/tui/**` | Add unit tests for at least `handle_event`, `render_tree_lines`, metrics counting, and worklog entry construction. |
| 2 | major | `TuiApp::handle_event` is a private method on a struct whose only public entry point (`run`) requires a real terminal. Event-handling logic cannot be exercised without standing up crossterm/ratatui. | `src/tui/mod.rs:132` | Make `handle_event` `pub(crate)` (or extract it to a free function / separate state struct) so tests can call it directly with synthetic `Event` values. |
| 3 | major | `TuiApp::event_loop` is hardwired to `CrosstermBackend<io::Stdout>` and `ct_event::poll`/`ct_event::read`. There is no trait boundary or injection point for a test backend or simulated keyboard input. | `src/tui/mod.rs:83-129` | Accept the terminal/backend as a generic parameter (`Terminal<B: Backend>`) and inject a keyboard-event source trait, or split the loop into testable tick/step functions. |
| 4 | minor | `WorklogEntry` constructors call `Instant::now()` internally with no way to inject a clock. Tests cannot assert on deterministic timestamps or elapsed-time formatting. | `src/tui/worklog.rs:16-46` | Accept a `now: Instant` parameter (or a clock trait) instead of calling `Instant::now()` inside the constructors. |
| 5 | minor | `session_start: Instant` is set at construction time with no override. Tests that exercise time-dependent header rendering or worklog formatting must cope with wall-clock non-determinism. | `src/tui/mod.rs:49` | Allow `session_start` to be injected (e.g., a `new_with_start` constructor) for deterministic tests. |
| 6 | minor | `render_tree_lines` is a well-isolated pure function suitable for unit testing, but has no tests for edge cases: empty tree, deep nesting, Unicode truncation boundary, zero-width area. | `src/tui/task_tree.rs:41-102` | Add targeted unit tests covering at least the empty-root, single-task, multi-level, and Unicode-boundary cases. |
| 7 | minor | `MetricsWidget::render` re-derives counts by iterating the full task map each frame. The counting logic is embedded inside a `Widget::render` impl that requires a `Buffer`, making it hard to test the counting in isolation. | `src/tui/metrics.rs:16-89` | Extract phase-counting into a separate pure function (e.g., `count_by_phase(&HashMap<TaskId, TuiTask>) -> PhaseCounts`) that can be unit-tested independently. |
| 8 | note | `TuiApp` state transitions (phase changes, current-task tracking, worklog cap eviction) form an implicit state machine. There is no documentation or test harness that verifies valid transition sequences or invariants (e.g., `current_task` always points to an active-phase task). | `src/tui/mod.rs:132-349` | Consider adding property-based or sequence-driven tests that feed randomized event streams and assert structural invariants on the resulting state. |

## Notes

- The widget types (`TaskTreeWidget`, `WorklogWidget`, `MetricsWidget`) follow ratatui's `Widget` trait, which takes `&mut Buffer`. Ratatui provides `Buffer::empty(Rect)` for test rendering, so widget-level tests are feasible today without a real terminal -- they just haven't been written.
- `render_tree_lines` is the most immediately testable function in the module since it is public, pure, and returns structured data. It should be the first target for coverage.
- The `handle_event` method is the highest-value test target: it contains all state-mutation logic and could be tested with minimal refactoring (just a visibility change).
