# U5-R3: agent/tools / Error handling

## Summary

Error handling is generally well-structured, with most I/O and parsing operations using `map_err` to produce descriptive `Result<_, String>` errors. However, there are a few cases where errors are silently swallowed, and one `unwrap_or` masks a conversion that cannot practically fail but deserves a comment.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `tool_bash` returns `Ok(...)` even when the command exits with a non-zero status (the exit code is embedded in the content string). Callers that check only `is_error` will treat a failing command as successful. Whether this is intentional is unclear, but the test on line 929 acknowledges it with a comment. | `tools.rs:610-621` | Document the design decision explicitly, or return `is_error: true` for non-zero exits so callers can distinguish success from failure without parsing the content string. |
| 2 | minor | `tool_glob` silently ignores all `WalkDir` entry errors via `.filter_map(Result::ok)`. Permission-denied or broken-symlink errors are dropped without any indication to the user, which can make debugging confusing when expected files are missing from results. | `tools.rs:371` | Collect a count of skipped errors and append a note (e.g., "[N entries skipped due to errors]") to the output, similar to how truncation is reported. |
| 3 | minor | `tool_grep` also silently ignores `WalkDir` entry errors and silently skips unreadable/binary files without any summary. A directory with permission issues will produce empty grep output with no explanation. | `tools.rs:421-422, 451-453` | Same as finding #3: report a count of skipped entries so the agent can reason about incomplete results. |
| 4 | minor | `tool_read_file` truncation reads exactly `MAX_READ_BYTES` via `read_exact` into a `Vec<u8>` and then uses `from_utf8_lossy`. If the cut lands in the middle of a multi-byte UTF-8 sequence, the last character will be replaced with the Unicode replacement character. This is not incorrect but could confuse an agent reading code. | `tools.rs:338-342` | After reading, back up to the last valid UTF-8 char boundary (similar to the bash truncation logic on lines 580-583) before converting. |
| 5 | minor | `output.status.code().unwrap_or(-1)` silently maps signal-killed processes (which return `None` on Unix) to exit code -1. The agent loses information about which signal killed the process (e.g., SIGSEGV vs SIGKILL). | `tools.rs:596` | On Unix, check `output.status.signal()` and include the signal number in the output (e.g., "[killed by signal 9]"). |
| 6 | note | All tool functions return `Result<String, String>` rather than typed errors. This is adequate for the current use case (errors are forwarded as text to the LLM), but makes programmatic error handling or retry logic harder to build on top of this layer. | `tools.rs:319,353,395,482,519,551` | Consider introducing a `ToolError` enum if retry or error-category logic is needed in the future. |
| 7 | note | `try_from(MAX_READ_BYTES).unwrap_or(usize::MAX)` on line 334 is safe in practice (256 KiB always fits in `usize`), but the fallback to `usize::MAX` would cause an enormous allocation if it somehow triggered. | `tools.rs:334` | Use `.expect("MAX_READ_BYTES fits in usize")` to make the invariant explicit, or use a `const` assertion. |

## Notes

- No `unwrap()` or `expect()` calls were found in non-test production code. The codebase consistently uses `map_err` for error propagation, which is good practice.
- The `safe_path` function and `verify_ancestors_within_root` helper provide solid path-traversal protection with proper error returns.
- Test code uses `unwrap()` freely, which is acceptable for tests.
