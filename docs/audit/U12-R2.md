# U12-R2: cli + main / Security

## Summary
The `cli.rs` and `main.rs` files are relatively thin entry-point layers with limited direct security surface. The primary concerns relate to TOCTOU races on state-file existence checks and the `.epic` working directory being created with default (umask-dependent) permissions, which could expose task state to other local users.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | TOCTOU on state file existence: `state_path.exists()` is checked before `EpicState::load()` on lines 65 and 121. A symlink or file could be swapped between the check and the load, potentially causing the program to load attacker-controlled JSON. | `src/main.rs:65`, `src/main.rs:121` | Use an open-or-create pattern with file locking (`flock`/`fcntl`) instead of exists-then-load. Alternatively, always attempt to load and handle `NotFound` distinctly. |
| 2 | minor | TOCTOU on `epic.toml` existence: `config_path.exists()` is checked on line 41 then trusted, but the file could appear or disappear between the check and later use by `init::run_init`. | `src/main.rs:41` | This is low-risk since it is a guard for the `Init` subcommand, but using `OpenOptions` with `create_new` would eliminate the race. |
| 3 | minor | `.epic/` directory created with default permissions via `create_dir_all`. On multi-user systems the state file (containing task goals, phase data, and potentially sensitive project context) is readable by other users unless the process umask is restrictive. | `src/main.rs:50` | Set restrictive permissions (0700) on the `.epic` directory after creation, or use platform-specific APIs to create it with explicit mode. |
| 4 | note | Credential name visible in process arguments: `--credential <value>` appears in `/proc/PID/cmdline`. While this is a credential *name* (not a secret), on shared systems it reveals which credential store entry is in use. | `src/cli.rs:10` | The `env = "EPIC_CREDENTIAL"` alternative already exists, which is slightly better. Document that the env-var form is preferred on shared systems. |
| 5 | note | No resource limit on state file size: `EpicState::load` reads the entire file into memory via `read_to_string` with no size guard. A maliciously crafted or corrupted multi-gigabyte `state.json` could cause OOM. | `src/main.rs:66`, `src/state.rs:75` | Add a size check before reading (e.g., reject files larger than a reasonable cap such as 100 MB). |
| 6 | note | `std::process::exit(1)` called in multiple error paths bypasses Drop destructors and any pending async cleanup. While not a direct security issue, it could leave temporary files (e.g., `state.json.tmp` from a concurrent save) on disk. | `src/main.rs:46,73,83,93,101,123,132,142,152,221` | Return `Err(...)` or use `anyhow::bail!` to allow normal unwinding, or at minimum flush/remove temp files before exiting. |

## Notes
- The actual command execution surface (bash tool, file read/write, path traversal) lives in `src/agent/tools.rs`, not in the files under review. That module has its own `safe_path` function with symlink and traversal checks, reviewed separately.
- The credential value itself is never logged or printed by `cli.rs` or `main.rs`; it is passed opaquely to `FlickAgent::new`.
- The atomic-rename pattern in `EpicState::save` (write to `.tmp`, rename) is a good practice for crash safety but does not address the TOCTOU concern on the load side.
