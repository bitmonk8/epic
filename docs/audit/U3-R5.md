# U3-R5: agent/config_gen / Simplification

## Summary

The file contains significant duplicated JSON schema definitions, a pattern of near-identical `build_*_config` wrapper functions that could be collapsed, and string-based enum parsing that could leverage serde or `FromStr` impls on the domain types themselves.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Subtask item schema is duplicated verbatim between `decomposition_schema()` and `recovery_plan_schema()` | `config_gen.rs:229-239` / `config_gen.rs:303-315` | Extract a `subtask_item_schema() -> JsonValue` helper and reference it in both places. |
| 2 | minor | Seven `build_*_config` wrappers (lines 375-530) follow an identical pattern: call `tool_definitions`, call a schema fn, delegate to `build_config`. Four of them share the exact same signature `(system_prompt, model, credential, grant)`. | `config_gen.rs:375-530` | Consider a single public entry point, e.g. `build_config_for(role: AgentRole, ...)`, using an enum to select the schema, or at minimum a generic helper that accepts a schema-producing closure. |
| 3 | minor | `parse_model_name` (line 536) hand-matches string variants that already appear as the `Model` enum. The same pattern repeats for `TaskPath` (line 31), `MagnitudeEstimate` (line 68), `TaskOutcome` (line 106), `VerificationOutcome` (line 130), `CheckpointDecision` (line 153), and recovery approach (line 186). | `config_gen.rs:31-189`, `config_gen.rs:536-543` | Derive or implement `FromStr` (or use `#[serde(rename_all = "lowercase")]`) on the domain enums so deserialization handles validation automatically, eliminating per-variant match arms here. |
| 4 | minor | `build_config` serializes a `serde_json::Value` to a string and immediately re-parses it via `flick::Config::from_str`. This round-trip exists only because flick's config API takes a string. | `config_gen.rs:369-373` | If flick exposes a `from_value` or builder API, use it directly. If not, add a comment explaining the round-trip is intentional so future readers do not try to "fix" it. |
| 5 | note | The `VerificationWire` / `TaskOutcomeWire` conversions both use the identical pattern for a variant with an optional `reason` field and a `"no reason provided"` fallback. | `config_gen.rs:108-109` / `config_gen.rs:132-133` | Minor, but could share a tiny helper: `fn require_reason(r: Option<String>) -> String`. |
| 6 | note | Schema functions are pure and deterministic but allocate a new `JsonValue` on every call. In hot paths this is wasteful. | `config_gen.rs:208-321` | Use `LazyLock<JsonValue>` (or `once_cell`) to compute each schema once and return a reference. Only worthwhile if profiling shows allocation pressure. |

## Notes

- The domain types (`Model`, `TaskPath`, `MagnitudeEstimate`, etc.) live outside this file, so adding `FromStr` / serde rename attributes would touch multiple modules. The payoff is eliminating roughly 40 lines of match arms in this file and preventing the two sites (schema enum values and match arms) from drifting out of sync.
- Finding 2 is the highest-leverage simplification: collapsing the seven thin wrappers would remove about 60 lines with no behavioral change.
- Finding 6 is flagged as `note` because config generation is not on a hot path today; it runs once per agent invocation.
