# U3-R3: agent/config_gen / Error handling

## Summary

Error handling in `config_gen.rs` is well-structured overall, using `anyhow::Result` with `bail!` for invalid enum variants and `?` for propagation. A few minor issues exist around silent defaults that could mask LLM misbehavior, and test code uses `.unwrap()` liberally (acceptable for tests but noted for completeness).

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `TaskOutcomeWire::try_from` silently substitutes `"no reason provided"` when an LLM returns `outcome: "failed"` without a `reason` field. This masks a malformed response that the caller cannot distinguish from a legitimate failure with that exact string. | `config_gen.rs:109` | Return an error (or at least log a warning) when `outcome == "failed"` but `reason` is `None`, so callers know the LLM omitted a required explanation. |
| 2 | minor | `VerificationWire::try_from` similarly substitutes a default `"no reason provided"` for a missing `reason` on `"fail"` outcomes. Same masking concern as finding #1. | `config_gen.rs:133` | Treat a missing reason on a fail outcome as an error, or propagate an `Option` so the caller can decide. |
| 3 | minor | `CheckpointWire::try_from` silently converts `guidance: None` to an empty string for the `"adjust"` decision. An adjust without guidance is arguably invalid since the whole point of adjust is to provide corrective guidance. | `config_gen.rs:156` | Either require `guidance` when `decision == "adjust"` (return an error if absent) or use `Option<String>` in `CheckpointDecision::Adjust` to represent the absence explicitly. |
| 4 | minor | `RecoveryWire::into_strategy` returns `Some(String::new())` when `recoverable == true` but `strategy` is `None`. Downstream code receiving `Some("")` may not handle an empty strategy correctly and could silently proceed with no actual recovery plan. | `config_gen.rs:172` | Return an error or require a non-empty strategy when `recoverable` is true, rather than defaulting to an empty string. |
| 5 | note | `DetectedStepWire::from` defaults `timeout` to 300 seconds when the LLM omits it. This is a reasonable default but is hard-coded; if the project-level default changes, this value would drift. | `config_gen.rs:466` | Consider pulling the default timeout from a constant or the project config rather than a magic number. |
| 6 | note | `build_config` at line 371-372 converts the `flick::Config::from_str` error via `map_err` into a new `anyhow::anyhow!` string. This discards any structured error type from the `flick` crate. Not a bug, but it loses error chain context compared to using `.context()`. | `config_gen.rs:371-372` | If `flick`'s error type implements `std::error::Error`, use `anyhow::Error::from(e).context("failed to parse flick config")` to preserve the error chain. |

## Notes

- No `unwrap()` or `expect()` calls exist in production code; all `.unwrap()` usage is confined to the `#[cfg(test)]` module, which is appropriate.
- No panicking paths (`panic!`, `todo!`, `unreachable!`) exist in production code.
- No resource cleanup concerns apply; the module is purely data transformation with no I/O handles, locks, or file descriptors.
- Error propagation through `TryFrom` and `anyhow::Result` is consistent and idiomatic. All public builder functions return `Result`, giving callers full control over error handling.
