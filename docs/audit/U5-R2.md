# U5-R2: agent/tools / Security

## Summary

The bash tool executes shell commands inside a lot sandbox with per-phase OS-level isolation; environment is scrubbed via `forward_common_env()`. Path-containment logic is sound for file tools but has a narrow TOCTOU window in `write_file`, and the regex-based grep tool accepts unbounded patterns that can cause excessive compilation memory.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **No write-content size limit.** `tool_write_file` and `tool_edit_file` accept unbounded `content`/`new_string` from the LLM. A single call can write gigabytes to disk, exhausting storage and causing denial of service. | `tools.rs:482-517` | Enforce a `MAX_WRITE_BYTES` constant (e.g. 1 MiB) and reject writes that exceed it. |
| 2 | major | **ReDoS via user-supplied regex.** `tool_grep` compiles an LLM-provided pattern with `Regex::new` and applies it line-by-line across potentially millions of lines. Rust's `regex` crate is resilient to catastrophic backtracking but still allows patterns with very large automaton states; a pathological pattern can cause excessive memory allocation during compilation. | `tools.rs:402` | Set a size limit on the pattern string (e.g. 1024 chars) and/or use `RegexBuilder` with `size_limit()` and `dfa_size_limit()` to cap automaton memory. |
| 3 | minor | **TOCTOU in `write_file` between `create_dir_all` and `safe_path`.** Directories are created at line 505 before `safe_path` validates the final path at line 509. A concurrent symlink attack between those two operations could redirect the write outside the project root. | `tools.rs:505-509` | Perform `safe_path` validation (or at least the symlink check) after `create_dir_all` but before `tokio::fs::write`, and consider using `O_NOFOLLOW` or atomically checking the target. |
| 4 | minor | **Glob pattern not size-bounded.** An adversarial glob pattern (e.g. deeply nested `{a,b}{a,b}...` alternation) could cause expensive compilation in `GlobBuilder`. | `tools.rs:360-364` | Limit pattern string length before passing to `GlobBuilder`. |
| 5 | note | **`edit_file` read-then-write is not atomic.** Between reading the file and writing it back, another process (or a concurrent tool call) could modify it, causing silent data loss. This is a classic TOCTOU pattern. | `tools.rs:525-546` | Use advisory file locking (`flock`) or write to a temporary file and atomically rename. |

## Notes

- The `safe_path` containment logic (lines 162-217) is well-structured: it canonicalizes paths, checks symlink targets, and blocks traversal. The `verify_ancestors_within_root` helper is a good defense against `create_dir_all` abuse. The main weakness is the ordering gap in `write_file`.
- The `regex` crate's guarantee of linear-time matching mitigates the worst ReDoS scenarios, but compilation-time memory is still unbounded without explicit limits.
- The grant/permission system (bitflags + `required_grant`) is correctly implemented and cannot be bypassed through the `execute_tool` entry point. The BASH grant is backed by OS-level sandbox isolation via lot.
- Resource exhaustion caps exist for reads (`MAX_READ_BYTES`), grep output (`MAX_GREP_OUTPUT`), bash output (`MAX_BASH_OUTPUT`), and glob results (`MAX_GLOB_RESULTS`), which is good practice. The gap is on the write/input side.
