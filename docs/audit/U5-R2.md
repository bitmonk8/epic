# U5-R2: agent/tools / Security

## Summary

The nu tool executes NuShell commands via a persistent MCP session inside a lot sandbox with per-phase OS-level isolation; environment is scrubbed via `forward_common_env()` (`nu_session.rs:355`). Path-containment logic is sound for file tools but has a narrow TOCTOU window in `write_file`.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | ~major~ fixed | ~**No write-content size limit.**~ `MAX_WRITE_BYTES` (1 MiB) now enforced in `tool_write_file`. | `tools.rs` | Done. |
| 2 | ~major~ fixed | ~**ReDoS via user-supplied regex.**~ `RegexBuilder` with `size_limit(1 << 20)` and `dfa_size_limit(1 << 20)` now caps automaton memory. | `tools.rs` | Done. |
| 3 | minor | **TOCTOU in `write_file` between `create_dir_all` and `safe_path`.** Directories are created at line 534 before `safe_path` validates the final path at line 538. A concurrent symlink attack between those two operations could redirect the write outside the project root. | `tools.rs:534-538` | Perform `safe_path` validation (or at least the symlink check) after `create_dir_all` but before `tokio::fs::write`, and consider using `O_NOFOLLOW` or atomically checking the target. |
| 4 | minor | **Glob pattern not size-bounded.** An adversarial glob pattern (e.g. deeply nested `{a,b}{a,b}...` alternation) could cause expensive compilation in `GlobBuilder`. | `tools.rs:377-381` | Limit pattern string length before passing to `GlobBuilder`. |
| 5 | note | **`edit_file` read-then-write is not atomic.** Between reading the file and writing it back, another process (or a concurrent tool call) could modify it, causing silent data loss. This is a classic TOCTOU pattern. | `tools.rs:552-570` | Use advisory file locking (`flock`) or write to a temporary file and atomically rename. |

## Notes

- The `safe_path` containment logic is well-structured: it canonicalizes paths, checks symlink targets, and blocks traversal. The `verify_ancestors_within_root` helper is a good defense against `create_dir_all` abuse. The main weakness is the ordering gap in `write_file`.
- The `regex` crate's guarantee of linear-time matching mitigates the worst ReDoS scenarios. `RegexBuilder` with `size_limit()` and `dfa_size_limit()` now caps compilation memory (finding #2 fixed).
- The grant/permission system (bitflags + `required_grant`) is correctly implemented and cannot be bypassed through the `execute_tool` entry point. The NU grant is backed by OS-level sandbox isolation via lot.
- Resource exhaustion caps exist for reads (`MAX_READ_BYTES`), writes (`MAX_WRITE_BYTES`), grep output (`MAX_GREP_OUTPUT`), nu output (`MAX_NU_OUTPUT`), and glob results (`MAX_GLOB_RESULTS`).
