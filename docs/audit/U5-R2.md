# U5-R2: agent/tools / Security

## Summary

The bash tool executes arbitrary shell commands with no sandboxing, command filtering, or environment scrubbing, giving an LLM-controlled agent full host access. Path-containment logic is sound for file tools but has a narrow TOCTOU window in `write_file`, and the regex-based grep tool accepts unbounded patterns that can cause catastrophic backtracking.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | critical | **Unsandboxed shell execution.** `tool_bash` passes LLM-supplied strings directly to `sh -c` with no allow/deny-list, no seccomp/namespace/container isolation, and no restriction on network access, `sudo`, `curl`, package managers, etc. A prompt-injected or misguided agent can exfiltrate data, install backdoors, or destroy the host. | `tools.rs:559-562` | Run commands inside a container, nsjail, or at minimum a restricted shell with an `execve` allow-list. Drop network access unless explicitly granted. |
| 2 | major | **Environment inherited by child process.** `tokio::process::Command::new("sh")` inherits the full parent environment including `HOME`, API keys, cloud credentials (`AWS_SECRET_ACCESS_KEY`, `ANTHROPIC_API_KEY`, etc.). The agent can run `env` or `printenv` to harvest all secrets. | `tools.rs:559-565` | Call `cmd.env_clear()` and selectively re-add only safe variables (`PATH`, `LANG`, `TERM`, project-specific vars). |
| 3 | major | **No write-content size limit.** `tool_write_file` and `tool_edit_file` accept unbounded `content`/`new_string` from the LLM. A single call can write gigabytes to disk, exhausting storage and causing denial of service. | `tools.rs:482-517` | Enforce a `MAX_WRITE_BYTES` constant (e.g. 1 MiB) and reject writes that exceed it. |
| 4 | major | **ReDoS via user-supplied regex.** `tool_grep` compiles an LLM-provided pattern with `Regex::new` and applies it line-by-line across potentially millions of lines. Rust's `regex` crate is resilient to catastrophic backtracking but still allows patterns with very large automaton states; a pathological pattern can cause excessive memory allocation during compilation. | `tools.rs:402` | Set a size limit on the pattern string (e.g. 1024 chars) and/or use `RegexBuilder` with `size_limit()` and `dfa_size_limit()` to cap automaton memory. |
| 5 | minor | **TOCTOU in `write_file` between `create_dir_all` and `safe_path`.** Directories are created at line 505 before `safe_path` validates the final path at line 509. A concurrent symlink attack between those two operations could redirect the write outside the project root. | `tools.rs:505-509` | Perform `safe_path` validation (or at least the symlink check) after `create_dir_all` but before `tokio::fs::write`, and consider using `O_NOFOLLOW` or atomically checking the target. |
| 6 | minor | **Bash tool timeout kills only the direct child, not the process group.** `kill_on_drop(true)` sends SIGKILL to the `sh` process, but any grandchild processes it spawned (e.g. `nohup`, background jobs) survive and continue consuming resources. | `tools.rs:565` | Spawn the child in its own process group (`pre_exec` + `setsid`/`setpgid`) and kill the entire group on timeout. |
| 7 | minor | **Glob pattern not size-bounded.** An adversarial glob pattern (e.g. deeply nested `{a,b}{a,b}...` alternation) could cause expensive compilation in `GlobBuilder`. | `tools.rs:360-364` | Limit pattern string length before passing to `GlobBuilder`. |
| 8 | note | **`edit_file` read-then-write is not atomic.** Between reading the file and writing it back, another process (or a concurrent tool call) could modify it, causing silent data loss. This is a classic TOCTOU pattern. | `tools.rs:525-546` | Use advisory file locking (`flock`) or write to a temporary file and atomically rename. |

## Notes

- The `safe_path` containment logic (lines 162-217) is well-structured: it canonicalizes paths, checks symlink targets, and blocks traversal. The `verify_ancestors_within_root` helper is a good defense against `create_dir_all` abuse. The main weakness is the ordering gap in `write_file`.
- The `regex` crate's guarantee of linear-time matching mitigates the worst ReDoS scenarios, but compilation-time memory is still unbounded without explicit limits.
- The grant/permission system (bitflags + `required_grant`) is correctly implemented and cannot be bypassed through the `execute_tool` entry point. However, the grants are only as meaningful as the sandboxing behind them -- the BASH grant currently confers near-root-equivalent access.
- Resource exhaustion caps exist for reads (`MAX_READ_BYTES`), grep output (`MAX_GREP_OUTPUT`), bash output (`MAX_BASH_OUTPUT`), and glob results (`MAX_GLOB_RESULTS`), which is good practice. The gap is on the write/input side.
