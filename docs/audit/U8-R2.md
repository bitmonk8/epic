# U8-R2: state / Security

## Summary
The state module handles task-tree persistence via JSON serialization to disk. It has no critical vulnerabilities but presents several minor concerns around resource exhaustion, file permissions, and cycle safety in graph traversal.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `dfs_order()` has no cycle detection; a cyclic `subtask_ids` graph causes infinite loop and unbounded memory growth | `state.rs:40-53` | Maintain a `HashSet<TaskId>` of visited nodes and skip already-visited IDs |
| 2 | minor | `load()` deserializes arbitrary JSON with no size or depth limits; a crafted state file could cause memory exhaustion or stack overflow | `state.rs:74-78` | Validate file size before reading (e.g., reject files over a reasonable cap) and/or limit the deserialized task count after loading |
| 3 | minor | `save()` writes the state file with default OS permissions; task goals and acceptance criteria may be sensitive | `state.rs:66` | Set restrictive permissions (0600) on the temp file before writing, or use a platform-aware helper that sets permissions atomically |
| 4 | minor | No upper bound on the number of tasks stored in the `HashMap`; unbounded insertion could exhaust memory | `state.rs:27-29` | Add a configurable maximum task count and return an error from `insert()` when the limit is exceeded |
| 5 | note | If the process is killed between `std::fs::write` (line 66) and `std::fs::rename` (line 67), the `.tmp` file is orphaned on disk | `state.rs:65-67` | Document the possibility or add a cleanup-on-load step that removes stale `.tmp` files |
| 6 | note | `save()` and `load()` accept an arbitrary `&Path` with no validation; if the path originates from user input, directory traversal is possible | `state.rs:63, 74` | If the path can come from untrusted input, validate it against an allowed base directory |

## Notes
- The atomic write-then-rename pattern in `save()` is a good practice and prevents partial-write corruption under normal operation.
- `next_task_id()` uses a monotonically increasing `u64` counter, which is safe from collision but will silently wrap on overflow (theoretical concern given the magnitude of `u64`).
- The module does not handle any credentials, secrets, or subprocess invocations, so injection and credential-exposure categories are not applicable.
