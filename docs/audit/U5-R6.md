# U5-R6: agent/tools / Testability

## Summary

The module has solid inline tests for the happy path and basic error cases, but all tool implementations are hardwired to real filesystem and process I/O with no trait-based abstraction boundaries, making it impossible to unit-test tool logic without touching disk or spawning shells. Several edge cases and the security-critical path helpers lack coverage.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | No abstraction over filesystem I/O ‚Äî `tool_read_file`, `tool_write_file`, `tool_edit_file`, `tool_glob`, `tool_grep` call `tokio::fs::*` and `std::fs::*` directly, preventing mock injection for fast, deterministic tests | `tools.rs:319-549` | Introduce a `trait Filesystem { async fn read, write, metadata, ‚Ä¶ }` and inject it into tool functions (or into a `ToolContext` struct); provide a real impl and an in-memory test double |
| 2 | major | No abstraction over process execution ‚Äî `tool_bash` spawns a real shell via `tokio::process::Command`, making tests slow, non-deterministic, and platform-dependent | `tools.rs:551-622` | Extract process spawning behind a `trait CommandRunner` so tests can supply a stub returning predetermined stdout/stderr/exit-code |
| 3 | minor | `safe_path` and `verify_ancestors_within_root` are private, so they can only be tested indirectly through `execute_tool`; the existing `test_safe_path_*` tests call them directly because they share the module, but external crates or integration tests cannot exercise them in isolation | `tools.rs:162-241` | Consider making the path-safety helpers `pub(crate)` (or placing them in a `path_safety` sub-module with `pub` visibility) so integration tests can target them directly |
| 4 | minor | No test for `verify_ancestors_within_root` ‚Äî this function guards against `create_dir_all` escaping the project root, a security-sensitive path, yet has zero dedicated test cases | `tools.rs:222-241` | Add tests for: ancestor outside root, symlink ancestor, deeply nested non-existent path |
| 5 | minor | No test for symlink escape in `safe_path` ‚Äî the `allow_new_file` branch (lines 186-200) contains explicit symlink-target validation but no test exercises it | `tools.rs:186-200` | Create a temp dir with a symlink pointing outside root and assert `safe_path` rejects it |
| 6 | minor | `tool_bash` timeout test uses `sleep 10` with a 2-second timeout, introducing real wall-clock delay; the test also relies on a Unix shell being available | `tools.rs:904-916` | With a `CommandRunner` trait this becomes a pure unit test; short-term, reduce sleep to `sleep 60` with timeout 1 to minimise wasted time, and gate the test with `#[cfg(unix)]` |
| 7 | minor | `tool_grep` glob filter path ‚Äî when `glob` parameter is set but `strip_prefix` fails (search path is a file, not a directory), the filter is silently skipped rather than applied, and no test covers this branch | `tools.rs:442-448` | Add a test that greps a single file with a glob filter to verify the filter is correctly applied or intentionally bypassed |
| 8 | note | Non-zero exit codes from `tool_bash` return `is_error: false` ‚Äî this is a design choice but callers may not expect it; the test on line 929 documents it, yet there is no test verifying that the *caller* of `execute_tool` handles this correctly | `tools.rs:596-621` | Consider whether non-zero exits should set `is_error: true`, or add integration-level tests that verify upstream consumption of exit-code results |
| 9 | note | `tool_definitions` builds JSON schemas inline with no compile-time validation ‚Äî a schema typo (e.g., wrong `"required"` key) would only surface at runtime | `tools.rs:60-146` | Add a test that deserializes each tool's `parameters` into a `serde_json::Value` and asserts `required` and `properties` keys exist, or use a typed schema builder |

## Notes

- The existing test suite (lines 624-932) is well-structured and covers grant checks, basic CRUD, glob, grep, and bash, all using `tempfile::TempDir` for isolation from the real project tree.
- Because every tool function is `async fn` with a concrete `&Path` parameter, the only way to test without real I/O today is the tempdir approach already in use. Introducing a filesystem trait would be the single highest-leverage change for testability.
- The `WEB` and `TASK` grant flags are declared in `ToolGrant` but no tools use them yet; when those tools are added, the same trait-boundary pattern should be followed from the start.
