# U2-R2: agent/flick / Security

## Summary

The `flick.rs` module is a thin orchestration layer that delegates tool execution to `tools.rs`. The orchestration itself is clean, but the underlying tool implementations expose several security concerns: unsandboxed shell execution, TOCTOU race conditions in file operations, potential credential leakage through error messages, and a regex denial-of-service vector.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | critical | **Unsandboxed bash execution** -- The `bash` tool executes arbitrary shell commands via `sh -c` with no sandboxing, seccomp, namespace isolation, or allowlist filtering. An LLM-controlled agent can run any command the process user can: install packages, exfiltrate data, access the network, read credentials outside project root, or pivot to other hosts. The `project_root` cwd constraint is trivially escaped (`cd /; ...`). | `tools.rs:559-565` | Run commands inside a container, bubblewrap, or at minimum a restricted shell. Apply a network-deny policy. Consider an allowlist of permitted command prefixes. |
| 2 | major | **TOCTOU in write_file path validation** -- `tool_write_file` calls `verify_ancestors_within_root`, then `create_dir_all`, then `safe_path`. Between the ancestor check and directory creation, a concurrent process (or a prior bash tool call) could introduce a symlink at an intermediate path component, causing `create_dir_all` to follow it and create directories outside the project root before `safe_path` runs. | `tools.rs:498-509` | Perform directory creation and path validation atomically: open the parent directory with `O_NOFOLLOW` / `O_DIRECTORY` checks at each path component, or validate again after creation using `canonicalize` on the created directory. |
| 3 | major | **TOCTOU in edit_file between read and write** -- `tool_edit_file` reads the file, validates the match count, then writes back. A concurrent tool call or bash command could modify the file between read and write, causing data loss or corruption. | `tools.rs:525-546` | Use advisory file locking (`flock`) or open with `O_EXCL`-style coordination. Alternatively, verify file content hash has not changed before writing. |
| 4 | major | **Credential name passed through JSON config** -- `credential_name` is interpolated into a JSON config blob (`config_gen.rs:346`) and passed to `flick::Config`. If Flick logs the config, serializes it for debugging, or includes it in error messages, the credential identifier (and potentially the resolved credential value) is exposed. The error paths in `build_client` (line 53) and `run_structured` (line 75) propagate raw Flick errors that may contain config details. | `flick.rs:26,38` / `config_gen.rs:346` | Redact or mask `credential_name` in all logged/error output. Audit the Flick library's logging behavior. Consider passing credentials via a callback or environment variable rather than inline config. |
| 5 | minor | **Regex denial of service in grep tool** -- The `grep` tool compiles a user-supplied (LLM-supplied) regex pattern with `Regex::new` without size or complexity limits. Pathological patterns (e.g., `(a+)+$`) can cause exponential backtracking on certain file contents. | `tools.rs:402` | Use `regex::RegexBuilder` with a `size_limit` to cap NFA size, or wrap compilation in a timeout. The `regex` crate mostly avoids catastrophic backtracking but unbounded NFA size can still cause high memory usage. |
| 6 | minor | **No per-tool-call rate limiting or resource caps** -- The tool loop runs up to 50 rounds (`MAX_TOOL_ROUNDS`), each of which can spawn bash processes, read large files, or walk the entire directory tree. There is no aggregate CPU/memory/disk budget, only per-call timeouts. A malicious or runaway agent could exhaust system resources within 50 rounds. | `flick.rs:21,106` | Add aggregate resource accounting: total bytes read, total subprocess time, total wall-clock time. Consider cgroup limits for the overall agent run. |
| 7 | minor | **Structured output logged in error context** -- When `serde_json::from_str` fails, the raw LLM text is included in the error message via `format!("failed to parse structured output: {text}")`. If this text contains sensitive project information the agent read via tools, it could leak into logs. | `flick.rs:80-81`, `flick.rs:145-146` | Truncate the text in error messages (e.g., first 200 chars) and avoid logging full LLM output at levels visible outside debug. |
| 8 | note | **safe_path symlink check only covers the final component for new files** -- When `allow_new_file` is true, `safe_path` checks if the resolved file is a symlink, but intermediate directories could still contain symlinks. The `canonicalize` on the parent should resolve these, but the comment and code suggest this edge case was considered tricky. | `tools.rs:186-199` | Add a test case that creates a symlink in an intermediate directory pointing outside root and confirms it is rejected. Document the invariant explicitly. |

## Notes

- The most impactful finding is #1 (unsandboxed bash). The grant system ensures only `Execute`-phase calls get bash access, but within that phase the LLM has unrestricted shell access as the host process user. This is the primary attack surface for prompt injection or agent misbehavior.
- The TOCTOU issues (#2, #3) require concurrent operations, which are plausible since the tool loop processes multiple tool calls sequentially but bash commands can spawn background processes.
- The `credential_name` field appears to be an identifier (not the raw API key), but its handling path through Flick's config parsing and error reporting should be audited end-to-end in the Flick library itself.
- The `ToolGrant` permission model is well-designed in principle -- analysis phases get read-only access, which limits damage from prompt injection during those phases.
