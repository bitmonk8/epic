# U2-R7: agent/flick / Design intent

## Summary

`FlickAgent` faithfully implements the Epic-as-orchestrator / Flick-as-runtime split with per-call model selection, per-call tool scoping, and structured output parsing. However, several model assignments and tool grants diverge from the AGENT_DESIGN specification, and cost-control telemetry is absent.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **assess_recovery uses Sonnet instead of Opus.** AGENT_DESIGN specifies "Recovery assessment: Opus -- Requires strongest reasoning," but `assess_recovery` passes `Model::Sonnet`. This undermines the design intent that recovery decisions receive the strongest model, risking poor recovery strategies for complex failures. | `flick.rs:331` | Change to `Model::Opus`. |
| 2 | major | **assess hardcodes Sonnet instead of Haiku.** AGENT_DESIGN specifies assessment should use Haiku (escalating to Sonnet only if uncertain). The current implementation always uses Sonnet, overspending on a classification task that should be cheap. | `flick.rs:199` | Change to `Model::Haiku`. If two-phase escalation is deferred, document the deviation. |
| 3 | major | **design_and_decompose hardcodes Sonnet instead of using the assessment-selected model.** AGENT_DESIGN says branch decomposition should use the assessment-selected model since "decomposition difficulty varies." The method signature does not accept a `model` parameter at all, always using Sonnet regardless of assessment. | `flick.rs:250-267` | Add a `model: Model` parameter (like `execute_leaf` has) and pass the assessment-selected model from the caller. |
| 4 | major | **Decompose tool grant is READ-only, but AGENT_DESIGN specifies EXPLORE (READ\|EXECUTE\|WEB).** `phase_tools(AgentMethod::Decompose)` returns only `READ`, meaning decomposition agents cannot run shell commands or perform web searches during design. This limits the agent's ability to explore unfamiliar codebases during branch design. | `flick.rs:255` (via `tools.rs:47`) | Update `AgentMethod::Decompose` mapping in `phase_tools` to return `READ \| BASH \| WEB` to match the EXPLORE grant from the design. |
| 5 | major | **No usage/cost tracking.** `FlickResult.usage` is available from Flick but is never read or propagated in any production code path. The design docs reference a metrics layer for "Token/cost tracking" (`ARCHITECTURE.md` line 75), but `FlickAgent` discards all token usage data, making cost control and budget enforcement impossible. | `flick.rs:61-82`, `flick.rs:88-147` | After each `client.run()` / `client.resume()`, accumulate `result.usage` into a counter (per-call and per-tool-loop totals) and return it alongside the deserialized result, or emit it via the event system. |
| 6 | minor | **assess calls run_structured (no tool loop) despite receiving tool definitions in the config.** The assess config is built with `grant` tools via `build_assess_config`, but `run_structured` executes a single call with no tool handling. If the model returns `ToolCallsPending`, the response will be parsed as text, likely producing a deserialization error rather than a clear diagnostic. | `flick.rs:194-205` | Either (a) switch `assess` to use `run_with_tools` so the model can read files during assessment, or (b) remove tools from `build_assess_config` and pass an empty tool list to avoid misleading the model. |
| 7 | minor | **verify always uses Sonnet, ignoring the design's adaptive rule.** AGENT_DESIGN says leaf verification should use "max(Haiku, implementing model) capped at Sonnet." Since the implementing model can be Haiku, always using Sonnet overspends for simple tasks. The method does not accept a `model` parameter. | `flick.rs:291-304` | Accept a `model: Model` parameter and let the caller apply the max/cap rule based on the leaf's implementation model. |
| 8 | note | **ARCHITECTURE.md still says "Flick CLI" / "subprocess invocation" in the system layers diagram and tech stack table.** While `FLICK_INTEGRATION.md` and `AGENT_DESIGN.md` are updated, `ARCHITECTURE.md` lines 20 and 39 reference the old subprocess integration mode. This is a documentation inconsistency, not a code issue. | `ARCHITECTURE.md:20,39` | Update the diagram to say "Flick library" and the tech stack row to reference the `flick` crate dependency. |

## Notes

- Findings 1-4 represent deviations from the model-selection and tool-grant tables in AGENT_DESIGN.md. If these were intentional simplifications (e.g., "always Sonnet for assess" to avoid a two-pass escalation), they should be documented as deliberate deviations rather than left as silent divergences.
- Finding 5 (cost tracking) is flagged as major because the architecture explicitly calls for a metrics layer, and without usage data the system cannot enforce budget limits or provide cost visibility -- a core tenet of the cost-control design intent.
- Several prior audit reports (U2-R3 #3, U2-R5, U2-R6) have already noted the lack of an outer wall-clock deadline on the tool loop and the difficulty of testing the loop in isolation. Those findings are not repeated here but remain relevant to the overall design-intent picture.
- The `MAX_TOOL_ROUNDS = 50` constant provides a hard ceiling on agent autonomy per call, which is a reasonable safety rail, though (as noted in U2-R3) total wall-clock time is unbounded since each of the 50 rounds can run up to `call_timeout`.
