# U9-R5: events / Simplification

## Summary
The `Event` enum contains several structurally duplicate variant groups that differ only by context label (retry vs fix vs recovery, model escalation vs fix-model escalation). These could be unified with a discriminant field, reducing both variant count and the matching boilerplate in the TUI consumer.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `ModelEscalated` and `FixModelEscalated` are structurally identical (`task_id`, `from`, `to`). They are handled with near-identical TUI logic differing only in the log prefix string. | `src/events.rs:26-30`, `src/events.rs:66-70` | Unify into a single `ModelEscalated { task_id, from, to, context: EscalationContext }` variant where `EscalationContext` is `Retry` or `Fix`. This removes one variant and one TUI match arm. |
| 2 | minor | `RetryAttempt`, `FixAttempt`, and `BranchFixRound` share the same shape (`task_id`, attempt/round: u32, model`). All three are rendered as warnings in the TUI with only a label difference. | `src/events.rs:39-43`, `src/events.rs:61-65`, `src/events.rs:71-75` | Consolidate into a single `Attempt { task_id, round: u32, model, kind: AttemptKind }` where `AttemptKind` is `Retry`, `Fix`, or `BranchFix`. Cuts three variants to one. |
| 3 | minor | `SubtasksCreated`, `FixSubtasksCreated`, and `RecoverySubtasksCreated` carry the same semantic (children spawned) with slightly varying fields. `SubtasksCreated` carries `child_ids: Vec<TaskId>` while the other two carry `count: usize` and `round: u32`. | `src/events.rs:31-34`, `src/events.rs:76-80`, `src/events.rs:89-93` | Merge into one variant `SubtasksCreated { parent_id, child_ids: Vec<TaskId>, round: Option<u32>, kind: SubtaskOrigin }`. The fix/recovery sites already know the ids (they emit `SubtasksCreated` alongside `FixSubtasksCreated`), so supplying `child_ids` uniformly is feasible and removes the redundant count field. |
| 4 | note | `CheckpointAdjust` and `CheckpointEscalate` are unit-like variants differing only in semantics, each carrying just `task_id`. | `src/events.rs:55-59` | Could merge into `Checkpoint { task_id, action: CheckpointAction }` but the benefit is marginal given there are only two variants and they are small. Acceptable as-is. |
| 5 | note | `event_channel()` is a one-line wrapper around `mpsc::unbounded_channel()`. It adds no logic, but it does centralize the channel type so callers do not import `mpsc` directly. | `src/events.rs:99-101` | Acceptable as a convenience constructor. No change needed. |

## Notes
- Findings 1-3 each also reduce matching boilerplate in `src/tui/mod.rs` and test assertions in `src/orchestrator.rs`, so the net simplification extends well beyond `events.rs` itself.
- The enum currently has 19 variants. Applying findings 1-3 would reduce it to 14 variants without losing information.
- Any refactor touching the `Event` enum will require updating match arms in both `src/tui/mod.rs` and test code in `src/orchestrator.rs`. The changes are mechanical but spread across a large file.
