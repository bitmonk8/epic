# U12-R1: cli + main / Correctness

## Summary

The CLI and main modules are generally well-structured with proper error handling for corrupt state files. The most significant issue is state loss when the orchestrator is forcibly aborted after a TUI exit, and a duplicated guard that could drift out of sync.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | When the TUI exits and the orchestrator does not finish within 2 seconds, the handle is aborted and state is never saved. All in-flight progress is silently lost. The no-TUI path (line 170) always saves, but the TUI timeout path (lines 197-199) does not. | `src/main.rs:197-199` | Before aborting, attempt to extract and save the state (e.g., via a shared `Arc<Mutex<EpicState>>` or a cancellation-token pattern that lets the orchestrator flush state on shutdown). |
| 2 | minor | The `epic.toml`-already-exists guard is duplicated: once in `main.rs` (lines 39-47, exits with `process::exit(1)`) and again in `init::run_init` (lines 14-18, returns `anyhow::Error`). If one is updated and the other is not, behavior diverges. The `main.rs` check also uses `process::exit` which skips destructors. | `src/main.rs:39-47`, `src/init.rs:14-18` | Remove the guard from `main.rs` and rely solely on `init::run_init`, or remove it from `init.rs`. Prefer returning an error over `process::exit`. |
| 3 | minor | Multiple error paths use `std::process::exit(1)` (lines 46, 73, 83, 93, 101, 123, 131, 141, 151, and `print_status` line 221) instead of returning `anyhow::Result`. This bypasses drop glue and any future cleanup logic (e.g., terminal restore in TUI mode). | `src/main.rs` (multiple) | Return `anyhow::bail!()` or a typed error instead of calling `process::exit`. |
| 4 | minor | `Command::Run` with an existing state file silently resumes (line 103) when the goals happen to match. The user typed `epic run "goal"` expecting a fresh start but gets a resume. This is surprising UX and could mask stale state. | `src/main.rs:96-104` | Prompt the user or require an explicit `--resume` flag rather than auto-resuming based on goal string equality. |
| 5 | note | The `Command::Init` path (line 50) creates the `.epic/` working directory even though `init` only writes `epic.toml` at the project root. This leaves an empty `.epic/` directory on disk if `init` fails or succeeds without needing it. | `src/main.rs:50` | Move `create_dir_all` to after the `Init` early-return, so it only runs for `Run`/`Resume`. |
| 6 | note | `cli.credential` (a `String`) is moved into `FlickAgent::new` on line 55, consuming it. This currently compiles because `cli.no_tui` is `Copy` and `cli.command` was already matched by reference. However, any future addition that needs `cli.credential` after line 55 will fail to compile, and the ownership split is non-obvious. | `src/main.rs:55` | Clone the credential or restructure so the CLI struct is consumed in one place. |

## Notes

- The `unreachable!()` on line 158 (`Command::Init | Command::Status`) is sound because both variants are handled with early returns above the match. If a new variant is added to `Command` without updating this match, the compiler will flag a non-exhaustive pattern, so this is safe.
- `EpicState::next_task_id` is declared `const fn` with `&mut self`. This compiles on Rust 1.83+ but is unusual for a side-effecting method; it may confuse readers expecting `const fn` to be pure.
- Goal comparison on line 96 (`*goal != persisted_goal`) uses byte-level `String` equality. Goals differing only in whitespace or casing will be treated as different, which seems intentional but is worth noting.
