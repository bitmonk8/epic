# U8-R6: state / Testability

## Summary
`EpicState` is a small, focused module with reasonable test coverage for the happy path, but its `save`/`load` methods are tightly coupled to the real filesystem with no abstraction boundary, and several state-machine edge cases lack test coverage.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `save` and `load` call `std::fs` directly with no trait or injected writer, making it impossible to unit-test persistence without touching the real filesystem. The existing test writes to `std::env::temp_dir`, which is non-hermetic and can collide across parallel test runs. | `state.rs:63-78` | Extract an `io::Write`/`io::Read`-based serialization layer (e.g., `save_to(&self, w: impl Write)` / `load_from(r: impl Read)`) so the atomic-rename logic can be tested separately from serialization, and serialization can be tested in-memory. |
| 2 | major | No test exercises the error/failure paths of `save` (e.g., write to non-existent directory, rename failure, tmp cleanup) or `load` (missing file, corrupt JSON). The atomic-write contract (tmp file removed on rename failure) is asserted only for the success case. | `state.rs:87-137` | Add tests for: save to invalid path, load of missing file, load of malformed JSON, and simulated rename failure to verify tmp cleanup. |
| 3 | minor | The single test (`persistence_round_trip`) covers insert, get, dfs_order, save, load, and root_id in one monolithic function. A failure in any assertion makes it unclear which capability is broken. | `state.rs:87-137` | Split into focused tests: one for pure in-memory operations (insert/get/dfs_order), one for serialization round-trip, one for edge cases. |
| 4 | minor | `dfs_order` is untested for edge cases: missing root id, a task referencing a non-existent child, empty tree, and diamond/DAG-shaped graphs. A missing child silently terminates that branch, which may mask bugs in callers. | `state.rs:40-53` | Add tests for `dfs_order` with an unknown root, a task whose `subtask_ids` contain a non-existent id, and an empty state. |
| 5 | minor | `next_task_id` is marked `const fn` but mutates `self` -- this compiles only on nightly/2024 edition with `const_mut_refs`. If the project ever targets stable without that feature, it will break. More relevantly for testability, there is no way to seed or reset the id counter, making deterministic multi-state test setups awkward. | `state.rs:21-25` | Drop the `const` qualifier. Consider adding a `#[cfg(test)] pub fn set_next_id(&mut self, id: u64)` helper or a builder to simplify test setup. |
| 6 | note | `EpicState` fields (`tasks`, `next_id`, `root_id`) are private, which is good encapsulation, but the test asserts `loaded.next_id` directly (accessing a private field from the in-module test). External integration tests cannot perform equivalent assertions without public accessors. | `state.rs:118` | Consider a `pub fn task_count(&self) -> usize` or similar query method so integration tests outside the module can verify state without field access. |

## Notes
- The module is intentionally small (79 lines of production code) and the state machine is essentially a task tree with an id allocator -- there is no explicit phase-transition logic here (that lives in `Task`). The testability concerns are primarily around I/O coupling and missing negative-path coverage rather than state-machine clarity.
- The temp-dir test path (`epic_test_state`) is a fixed name with no unique suffix, creating a race condition if tests run in parallel or a previous run left artifacts behind.
