# U5-R5: agent/tools / Simplification

## Summary

The file is reasonably well-structured, but contains significant boilerplate in tool definition construction, duplicated path-resolution logic in `tool_write_file`, duplicated truncation logic in `tool_bash`, and an unreachable branch in `execute_tool`.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `tool_definitions` builds each `FlickToolDef` inline with verbose `serde_json::json!` blocks. Every entry repeats the same `"type": "object"`, `"properties"`, `"required"` boilerplate. | `tools.rs:60-146` | Extract a helper like `fn make_tool(name, desc, props, required) -> FlickToolDef` or use a declarative macro/table to declare tools, cutting ~60 lines of near-identical JSON scaffolding. |
| 2 | minor | `tool_write_file` re-implements the absolute-vs-relative path resolution (`if Path::new(raw).is_absolute() { ... } else { project_root.join(raw) }`) that already exists inside `safe_path`. | `tools.rs:487-491` | Refactor `safe_path` to optionally create parent directories (or extract the candidate-building step into its own small helper) so `tool_write_file` can reuse it instead of duplicating the logic. |
| 3 | minor | Stdout and stderr truncation in `tool_bash` uses an identical char-boundary-safe truncation pattern repeated verbatim for both streams. | `tools.rs:579-594` | Extract a `fn truncate_utf8(s: &mut String, max: usize, label: &str)` helper to eliminate the duplicated block. |
| 4 | note | The `_ => Err(...)` arm in the `match name` dispatch inside `execute_tool` (line 298) is unreachable -- `required_grant` already returns `None` for unknown tool names, which is handled and returned as an error on line 281. | `tools.rs:298` | Replace with `_ => unreachable!()` to make the invariant explicit, or remove the earlier `None` guard and let the dispatch match handle unknown tools in one place. |
| 5 | note | `tool_definitions` uses a `Vec` that is grown incrementally with individual `push` calls. The maximum number of tools is statically known per grant combination. | `tools.rs:61` | Pre-allocate with `Vec::with_capacity` based on bit-count of the grant, or build from a static slice filtered by grant, to avoid repeated allocations. Minor, but cleaner. |
| 6 | note | `required_grant` and the dispatch `match` in `execute_tool` both maintain a parallel list of tool name strings. Adding a new tool requires updating both. | `tools.rs:244-251, 291-299` | Unify into a single registry (e.g., a static array of `(name, grant, handler_fn)` tuples) so each tool is declared in exactly one place. |
| 7 | note | `tool_glob` and `tool_grep` both resolve an optional `"path"` parameter with the same two-line pattern (`match get_str_opt(input, "path") { Some(p) => safe_path(...), None => project_root.to_path_buf() }`). | `tools.rs:355-358, 397-400` | Extract a small `resolve_search_dir(input, project_root)` helper to remove the duplication. |

## Notes

The file is ~930 lines including tests (~300 lines of tests). The overall architecture -- bitflag grants, a central dispatcher, individual tool functions -- is sound. The suggestions above are incremental improvements focused on reducing repeated patterns and making tool registration single-source-of-truth. None of the findings affect correctness; they target maintainability as the tool set grows.
