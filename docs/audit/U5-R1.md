# U5-R1: agent/tools / Correctness

## Summary

The tool implementations are generally sound, but contain several correctness issues: a path-traversal bypass via symlink race in `safe_path`, an incomplete security check in the glob filter for `tool_grep`, and a panic-possible edge case in binary detection on empty files.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `safe_path` with `allow_new_file=true` has a TOCTOU race: the symlink check at lines 186-200 runs before the file is written, so a concurrent process could create a symlink at the resolved path between the check and the subsequent `write`. In `tool_write_file` this means an attacker-controlled symlink could redirect writes outside the project root. | `tools.rs:186-200` | Use `O_NOFOLLOW` / `O_CREAT | O_EXCL` when opening the file for write, or open the parent dir with `O_PATH` and use `openat` to avoid the race. |
| 2 | major | In `tool_grep`, when a glob filter is supplied and `strip_prefix` fails (line 443), the file silently passes the filter instead of being skipped. If `search_dir` is not canonical but `path` is (or vice versa due to symlinks/mount points), `strip_prefix` will fail and the glob filter is bypassed, causing unfiltered results. | `tools.rs:442-448` | Change the logic so that a failed `strip_prefix` causes the file to be skipped (`continue`), not implicitly included. |
| 3 | minor | `tool_glob` does not canonicalize `project_root` before using it as `search_dir` (line 357, the `None` branch). However, `safe_path` canonicalizes its result. This means the default `search_dir` and a user-supplied `path` resolve differently, and `strip_prefix` in the glob matching may behave inconsistently when the project root contains symlinks. | `tools.rs:355-358` | Canonicalize `search_dir` in the `None` branch as well, matching what `safe_path` returns. |
| 4 | minor | `tool_read_file` truncation at line 339 uses `read_exact` to fill a buffer of exactly `MAX_READ_BYTES`. If the file shrinks between the `metadata()` call (line 323) and the `read_exact` call (line 339), `read_exact` will return `UnexpectedEof`. This is a TOCTOU issue that surfaces as a confusing error message. | `tools.rs:331-346` | Use `read` (not `read_exact`) in a loop, or just `read_to_end` with a capped buffer, to handle files that shrink. |
| 5 | minor | `tool_write_file` creates directories (line 505) before calling `safe_path` (line 509). If `safe_path` subsequently rejects the path (e.g., a symlink target escapes root), empty directories have already been created as a side effect. This is not a security hole due to the `verify_ancestors_within_root` guard, but it leaves behind artifact directories on rejected writes. | `tools.rs:505-509` | Call `safe_path` (or at least the symlink check) before `create_dir_all`, or clean up created directories on rejection. |
| 6 | note | `tool_bash` returns `Ok(result)` even when the command exits with a non-zero code (line 621). The caller receives `is_error: false` for failed commands and must parse the `[exit code: N]` string to detect failure. This is intentional per the test at line 929, but downstream consumers that only check `is_error` will miss command failures. | `tools.rs:610-621` | Document this contract clearly, or consider setting `is_error = true` for non-zero exits so callers do not silently ignore failures. |
| 7 | note | `edit_file` uses `str::matches().count()` to detect ambiguity, then `replacen(..., 1)` to perform the edit. `matches` finds overlapping occurrences (e.g., `old_string = "aa"` in content `"aaa"` yields count=2), but `replacen` replaces the first non-overlapping occurrence. The ambiguity check may therefore reject edits that `replacen` would have performed unambiguously. This is a conservative false-positive, not a correctness bug, but may surprise callers. | `tools.rs:529-543` | Use a non-overlapping count (e.g., iterating with `find` + offset) to match `replacen` semantics. |

## Notes

- Finding 1 (TOCTOU symlink race) requires a concurrent malicious process within the project directory to exploit. In single-user developer environments the practical risk is low, but it matters if Epic is used in shared or CI environments.
- Finding 2 (glob filter bypass) is data-correctness only -- it may return extra grep results but does not enable path-traversal escapes because `WalkDir` is rooted within `search_dir`.
