# B3: Event and state coupling

## Summary
The event enum is reasonably shaped for the orchestrator's needs, but several variants exist solely for TUI worklog display while triggering unnecessary checkpoint saves, and some state mutations in the orchestrator silently skip event emission -- creating divergence between persisted state and what the TUI shows.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `VerificationStarted` is emitted (orchestrator.rs:478, 623, 755) but never updates any state in `EpicState` or `Task` -- the phase transition to `Verifying` is handled separately by `transition()`. This means the event exists purely for TUI worklog display, yet each emit site is paired with a `checkpoint_save()` shortly after (via `complete_task_verified` or subsequent logic). If the TUI is not attached, the event is silently discarded with no state effect. Conversely, `VerificationComplete{passed: true}` is emitted inside `complete_task_verified` (orchestrator.rs:172) right before `TaskCompleted`, meaning the TUI receives two events for one logical state change with no additional information the `TaskCompleted` event doesn't already carry. | `events.rs:44-50`, `orchestrator.rs:172-182`, `tui/mod.rs:247-265` | Collapse `VerificationStarted` into the existing `PhaseTransition{Verifying}` event which already fires. Make `VerificationComplete{passed: true}` implicit in `TaskCompleted{Success}` -- the TUI already sets `task.phase = Completed` on that event. Keep only `VerificationComplete{passed: false}` if the TUI needs to distinguish verification failure from execution failure. |
| 2 | major | `SubtasksCreated` is emitted redundantly alongside `RecoverySubtasksCreated` in `attempt_recovery` (orchestrator.rs:1156-1164). Both events fire for the same set of child IDs. The TUI handles them independently, producing two worklog entries for one logical action. More importantly, `create_subtasks` already emits `TaskRegistered` for each child and calls `checkpoint_save`, so `SubtasksCreated` adds a third event for the same operation. The only TUI consumer of `SubtasksCreated` is a worklog line -- the tree is already updated by `TaskRegistered`. | `orchestrator.rs:1148-1164`, `tui/mod.rs:198-209` | Remove the second `SubtasksCreated` emit after `RecoverySubtasksCreated` in `attempt_recovery`, or unify the two into a single event with an optional `recovery_round` field. |
| 3 | minor | `ModelSelected` event (events.rs:22-25) is emitted in `execute_task` (orchestrator.rs:454) but the TUI only appends a worklog line -- it does not store the model on `TuiTask`. Meanwhile, `RetryAttempt` and `FixAttempt` both carry a `model` field. These three could be a single variant with a discriminant, reducing match arms in the TUI (currently 3 separate arms that all call `worklog.push`). | `events.rs:22-25, 39-43, 61-65`, `tui/mod.rs:186-191, 237-246, 284-293` | Consider a single `AgentInvocation { task_id, kind: Initial/Retry/Fix, model, attempt }` variant. |
| 4 | minor | `ModelEscalated` and `FixModelEscalated` are structurally identical (both carry `task_id, from, to`) but are separate variants. The TUI handles them with nearly identical code (both produce a `WorklogEntry::warn`). The orchestrator emits `ModelEscalated` for leaf execution escalation (orchestrator.rs:795) and `FixModelEscalated` for leaf fix escalation (orchestrator.rs:543, 653). The distinction adds two match arms across events.rs and tui/mod.rs for no state or display difference. | `events.rs:26-30, 66-70`, `orchestrator.rs:543-548, 653-657, 795-799`, `tui/mod.rs:192-196, 294-298` | Merge into `ModelEscalated { task_id, from, to, context: Execution/Fix }` or drop the distinction entirely since the worklog text is the only consumer. |
| 5 | minor | Full-redecomposition recovery in `attempt_recovery` (orchestrator.rs:1131-1145) sets `child.phase = TaskPhase::Failed` directly on state and emits `TaskCompleted{Failed}` but does NOT call `transition()`, so no `PhaseTransition{Failed}` event fires for those children. The TUI's `handle_event` for `TaskCompleted` defensively sets phase (tui/mod.rs:214), masking the gap, but the worklog never shows a phase transition line for these tasks -- they jump from Pending to Failed in the tree without a worklog entry. | `orchestrator.rs:1131-1145`, `tui/mod.rs:210-236`, `state.rs` (no event coupling) | Route through `fail_task()` or `transition()` so that `PhaseTransition{Failed}` fires before `TaskCompleted`, matching the pattern used everywhere else. |
| 6 | note | `EpicState` (state.rs) is entirely event-unaware -- it has no subscription to the event channel and no event-triggered persistence. All checkpoint saves are manually placed in the orchestrator after `emit()` calls. This means adding a new event variant requires the developer to remember to also add a `checkpoint_save()` call at the right point in the orchestrator. There is no compile-time or runtime guard against omitting persistence after a state-changing event. For example, `CheckpointAdjust` modifies `task.checkpoint_guidance` (orchestrator.rs:970-975) and calls `checkpoint_save`, but `RecoveryPlanSelected` (orchestrator.rs:1117-1120) does not mutate state at all -- it is purely informational. The inconsistency is not a bug today, but the pattern is fragile as the event set grows. | `state.rs`, `orchestrator.rs:137-147` | Document the convention (state-mutating events must be followed by `checkpoint_save`) or introduce a `emit_and_save()` helper that bundles both actions for state-mutating events. |

## Notes
- The event channel is unbounded (`mpsc::unbounded_channel`), so high-frequency TUI-only events do not cause backpressure, but they do allocate and clone data (e.g., `goal: String` in `TaskRegistered`) that only the TUI consumes. If the TUI is disabled, this is pure waste. Not flagged as a finding because the overhead is marginal at current scale.
- `EpicState::save` uses atomic rename (write-to-tmp then rename), so checkpoint overhead per save call is one full JSON serialization + one fsync-equivalent. The events themselves do not trigger saves; only the orchestrator's manual `checkpoint_save()` calls do. The concern is therefore about unnecessary saves rather than unnecessary events.
- Several events carry cloned `String` or `Vec` data that is already present in the persisted `Task` struct. The TUI maintains a parallel `TuiTask` struct that stores a subset. This duplication is a design trade-off (decoupling TUI from state internals) rather than a defect.
