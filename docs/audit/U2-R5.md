# U2-R5: agent/flick / Simplification

## Summary

The file is well-structured overall, but contains duplicated timeout/error-mapping boilerplate across call sites and several AgentService methods that follow nearly identical patterns differing only in config builder and wire type, which could be consolidated.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | Timeout-and-map-error pattern is duplicated three times: once in `run_structured`, once for the initial call in `run_with_tools`, and once for each resume in the tool loop. All three share identical `.await.map_err(...).map_err(...)` chains. | `flick.rs:69-75`, `flick.rs:98-104`, `flick.rs:129-135` | Extract a private helper, e.g. `async fn timed_call<F, T>(&self, fut: F) -> anyhow::Result<T>` that wraps `tokio::time::timeout` with the standard error mapping. |
| 2 | minor | `execute_leaf` and `fix_leaf` have identical bodies after the prompt-building line: same `phase_tools`, same `build_execute_leaf_config`, same `run_with_tools::<TaskOutcomeWire>`, same `LeafResult::try_from`. | `flick.rs:208-248` | Extract a shared `run_leaf(&self, pair, model) -> Result<LeafResult>` helper and call it from both methods. |
| 3 | minor | `design_and_decompose` and `design_fix_subtasks` share the same config-build/run/convert tail (both use `build_decompose_config`, `DecompositionWire`, `DecompositionResult::try_from`). | `flick.rs:250-289` | Extract a shared `run_decompose(&self, pair, model) -> Result<DecompositionResult>` helper. |
| 4 | note | `extract_text` manually iterates with a `for` loop and a mutable `last_text` variable to find the last text block. | `flick.rs:378-388` | Replace with `result.content.iter().rev().find_map(...)` for a more idiomatic one-liner: `result.content.iter().rev().find_map(\|b\| match b { flick::ContentBlock::Text { text } => Some(text.clone()), _ => None }).context(...)`. |
| 5 | note | `explore_for_init` embeds a multi-line system prompt string directly in the method body, unlike every other agent call which delegates to the `prompts` module. | `flick.rs:159-173` | Move the init-exploration prompt into `prompts` for consistency, keeping all prompt text in one place. |
| 6 | note | The `build_client` method wraps the provider error with `map_err` into `anyhow::anyhow!(...)`, but the surrounding functions already use `anyhow::Context`. | `flick.rs:51-53` | Use `.context("failed to resolve provider")` instead of `.map_err(\|e\| anyhow::anyhow!("failed to resolve provider: {e}"))` for consistency and to preserve the original error chain. |

## Notes

- Findings 1-3 represent real boilerplate reduction opportunities that would shrink the file meaningfully and reduce the chance of divergent behavior between parallel code paths (e.g., if timeout handling needs to change, it currently must be updated in three places).
- The AgentService trait methods are inherently repetitive by design (each maps a different phase), so some repetition is expected. The suggestion is to extract the shared tail within groups (leaf methods, decompose methods) rather than to abstract all methods into a single generic dispatcher, which would hurt readability.
- The file is otherwise clean: good separation between core call mechanics and the trait impl, helper functions are well-scoped, and tests cover the key helpers.
