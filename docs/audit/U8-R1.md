# U8-R1: state / Correctness

## Summary

`EpicState` is a compact module with sound core logic, but lacks post-deserialization validation, which can lead to duplicate `TaskId` generation from corrupted or hand-edited state files. The `dfs_order` traversal is also vulnerable to infinite loops if the task graph ever contains a cycle.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `load` does not validate that `next_id` exceeds all existing `TaskId` values in the deserialized map. A corrupted or manually edited state file with `next_id` lower than existing task IDs will cause `next_task_id` to hand out duplicate IDs, silently corrupting the task tree. | `state.rs:74-78` (`load`) | After deserializing, iterate `tasks.keys()` and assert `next_id > max(task_id.0)`. |
| 2 | major | `dfs_order` has no cycle detection. If a bug elsewhere creates a cycle in `subtask_ids` (e.g., a task listing itself or an ancestor as a child), this function will loop forever, consuming unbounded memory. | `state.rs:40-53` (`dfs_order`) | Add a `HashSet<TaskId>` visited guard, or cap iterations at `self.tasks.len()`. |
| 3 | minor | `insert` silently overwrites an existing task if called with a duplicate `TaskId`. There is no check that the ID was freshly generated by `next_task_id` or that it does not collide with an existing entry. | `state.rs:27-29` (`insert`) | Return an error or use `entry` API to reject duplicate inserts, or at least `debug_assert` uniqueness. |
| 4 | minor | `load` does not validate parent/child referential integrity. A deserialized state could have a task whose `parent_id` points to a nonexistent `TaskId`, or a parent whose `subtask_ids` references a missing child. Downstream code calling `get(parent_id).unwrap()` would panic. | `state.rs:74-78` (`load`) | Add a post-load consistency check that all `parent_id` and `subtask_ids` references resolve to entries in the map. |
| 5 | note | `next_task_id` will panic on u64 overflow in debug builds (arithmetic overflow) and silently wrap in release builds, producing a duplicate `TaskId(0)`. Practically unreachable but theoretically unsound. | `state.rs:21-25` (`next_task_id`) | Use `checked_add` and return `Result` or `Option`, or add a debug assertion. |

## Notes

- The `const fn` on `next_task_id` and `set_root_id` compiles on the project's Rust 1.93 toolchain (const mutable references are stabilized), so no compilation issue exists.
- The atomic-save pattern (`write` to `.tmp` then `rename`) is correct for Unix. On Windows, `std::fs::rename` is not guaranteed atomic if the destination already exists, but this project appears Unix-targeted.
- The silent `let _ = std::fs::remove_file(&tmp_path)` on rename failure (line 68) is intentional best-effort cleanup and is fine.
