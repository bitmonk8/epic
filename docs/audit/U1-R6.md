# U1-R6: orchestrator / Testability

## Summary

The orchestrator has a well-designed mock boundary via the `AgentService` trait and an extensive test suite (35+ tests, ~3300 lines) covering happy paths, retry/escalation, resume, recovery, and checkpoint flows. Key testability concerns are the hard-coded constants that prevent edge-case testing, the `check_scope_circuit_breaker` method's direct dependency on `tokio::process::Command` (shelling out to `git`), and verbose mock setup that creates friction for writing new tests.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `check_scope_circuit_breaker` shells out to `git diff --numstat` via `tokio::process::Command` directly, bypassing the `AgentService` trait boundary. This makes it impossible to test scope-exceeded code paths without a real git repo on disk. The existing tests only cover the no-magnitude and git-failure fallback cases, never the `ScopeCheck::Exceeded` branch within `leaf_fix_loop` or `branch_fix_loop`. | `orchestrator.rs:345-356` | Extract a trait or closure for running git commands (e.g., `trait GitService { async fn diff_numstat(&self) -> Option<String> }`) and inject it into `Orchestrator`, or move the subprocess call behind `AgentService`. This would allow tests to simulate scope exceeded mid-fix-loop. |
| 2 | minor | Constants `MAX_DEPTH`, `RETRIES_PER_TIER`, `MAX_BRANCH_FIX_ROUNDS`, `MAX_ROOT_FIX_ROUNDS`, and `MAX_RECOVERY_ROUNDS` are module-level `const` values. Tests must exercise the full retry budget (e.g., 9 leaf attempts to exhaust all tiers) rather than testing boundary behavior with smaller values. This inflates test setup and makes tests fragile if constants change. | `orchestrator.rs:15-19` | Move these into an `OrchestratorConfig` struct passed at construction time. Tests can then use small values (e.g., `RETRIES_PER_TIER=1`) to reduce mock boilerplate and isolate boundary logic. |
| 3 | minor | `MockAgentService` uses `Mutex<VecDeque<T>>` queues that panic on underflow with generic messages like `"no assess response queued"`. When a test miscounts the expected call sequence, the panic trace does not indicate which call number failed or what the orchestrator state was. This creates debugging friction. | `orchestrator.rs:1183-1330` | Add call counters or richer panic messages (e.g., include the call count and task ID). Alternatively, adopt a mock framework or builder pattern that validates expected call counts at test teardown. |
| 4 | minor | The `MockAgentService` discards all input arguments (`_ctx`, `_model`, `_failure_reason`, etc.). No test asserts that the orchestrator passes the correct context, model tier, or failure reason to the agent. A bug where the wrong model is forwarded to `fix_leaf` or the wrong failure reason is passed to `design_fix_subtasks` would go undetected. | `orchestrator.rs:1213-1330` | Record inputs in a `Vec` behind a `Mutex` (call log) and assert on them in tests. At minimum, verify that escalated model tiers reach the agent and that failure reasons are forwarded correctly. |
| 5 | minor | `checkpoint_save` swallows write errors via `eprintln!` (best-effort). There is no test verifying behavior when state persistence fails mid-run -- e.g., confirming the orchestrator continues rather than aborting, and that the in-memory state remains consistent. | `orchestrator.rs:137-143` | Add a test with a state path pointing to a read-only directory (or inject a fallible persistence layer) to verify graceful degradation on write failure. |
| 6 | note | State machine transitions are implicit -- `TaskPhase` is set directly on the `Task` struct via `transition()` with no guard against invalid transitions (e.g., `Completed -> Executing`). Tests rely on the orchestrator's control flow to enforce valid sequences but there is no standalone state machine validation. | `orchestrator.rs:149-157` | Consider adding a `Task::try_transition(new_phase)` method that validates the from/to pair and returns `Result`. This would make invalid transitions detectable in tests and prevent regressions if new code paths are added. |
| 7 | note | The `execute_task` method returns `Pin<Box<dyn Future<...>>>` to support recursive async calls. This erases the concrete future type, which means the compiler cannot verify `Send` bounds at the call site as precisely. While this is a necessary trade-off for async recursion, it is worth noting that any future refactor introducing non-`Send` state inside the boxed future would only be caught at runtime. | `orchestrator.rs:360-363` | No action needed currently. If the recursion depth is bounded (it is, by `MAX_DEPTH`), consider whether an iterative approach with an explicit stack could replace the recursive boxing in a future refactor. |
| 8 | note | Test coverage is thorough for the happy path, retry, escalation, resume, recovery, and checkpoint-adjust/escalate flows. However, there is no test for the `build_context` method's sibling aggregation logic in isolation -- it is only tested indirectly through full `run()` invocations. A targeted unit test for `build_context` with various sibling states would improve confidence and be cheaper to maintain. | `orchestrator.rs:240-324` | Add focused tests that construct `EpicState` with specific sibling configurations and call `build_context` directly, asserting on the `TaskContext` fields (ancestor goals, completed/pending siblings, checkpoint guidance). |

## Notes

- The `AgentService` trait is a clean seam: it is generic (not `dyn`), covers all agent interactions, and the mock implementation is co-located with tests. This is a strong design for testability.
- The `evaluate_scope` pure function is properly unit-tested in isolation (lines 2579-2636), which is a good practice the rest of the module could emulate for `build_context`.
- Test verbosity is high due to the queue-based mock pattern requiring explicit setup of every response in sequence. A builder or scenario helper could reduce boilerplate significantly -- the existing `make_orchestrator`, `leaf_assessment`, `leaf_success` helpers are a good start but cover only the simplest cases.
- The `EventReceiver` is used in several tests to assert event emission, which is a useful secondary verification channel. However, event assertions are scattered and inconsistent -- some tests check events, most do not.
