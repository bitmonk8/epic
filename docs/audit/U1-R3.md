# U1-R3: orchestrator / Error handling

## Summary

The orchestrator has strong error-handling discipline overall -- nearly all fallible operations use `?` propagation and agent-call failures in non-critical paths (checkpoint, recovery) degrade gracefully. A handful of issues remain: one non-idiomatic `unwrap`, an `unreachable!` that could panic on a future enum variant, silently swallowed event-send errors, and missing structured logging for degraded-path warnings.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `task.path.clone().unwrap()` after an `is_some()` guard. Logically safe but non-idiomatic; a refactor that separates the guard from the use could introduce a panic. | `orchestrator.rs:400` | Replace with `if let Some(path) = task.path.clone() { ... }` or use `let else`. |
| 2 | minor | `unreachable!("non-Success outcome must be Failed")` assumes `TaskOutcome` is a two-variant enum. If a third variant is ever added, this panics at runtime instead of producing a compile error. | `orchestrator.rs:512` | Use an exhaustive `match` on `TaskOutcome` instead of a `let else` + `unreachable!`. |
| 3 | note | `let _ = self.events.send(event);` silently discards send errors. If the event receiver is dropped (e.g., TUI crash), every subsequent event is silently lost with no diagnostic. | `orchestrator.rs:146` | Log a warning (once or at a throttled rate) when send fails, or set a flag to stop emitting. |
| 4 | note | Degraded-path warnings use raw `eprintln!` (lines 140, 959, 1065, 1099) instead of structured logging. These messages are invisible when stderr is redirected or when a TUI is active. | `orchestrator.rs:140,959,1065,1099` | Route through a logging framework (e.g., `tracing::warn!`) so they appear in log files and are filterable. |
| 5 | note | `check_scope_circuit_breaker` treats any `git diff` failure (including git-not-found or corrupt repo) as `WithinBounds`, silently skipping the scope check. | `orchestrator.rs:351-352` | Distinguish transient failures (return `WithinBounds`) from permanent ones (log a warning or return an error). |
| 6 | note | Agent errors from `assess`, `verify`, `execute_leaf`, `design_and_decompose`, and `design_fix_subtasks` all propagate via `?` and abort the entire run. Only `checkpoint`, `assess_recovery`, and `design_recovery_subtasks` degrade gracefully. A transient API error during verification or leaf execution terminates the run with no retry at the orchestrator level. | `orchestrator.rs:435,481,625,757,827,902` | Consider catching transient agent errors (rate-limit, timeout) at the orchestrator level and retrying with backoff before aborting. |

## Notes

- All `unwrap()` and `expect()` calls beyond line 1171 are inside `#[cfg(test)]` and are appropriate for test code.
- The single `panic!` at line 2611 is also in test code (asserting an expected enum variant).
- Resource cleanup is handled implicitly: the orchestrator owns `EpicState` by value and checkpoints to disk at every significant state transition, so a crash at any point can resume from the last checkpoint. No explicit `Drop` impl is needed.
- Error propagation is consistent: `OrchestratorError::TaskNotFound` is used uniformly for missing task lookups, and `anyhow::Error` from agent calls is wrapped via the `#[from]` derive.
