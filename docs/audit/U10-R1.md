# U10-R1: config / Correctness

## Summary

The config module is small and mostly declarative (serde structs + defaults). The main correctness concerns are: no validation of user-supplied values, no loading of the config file at runtime (the module comment promises loading from `~/.config/epic/config.toml` but no code does so), and a verification step with an empty command vector would be silently accepted.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `mod.rs` comment states "Configuration loading: ~/.config/epic/config.toml + project epic.toml" but the module contains no loading logic. `epic.toml` is only read/written in `init.rs`; the `run` command path in `main.rs` never parses `epic.toml`, so the config structs are effectively dead code at runtime (only used during `init`). Any user edits to `epic.toml` after init are silently ignored. | `src/config/mod.rs:1`, `src/main.rs` | Add a `load()` function in `mod.rs` that reads and parses `epic.toml` (and optionally the global config), and call it from the `run` path in `main.rs`. |
| 2 | major | `max_depth`, `max_recovery_rounds`, and `retry_budget` accept `u32` with no validation. A value of `0` for `max_depth` would likely cause the orchestrator to immediately reject any task tree, and `0` for `retry_budget` would disable retries entirely. These are likely unintended configurations. Similarly, extremely large values could cause runaway execution. | `src/config/project.rs:43-50` | Add validation (e.g., `1..=32` for depth, `1..=10` for budgets) either via a custom deserializer or a post-parse `validate()` method. |
| 3 | minor | `VerificationStep` requires `name` and `command` but does not validate that `command` is non-empty. An empty `command = []` would deserialize successfully but would fail or behave unexpectedly when passed to `std::process::Command` (which requires at least a program name). | `src/config/project.rs:58-64` | Add a `#[serde(deserialize_with = ...)]` or post-parse check that `command.len() >= 1`. |
| 4 | minor | `timeout` on `VerificationStep` is `u32` (seconds). A user could set `timeout = 0`, which would cause an immediate timeout on any verification command. There is no lower-bound check. | `src/config/project.rs:63` | Validate that timeout is at least 1 second, or document that 0 means "no timeout". |
| 5 | note | `ProjectConfig::root` defaults to `"."` (a relative path). If the working directory changes between config load and use, the root will resolve to the wrong location. This is safe today because the config is not loaded at runtime, but will become a bug once loading is implemented (see finding 1). | `src/config/project.rs:68` | Canonicalize `root` to an absolute path at load time, or document that it is relative to the project directory containing `epic.toml`. |
| 6 | note | The global config path (`~/.config/epic/config.toml`) mentioned in the module comment has no corresponding struct or merge logic. If both a global and project config are eventually supported, field-level merging (project overrides global) needs to be implemented; a naive approach would silently discard one config entirely. | `src/config/mod.rs:1` | When implementing global config, use a layered merge strategy (e.g., global defaults < project overrides). |

## Notes

- The config module is currently only exercised during `epic init`. The `run` / orchestrator path hard-codes its own constants (e.g., `MAX_ROOT_FIX_ROUNDS`, `MAX_BRANCH_FIX_ROUNDS` in `orchestrator.rs`) rather than reading from `EpicConfig`. This means the config structs define defaults that are never actually consulted at runtime, which is the most significant correctness gap.
- All serde `Default` impls are consistent with the `default_*` functions used in `#[serde(default = "...")]` attributes, so round-tripping is correct (confirmed by existing test).
