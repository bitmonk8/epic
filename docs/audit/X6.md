# X6: Constants vs. config

## Summary
The orchestrator defines five hardcoded constants that duplicate (and potentially shadow) values already configurable via `epic.toml`. Several agent-layer tool limits are also hardcoded with no configuration path, which restricts operators from tuning behavior for large or constrained projects.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `MAX_DEPTH` (8) duplicates `LimitsConfig::max_depth` but the orchestrator uses the hardcoded constant instead of reading from config. If a user sets `limits.max_depth` in `epic.toml`, it is silently ignored. | `src/orchestrator.rs:15` | Remove the constant; thread `LimitsConfig::max_depth` into the orchestrator. |
| 2 | major | `RETRIES_PER_TIER` (3) duplicates `LimitsConfig::retry_budget` but the orchestrator uses the hardcoded constant. Config value is ignored at runtime. | `src/orchestrator.rs:16` | Remove the constant; use `config.limits.retry_budget`. |
| 3 | major | `MAX_RECOVERY_ROUNDS` (2) duplicates `LimitsConfig::max_recovery_rounds` but the orchestrator uses the hardcoded constant. Config value is ignored at runtime. | `src/orchestrator.rs:19` | Remove the constant; use `config.limits.max_recovery_rounds`. |
| 4 | minor | `MAX_BRANCH_FIX_ROUNDS` (3) and `MAX_ROOT_FIX_ROUNDS` (4) are not exposed in config. These control how many verification-fix cycles are allowed before giving up. Operators with expensive verification steps may want to reduce these; operators with fast test suites may want to increase them. | `src/orchestrator.rs:17-18` | Add `max_branch_fix_rounds` and `max_root_fix_rounds` to `LimitsConfig`. |

## Notes
- Findings 1-3 are the most pressing: `epic.toml` exposes `limits.max_depth`, `limits.retry_budget`, and `limits.max_recovery_rounds`, yet the orchestrator never reads them, using its own `const` values instead. This means the config fields are dead code from the user's perspective -- a silent contract violation.
- The default values of the hardcoded constants happen to match the config defaults (`default_max_depth()`, `default_retry_budget()`, `default_max_recovery_rounds()`), so behavior is correct only when using defaults. Any user override is silently dropped.
- Findings 4-6 are enhancement suggestions for values that power users may reasonably want to tune.
- Findings 7-10 are constants that serve as internal safety limits or display parameters and are appropriate as hardcoded values.
