# U17-R7: docs / Design intent

## Summary

The documentation tells a coherent and internally consistent design story of a recursive problem-solver orchestrator. The core model (assess/execute/verify with leaf/branch paths) is described consistently across all documents. However, there are several stale references and minor contradictions left over from the subprocess-to-library migration that create confusion about the current integration mode.

## Notes

The stale subprocess references (findings 1-5) are a predictable artifact of the Flick library migration -- the migration spec (FLICK_LIBRARY_MIGRATION.md) and STATUS.md were updated, but several upstream design documents were not swept for consistency. None of these stale references indicate a design confusion; they are documentation lag.

The recursive problem-solver model is described consistently across OVERVIEW.md, TASK_MODEL.md, AGENT_DESIGN.md, VERIFICATION.md, and FIX_LOOP_SPEC.md. The three-phase lifecycle (assess/execute/verify), the leaf/branch path dichotomy, the recovery hierarchy (retry, escalate, fail-to-parent, Opus recovery), and the discovery/checkpoint flow are all mutually reinforcing with no contradictions detected.

Cost control mechanisms are individually well-specified: model tier escalation (cheapest first), retry budgets (3 per tier), scope circuit breaker (3x magnitude), depth cap (8), branch fix round limits (3+1 for root), recovery round limits (2), and sequential execution (deliberate throughput constraint). These are consistent across documents but have no single consolidated view.
