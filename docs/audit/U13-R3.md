# U13-R3: tui / Error handling

## Summary
The TUI module demonstrates strong error handling overall: no `unwrap()` or `expect()` calls exist, `Result` is propagated with `?` through the main path, and a panic hook restores the terminal. A few minor robustness gaps remain around terminal cleanup on error and the panic hook lifecycle.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `disable_raw_mode()?` and `crossterm::execute!(..., LeaveAlternateScreen)?` on the cleanup path (lines 77-78 of `mod.rs`) use `?`, so if `disable_raw_mode` fails, `LeaveAlternateScreen` is never called and the terminal is left in alternate-screen mode. | `src/tui/mod.rs:77-78` | Run both cleanup calls unconditionally (ignore or log the first error), similar to the panic hook which already uses `let _ =`. |
| 2 | minor | The panic hook captures and moves the original hook via `take_hook`, but if a second panic occurs inside the TUI event loop (e.g. in a `render` call), the hook performs terminal cleanup correctly. However, after `run()` returns, `take_hook()` on line 76 discards the cleanup-wrapping hook and replaces it with the default hook. If any code panics between `take_hook` (line 76) and the completion of lines 77-78, the terminal will not be restored. | `src/tui/mod.rs:60-78` | Defer terminal restoration into a drop guard (RAII struct) so cleanup is guaranteed regardless of panic timing. |
| 3 | note | `task.depth as usize` (task_tree.rs line 59) performs a u32-to-usize cast for indentation. With a pathological depth, this could allocate a very large indent string via `"  ".repeat(...)`. In practice, task depths are bounded by decomposition logic, so this is theoretical. | `src/tui/task_tree.rs:59` | Consider clamping depth to a reasonable maximum (e.g., 32) before computing indentation. |
| 4 | note | The `event_loop` never returns on its own when the orchestrator disconnects (`orchestrator_done = true`); it continues looping until the user presses 'q' or Ctrl-C. This is intentional (lets the user inspect final state), but there is no visible indication that further waiting is futile. | `src/tui/mod.rs:118-128` | The header already shows "DONE", so this is informational only. No change required unless auto-exit is desired. |

## Notes
- Zero `unwrap()` / `expect()` calls across all four files -- all fallible operations either use `?` or are guarded with `if let` / `let ... else`.
- The worklog is capped at 10,000 entries with an eviction drain, preventing unbounded memory growth.
- Widget `render()` implementations are infallible by trait design (no `Result` return), and all four widgets avoid any panicking operations internally.
- HashMap lookups throughout use `get()` / `get_mut()` with pattern matching rather than indexing, so missing keys are handled gracefully.
