# U13-R1: tui / Correctness

## Summary
The TUI code is generally well-structured with defensive clamping and overflow-safe arithmetic. A few edge cases around scroll bounds, event-loop exit conditions, and truncation logic warrant attention.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `tree_scroll` can reach `self.tasks.len()` on Down-arrow (line 110 in mod.rs), but `max_scroll` in `TaskTreeWidget::render` is `lines.len().saturating_sub(inner.height)`. The tree line count can differ from `tasks.len()` (e.g., when `root_id` is `None` the fallback yields 1 line, or orphan tasks not reachable from root are omitted). This means the stored scroll value may exceed the actual max, though it is later clamped at render time so no panic occurs -- but it does mean the Down key overshoots, requiring many Up presses to scroll back. | `src/tui/mod.rs:110`, `src/tui/task_tree.rs:122-123` | Clamp `tree_scroll` against the actual rendered line count, or compute the tree line count once and share it between input handling and rendering. |
| 2 | minor | The event loop never exits when the orchestrator channel disconnects (`orchestrator_done = true`). The loop continues spinning, drawing frames and polling crossterm indefinitely. The user must press `q` to quit. This is intentional (allows reading final state) but if the TUI is launched non-interactively or the terminal is detached, it will spin forever at 50 ms polls. | `src/tui/mod.rs:88-128` | Consider an auto-exit mode or a timeout after orchestrator disconnect, especially for headless/CI usage. |
| 3 | minor | `prefix_len` calculation in `render_tree_lines` (line 64) counts `indent.len() + 2 + cursor.len()`. The `+2` accounts for the indicator character plus a space (`"X "`), but multi-byte UTF-8 indicators (e.g., the three-byte characters used for the phase indicators) occupy only one terminal column each. Since `indent.len()` is in bytes (ASCII spaces, so bytes == columns) and `cursor.len()` is also ASCII, the only mismatch is the indicator. The byte length of the indicator (3 for the Unicode chars) overstates its display width by 2, but it is not included in `prefix_len` -- only the hardcoded `2` is. So the `+2` is correct for display columns (1 indicator column + 1 space). No bug here, but the code is fragile: if the indicator were changed to a wider emoji, the math would silently break. | `src/tui/task_tree.rs:64` | Add a comment clarifying the column-width assumption, or use `unicode-width` for robustness. |
| 4 | minor | When `follow_tail` is toggled off, the worklog always renders from index 0 with no user-controllable scroll offset. The user can disable tail-follow but cannot scroll the worklog, so entries beyond the first screen-full are unreachable until tail-follow is re-enabled and new entries push the window forward. | `src/tui/worklog.rs:65-68`, `src/tui/mod.rs:99-101` | Add dedicated worklog scroll keys (e.g., PgUp/PgDn) when `follow_tail` is off. |
| 5 | minor | `WorklogEntry` stores both a raw `timestamp: Instant` field and a pre-formatted `[...s]` prefix baked into `text`. The `timestamp` field is marked `#[allow(dead_code)]` and is never read after construction. If structured log export is eventually added, the pre-formatted `text` would need to be re-parsed or a separate raw message field added. | `src/tui/worklog.rs:8-12` | Store the raw message separately from the formatted display string, or remove the unused `timestamp` field to avoid confusion. |
| 6 | note | The panic hook installed in `run()` captures the original hook by value and is never explicitly restored to the original on the normal-exit path. `take_hook()` on line 76 discards the cleanup wrapper (which itself holds the original hook). This is fine for normal program exit but means that if `run()` is called more than once in the same process, each call nests another layer of hooks. | `src/tui/mod.rs:60-76` | Restore the original hook explicitly rather than discarding the wrapper via `take_hook()`. |
| 7 | note | `worklog.drain(..drain)` after every event (line 347) is O(n) due to shifting the remaining elements. With `MAX_WORKLOG_ENTRIES = 10_000` this is unlikely to be a bottleneck in practice, but a `VecDeque` would make eviction O(1). | `src/tui/mod.rs:345-348` | Consider using `VecDeque` for the worklog buffer. |

## Notes
- No panics or undefined behavior were identified; the code uses safe Rust throughout.
- All `saturating_sub` / `.min()` usage is correct, preventing arithmetic underflow.
- The DFS traversal in `render_tree_lines` correctly reverses children before pushing onto the stack, producing the expected top-down order.
- `HashMap` iteration order is non-deterministic, but the tree rendering uses explicit DFS from `root_id` via `subtask_ids`, so display order is stable and correct.
- The `or_insert` on `TaskRegistered` means duplicate registration events are silently ignored (the existing task is kept). This is a reasonable defensive choice.
