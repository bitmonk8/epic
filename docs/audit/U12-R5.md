# U12-R5: cli + main / Simplification

## Summary
`main.rs` contains significant duplicated state-loading and validation logic between the `Run` (with existing state) and `Resume` branches, as well as repeated error-handling boilerplate that could be extracted into helper functions. `cli.rs` is clean and minimal.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Duplicated state-load-and-validate sequence between `Command::Run` (existing state path, lines 66-104) and `Command::Resume` (lines 125-156). Both perform identical `EpicState::load`, `root_id()`, `state.get(root_id)` with near-identical error messages. | `src/main.rs:66-156` | Extract a helper like `fn load_and_validate_state(path) -> Result<(EpicState, TaskId, String)>` that loads state, extracts root id, and retrieves the goal. Both branches call it, with `Run` adding its goal-mismatch check on top. |
| 2 | major | Identical `EpicState::load` + `eprintln!` + `process::exit(1)` pattern repeated three times (lines 66-75, 125-133, 214-222) across `Run`, `Resume`, and `print_status`. | `src/main.rs:66,125,214` | Create a helper `fn load_state_or_exit(path: &Path) -> EpicState` that encapsulates the load-or-die pattern, eliminating all three copies. |
| 3 | minor | `#[allow(clippy::too_many_lines)]` suppression on `main` is a code smell indicating the function should be decomposed. | `src/main.rs:26` | Split command dispatch into dedicated functions (e.g., `handle_run`, `handle_resume`) so `main` becomes a thin dispatcher and the lint suppression can be removed. |
| 4 | minor | Repeated `eprintln!(...); std::process::exit(1)` pattern (6 occurrences) instead of using `anyhow::bail!` or returning typed errors. The function already returns `anyhow::Result<()>`, so early-exit via `bail!` or a custom error would be idiomatic and testable. | `src/main.rs:42-47,69-74,79-83,89-94,98-101,128-133` | Replace `eprintln!` + `process::exit(1)` with `anyhow::bail!()` to leverage the existing `Result` return type. This also makes the logic unit-testable. |
| 5 | minor | `project_root.clone()` is passed twice (lines 56 and 164) and `state_path.clone()` once (line 163). With decomposition into sub-functions these clones can be reduced or eliminated by passing references. | `src/main.rs:56,163-164` | After extracting handler functions, pass `&Path` references instead of cloning `PathBuf` values. |
| 6 | note | The `Init` command is checked with two separate `matches!` guards (lines 39 and 59) with agent construction sandwiched between them. This is functional but splits a single command's logic across two distant blocks. | `src/main.rs:39-61` | Consider an early-return `handle_init` function called before agent construction, combining both checks and the `init::run_init` call in one place. |

## Notes
- `cli.rs` (32 lines) is well-structured with no simplification opportunities; all findings are in `main.rs`.
- The duplicated state-validation logic (findings 1 and 2) is the highest-value refactor target since it would remove roughly 40 lines of near-identical code.
- Replacing `process::exit` with `bail!` (finding 4) would additionally make the logic amenable to integration testing without process termination side effects.
