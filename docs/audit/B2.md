# B2: Data flow and type conversions

## Summary
The domain-to-wire-to-domain pipeline is clean in structure but carries unnecessary data at several points: the full `Task` struct is cloned into `TaskContext` though only 3 of 21 fields are consumed by prompts, multiple wire-to-domain conversions produce fields the orchestrator never reads, and one agent call (`assess`) includes tool definitions in its config despite using the tool-less `run_structured` path.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `assess` builds config with tool definitions (`build_assess_config` accepts a `ToolGrant` and embeds tools) but dispatches via `run_structured`, which has no tool-call loop. If the LLM returns a `ToolCallsPending` response, `run_structured` will fail trying to extract a text block. The config and the dispatch path disagree on whether tools are available. | `flick.rs:194-205` builds config with tools; `flick.rs:61-82` (`run_structured`) ignores tool calls; `config_gen.rs:375-384` (`build_assess_config`) accepts `ToolGrant` | Either switch `assess` to `run_with_tools` so the agent can explore the codebase before deciding leaf/branch, or remove the tool grant from `build_assess_config` to avoid sending tool definitions to a no-tool call path. |
| 2 | minor | `TaskContext` clones the entire 21-field `Task` struct, but `format_context` in `prompts.rs` only reads `task.goal`, `task.verification_criteria`, and `task.depth`. The remaining 18 fields (model, attempts, subtask_ids, magnitude, fix_attempts, etc.) are serialized into the context bundle but never reach any prompt or agent decision. | `orchestrator.rs:240-324` (`build_context`) clones full `Task`; `prompts.rs:12-77` reads 3 fields | Introduce a lightweight `TaskSummary` (goal, criteria, depth) for `TaskContext` instead of carrying the full `Task`. This also avoids accidentally leaking internal state into future prompt changes. |
| 3 | minor | `AssessmentResult.rationale` is deserialized from `AssessmentWire` and returned to the orchestrator, which never reads it. The field is silently dropped after the `TryFrom` conversion. | `config_gen.rs:28-52` produces `AssessmentResult` with `rationale`; `orchestrator.rs:438-448` reads only `path`, `model`, `magnitude` | Either remove `rationale` from `AssessmentResult` (keep it wire-only for logging) or store it on `Task` the way `decomposition_rationale` is stored, so there is a consistent policy. |
| 4 | minor | `VerificationResult.details` is deserialized from `VerificationWire` but the orchestrator only pattern-matches on `.outcome` (Pass/Fail). The `details` string is allocated, returned, and dropped without being read or logged. | `config_gen.rs:127-142` produces `VerificationResult` with `details`; `orchestrator.rs:483,627,759` match only on `.outcome` | Either log/emit `details` for observability (it contains the verifier's rationale) or drop it from the domain type to make the discard explicit. |
| 5 | minor | `RecoveryPlan.rationale` is deserialized from `RecoveryPlanWire` but the orchestrator reads only `.full_redecomposition` and `.subtasks`. The field already has `#[allow(dead_code)]`. | `config_gen.rs:183-202` produces `RecoveryPlan` with `rationale`; `orchestrator.rs:1106-1155` uses only `subtasks` and `full_redecomposition` | Same as #3/#4: either emit/log the rationale for debugging, or remove it from the domain type. |
| 6 | note | `decomposition_rationale` is stored on `Task` (from `DecompositionResult.rationale`) but never read back by any code path -- not by prompts, not by the orchestrator, not by events. It survives only in serialized state files. | `orchestrator.rs:909` writes it; `task/mod.rs:117` declares it; no read sites found | Decide whether this is intentional persistence for post-hoc debugging or dead storage. If the former, document it; if the latter, remove it. |
| 7 | note | `SiblingSummary.id` (a `TaskId`) is populated by `build_context` in the orchestrator but never referenced in `format_context` or any prompt template. It carries `#[allow(dead_code)]`. | `orchestrator.rs:280,296` sets `id`; `agent/mod.rs:18` declares it with dead-code allow; `prompts.rs:31-49` formats siblings without using `id` | Remove the field or include it in sibling formatting (e.g., "[T3] SUCCESS setup scaffolding") for agent context. |
| 8 | note | `build_config` constructs a `serde_json::Value`, serializes it to a `String`, then re-parses via `flick::Config::from_str`. This JSON-to-string-to-config roundtrip happens on every agent call. | `config_gen.rs:334-373` | If Flick exposes a `from_value` or builder API, use it to avoid the serialize-parse cycle. Low priority since this is not on a hot path. |

## Notes
- Finding #1 is the most impactful: it creates a latent failure mode where the assessment agent could attempt tool use (the config says tools are available) but the dispatch path would not handle the response. Whether this manifests depends on the LLM's tendency to call tools when they are defined but not strictly needed.
- Findings #3-#6 share a common pattern: the wire types carry fields for LLM explainability (rationale, details) that are deserialized into domain types but never consumed. A consistent policy -- either always emit these for observability or strip them at the wire boundary -- would reduce the ambiguity.
- The `Task` struct being `Serialize + Deserialize` for state persistence is the reason it carries many fields that are irrelevant to prompt construction. The suggestion in #2 does not conflict with persistence; it only narrows the view passed to agents.
