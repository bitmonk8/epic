# U12-R3: cli + main / Error handling

## Summary
The codebase avoids `unwrap`/`expect` entirely and handles most errors explicitly, but relies heavily on `std::process::exit(1)` for user-facing errors, which bypasses Drop-based resource cleanup and prevents callers from catching failures programmatically. The `unreachable!()` macro on line 158 is a latent panic risk.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `std::process::exit(1)` used in 6 places bypasses destructors and async runtime cleanup. Any pending state writes, file handles, or TUI terminal restore logic will be skipped. | `main.rs:73,83,93,101,123,131,222` | Return typed errors (e.g., `anyhow::bail!`) and let `main` propagate them so Rust's normal cleanup runs before exit. |
| 2 | minor | `unreachable!()` on `Command::Init \| Command::Status` will panic at runtime if control flow is ever refactored to reach that arm (e.g., a new subcommand added without updating the early-return guards). | `main.rs:158` | Replace with an explicit error return: `_ => anyhow::bail!("unexpected command variant")` to avoid a panic in production. |
| 3 | minor | When the orchestrator times out after TUI exit (line 197-199), the task is aborted and no state is saved. In-progress work since the last checkpoint is silently lost. | `main.rs:197-199` | Attempt to save state from the orchestrator before aborting, or at minimum log a warning that unsaved progress was lost. |
| 4 | minor | Orchestrator error after TUI mode (line 190-192) is printed to stderr but the process exits with code 0 (`Ok(())`). The caller has no way to detect failure via exit code. | `main.rs:186-203` | Propagate the error or set a non-zero exit code when the orchestrator fails or panics. |
| 5 | note | Duplicated state-load-and-validate logic between `Command::Run` (lines 66-104) and `Command::Resume` (lines 125-156) is identical in structure. If error messages or validation evolve, they may drift. | `main.rs:66-104, 125-156` | Extract a shared helper, e.g., `fn load_and_validate_state(path) -> Result<(EpicState, TaskId, String)>`. |
| 6 | note | `cli.rs` is clean: no panics, no error handling needed. All validation is declarative via clap derive macros with sensible defaults and env fallbacks. | `cli.rs` | No action needed. |

## Notes
- No `unwrap()` or `expect()` calls were found in either file, which is commendable.
- The `std::process::exit` pattern is the most impactful issue: it is safe in a simple CLI context today, but will cause problems if the binary is ever used as a library, embedded in tests, or if TUI terminal state needs restoring on error paths.
- The `#[allow(clippy::too_many_lines)]` suppression on `main` (line 26) is a secondary code-quality signal that the function would benefit from decomposition, which would also make error handling easier to reason about.
