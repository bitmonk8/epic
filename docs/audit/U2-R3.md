# U2-R3: agent/flick / Error handling

## Summary

The file demonstrates consistently good error handling with proper `anyhow` propagation and no panics in production code. A few minor gaps exist around silent loop termination and tool execution fault isolation.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | Tool loop silently exits on non-`ToolCallsPending` status without checking for errors before continuing to parse. If `result.status` is `Error` mid-loop, the loop breaks and falls through to `check_error` at line 142, which does handle it -- but an `Error` status with an unexpected text block could still reach `serde_json::from_str` at line 145 if `check_error` somehow passes. The current flow is technically safe but fragile; an early `check_error` inside the loop would be more defensive. | `flick.rs:106-147` | Call `check_error(&result)?` immediately after the `break` (or at the top of the loop body before extracting tool calls) so errors are caught closer to their source. |
| 2 | minor | A single tool execution failure inside `tools::execute_tool` is recorded via `is_error` in the tool result and sent back to the model, but if `execute_tool` itself panics (e.g., due to a bug in a tool handler), the entire agent call unwinds with no structured error. There is no `catch_unwind` or similar guard. | `flick.rs:114-121` | Wrap `tools::execute_tool` in `tokio::task::spawn` or `std::panic::catch_unwind` (via `AssertUnwindSafe`) and convert panics into an `is_error` tool result so one faulty tool does not crash the orchestrator. |
| 3 | minor | The timeout covers each individual `client.run` / `client.resume` call, but there is no overall wall-clock deadline for the entire tool loop. A model that consistently responds just under the per-call timeout could keep the loop running for up to `MAX_TOOL_ROUNDS * call_timeout` (e.g., 50 * timeout). | `flick.rs:98-136` | Add an outer `Instant::now()` + deadline check so the total elapsed time for `run_with_tools` is bounded independently of per-call timeouts. |
| 4 | note | `unwrap()` / `unwrap_err()` usage is confined entirely to `#[cfg(test)]` code (lines 433, 472, 490, 515, 530), which is acceptable. No `expect`, `panic!`, `todo!`, or `unimplemented!` macros appear in production paths. | `flick.rs:409-533` | None needed. |
| 5 | note | Error messages from `map_err` closures (lines 53, 74-75, 103-104, 134-135) are descriptive and include the upstream error via `{e}`, which is good practice for debugging. | `flick.rs:51-135` | None needed. |

## Notes

- The `FlickAgent` holds no file handles, sockets, or other resources requiring explicit cleanup; `flick::FlickClient` and `flick::Context` are created per-call and dropped naturally, so resource cleanup is not a concern in this file.
- All `AgentService` trait methods propagate errors via `anyhow::Result`, allowing callers to handle failures uniformly.
- The `check_error` helper at line 366 correctly handles the case where `result.error` is `None` on an `Error` status by falling back to a descriptive "unknown flick error" message.
