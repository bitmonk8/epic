# U14-R6: git / Testability

## Summary
`src/git.rs` is a stub module (single comment, no code). All git functionality lives inline in `orchestrator.rs` as a hard-wired `tokio::process::Command::new("git")` call inside `check_scope_circuit_breaker`, making the git interaction untestable in isolation and impossible to mock without spawning a real git process.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Empty module declared but unused** -- `git.rs` contains only a comment (`// Git operations: commit, rollback, diff.`) yet is declared via `mod git;` in `main.rs`. The promised abstractions (commit, rollback, diff) do not exist, leaving git operations scattered and unencapsulated. | `src/git.rs:1`, `src/main.rs:5` | Either remove the dead module declaration or implement a `GitOps` trait in `git.rs` and move git logic there. |
| 2 | major | **Hard-wired subprocess call prevents mocking** -- `check_scope_circuit_breaker` calls `tokio::process::Command::new("git")` directly. There is no trait boundary, no injected dependency, and no way to substitute a fake git backend in tests. Integration tests for scope-checking therefore require a real git repository on disk. | `src/orchestrator.rs:345-353` | Extract a `GitDiff` trait (e.g., `async fn numstat(&self) -> Result<String>`) and inject it into `Orchestrator`, allowing unit tests to supply a mock that returns canned numstat output. |
| 3 | minor | **`evaluate_scope` is well-tested but coupled to orchestrator** -- The pure parsing function `evaluate_scope` has good unit tests (within-bounds, exceeded, binary-skip, empty), but it is a private free function inside `orchestrator.rs`. Moving it to `git.rs` (or a dedicated scope module) would make it independently testable and reusable without compiling the entire orchestrator. | `src/orchestrator.rs:35-78` | Move `evaluate_scope` and `ScopeCheck` into `git.rs` and re-export; tests can then live in a focused test module. |
| 4 | minor | **Silent fallback hides git failures** -- When the git command fails or the project root is missing, `check_scope_circuit_breaker` silently returns `WithinBounds`. This is safe at runtime but means tests cannot distinguish "scope OK" from "git unavailable", reducing diagnostic value and making it harder to write assertions about error paths. | `src/orchestrator.rs:337-352` | Return a distinct variant (e.g., `ScopeCheck::Unavailable`) or log at warn level so tests and operators can observe the fallback. |
| 5 | note | **No integration test for the git subprocess path** -- The existing tests for `check_scope_circuit_breaker` only exercise the early-return branches (no magnitude, no project root). The actual subprocess execution path has zero test coverage. | `src/orchestrator.rs:2033-2051` | After extracting a trait (finding #2), add a unit test that supplies fake numstat output and asserts `Exceeded` / `WithinBounds` through the full method, not just the pure helper. |

## Notes
- The `evaluate_scope` pure-function tests (lines 2577-2635) are solid and demonstrate the team knows how to test parsing logic; the gap is specifically at the I/O boundary where the subprocess is invoked.
- The hardcoded `multiplier = 3` inside `evaluate_scope` is not configurable, which is a minor design concern but not a testability issue per se (it is covered by existing tests).
