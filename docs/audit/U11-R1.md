# U11-R1: init / Correctness

## Summary
The init module is generally sound. The main concerns are silent swallowing of I/O errors in `read_line`, a stale temp file left behind on rename failure, multiple independent stdin locks that could conflict, and TOML injection via unsanitized user/agent strings written into comments.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `read_line()` silently discards I/O errors via `.and_then(Result::ok)`, returning empty string on read failure. This means `prompt_custom_step` and `edit_step` silently treat I/O errors as blank input, potentially producing unexpected config entries or skipped steps with no indication of the problem. | `init.rs:310-316` | Propagate the error like `read_line_checked` does, or at minimum log a warning. |
| 2 | minor | If `std::fs::rename` fails (line 59), the `.toml.tmp` temp file is left behind and never cleaned up. A subsequent `run_init` call will also not detect it since only `epic.toml` existence is checked. | `init.rs:56-59` | Wrap the rename in a block that deletes the temp file on failure, e.g., use a guard/drop or explicit cleanup in the error path. |
| 3 | minor | `present_and_confirm` acquires a `stdin.lock()` (line 91), but after it returns `prompt_models` and `prompt_limits` each independently call `io::stdin().lock()` again (lines 229, 261). While not UB, taking multiple independent locks on stdin across sequential calls is fragile. If the code were ever parallelized or if buffered data remained in the first lock's buffer, input could be lost. | `init.rs:90-91, 228-229, 260-261` | Pass a single locked stdin handle through all prompt functions, or restructure to use one lock for the entire interactive session. |
| 4 | minor | Declined-step comments written to TOML (lines 42-53) embed `step.name` and command strings without escaping. If the agent returns a name containing `"` or newlines, the generated comment will be malformed and confusing when a user tries to uncomment it. | `init.rs:42-53` | Escape or sanitize the name and command values, or use `toml::to_string` to serialize each declined step individually. |
| 5 | minor | `edit_step` splits the edited command on whitespace (`split_whitespace`, line 173), which cannot represent arguments containing spaces (e.g., `grep "hello world"`). The code has a doc comment acknowledging this (line 167-168), but the user-facing prompt gives no indication of this limitation. | `init.rs:167-173` | Add a note in the interactive prompt text, e.g., `"Command (space-separated, no quoted args) [...]:"`. |
| 6 | note | The `config_path.exists()` check (line 14) followed by later `std::fs::write` / `std::fs::rename` (lines 58-59) is a TOCTOU race: another process could create `epic.toml` between the check and the rename. In practice this is low-risk for an interactive CLI tool. | `init.rs:14-19, 56-59` | Use `OpenOptions::new().create_new(true)` for atomic create-or-fail semantics if stricter guarantees are desired. |
| 7 | note | `timeout` parse failure in `edit_step` (line 182) and `prompt_limits` (lines 272, 277, 282) silently falls back to the default via `unwrap_or`. The user gets no feedback that their input was invalid. | `init.rs:182, 271-283` | Print a warning when parsing fails so the user knows their input was discarded. |

## Notes
- The `with_extension("toml.tmp")` call on `"epic.toml"` correctly produces `"epic.toml.tmp"` since Rust's `Path::with_extension` replaces only the final extension component.
- The `From<DetectedStepWire> for VerificationStep` impl defaults timeout to 300, consistent with the hardcoded defaults used throughout this file.
- Finding #3 is safe today because the locks are taken sequentially and each is dropped before the next, but the pattern is brittle against future refactoring.
