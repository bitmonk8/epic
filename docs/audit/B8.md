# B8: Prompt and context pipeline

## Summary
The context pipeline is well-structured overall, but has cross-module gaps where information stored in the orchestrator never reaches the prompt, and one case where the orchestrator builds context for the wrong perspective relative to the agent's decision scope.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Checkpoint agent cannot see child subtasks.** `execute_branch` calls `build_context(id)` where `id` is the branch parent, then passes it to `checkpoint()`. `build_context` populates `completed_siblings` and `pending_sibling_goals` from the task's *parent* (i.e., the branch task's own siblings/uncles), not the branch task's *children*. The checkpoint agent is asked to decide whether to proceed/adjust/escalate the branch's subtask plan, but it has no visibility into which subtasks have completed, which are pending, or what outcomes they had — only the just-completed child's discoveries are passed explicitly. | `orchestrator.rs:954` calls `build_context(id)` (branch parent); `prompts.rs:259` `build_checkpoint` formats sibling context that reflects the wrong level of the tree | Either introduce a separate context builder for checkpoint that populates child subtask summaries instead of sibling summaries, or add a `children_summary` field to `TaskContext` and populate it when the context is built for a branch task's checkpoint call. |
| 2 | major | **Decomposition rationale promised but never delivered.** The `build_design_recovery_subtasks` system prompt (prompts.rs:299) tells the agent "You have the full context: the original goal, the decomposition rationale, completed subtask results, and the failure details." However, `decomposition_rationale` is stored on the `Task` struct (`task/mod.rs:117`) and set in `execute_branch` (`orchestrator.rs:909`) but is never included in `TaskContext` or `format_context`. The recovery agent has no way to see it. | `prompts.rs:299` (promise), `orchestrator.rs:909` (stored), `agent/mod.rs:26-34` (`TaskContext` struct — field absent) | Add `decomposition_rationale: Option<String>` to `TaskContext`, populate it in `build_context` from the task (or its parent for child contexts), and render it in `format_context`. Alternatively, remove the misleading claim from the system prompt. |
| 3 | minor | **Assess config includes tools but runner ignores them.** `FlickAgent::assess` (flick.rs:194-206) uses `run_structured` which has no tool loop, but `build_assess_config` (config_gen.rs:375-384) accepts a `ToolGrant` and injects tool definitions into the Flick config. The tool definitions are sent to the LLM and consume tokens, but any tool call the LLM attempts would silently produce a malformed result (no text block or partial text extraction). | `flick.rs:194-206` (`run_structured`), `config_gen.rs:375-384` (tools included), `flick.rs:196` (`phase_tools` called) | Either switch `assess` to `run_with_tools` so the assessor can explore the codebase before deciding leaf/branch (matching the prompt's intent), or remove tools from the assess config to save tokens and avoid silent failures. |
| 4 | minor | **Recovery assess/checkpoint agents get no tools but receive tool-oriented sibling context.** `build_recovery_config` and `build_checkpoint_config` (config_gen.rs:419-435) pass an empty tool set `&[]`. These agents receive sibling discoveries and other context but cannot independently verify claims. Meanwhile, the checkpoint system prompt says to "decide whether to proceed/adjust/escalate" based on discoveries — a decision that might benefit from reading the codebase. This is a design choice, but it creates an asymmetry: the prompt implies informed judgment while the agent is limited to reasoning over text summaries only. | `config_gen.rs:419-435` (no tools), `prompts.rs:259-288` and `prompts.rs:336-356` (prompts suggest decision-making without tool access) | Consider whether checkpoint/recovery-assess should have READ tool access. If the current no-tool design is intentional (for speed/cost), add a note in the system prompt clarifying the agent should reason from the provided context only. |
| 5 | note | **Sibling failure reason extracted from `attempts` not `fix_attempts`.** In `build_context` (orchestrator.rs:288-294), when a sibling is `Failed`, the failure reason is extracted from `sib.attempts` (the initial execution attempts). If a task failed during the fix loop, the most recent and relevant failure reason would be in `sib.fix_attempts`, but that collection is not consulted. Downstream agents (checkpoint, recovery, decompose) see a potentially stale initial failure reason rather than the final fix-loop failure. | `orchestrator.rs:288-294` (extracts from `attempts`), `orchestrator.rs:600` (`fix_attempts` recorded separately) | Search both `fix_attempts` and `attempts` (in that order, most recent first) when extracting the sibling failure reason. |

## Notes
- The checkpoint context gap (finding #1) is the most impactful issue. Without visibility into completed/pending children, the checkpoint agent is making plan-adjustment decisions essentially blind to execution progress. The explicit discoveries parameter partially compensates, but the agent cannot correlate discoveries against the remaining plan.
- The assess tool waste (finding #3) is low-risk in practice since the LLM is unlikely to call tools given the prompt wording, but it represents unnecessary token spend on every assess call and a latent failure mode if prompt wording changes.
