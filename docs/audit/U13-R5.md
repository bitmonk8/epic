# U13-R5: tui / Simplification

## Summary
The TUI module contains several instances of duplicated boilerplate that could be reduced through small helper functions and single-pass iteration, plus a repeated block-construction pattern across all three widget files.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `WorklogEntry` constructors (`info`, `success`, `error`, `warn`) are near-identical -- they differ only in the `Style` color. All four duplicate the timestamp-formatting logic and struct construction. | `src/tui/worklog.rs:16-46` | Extract a private `fn new(text, session_start, style) -> Self` and have each named constructor delegate to it with only the color differing. |
| 2 | minor | Many `handle_event` arms are one-liner worklog pushes that only differ in severity level and format string (e.g., `ModelSelected`, `ModelEscalated`, `SubtasksCreated`, `DiscoveriesRecorded`, `CheckpointAdjust`, `CheckpointEscalate`, `FixAttempt`, `FixModelEscalated`, `BranchFixRound`, `FixSubtasksCreated`, `RecoveryStarted`, `RecoveryPlanSelected`, `RecoverySubtasksCreated`). Multiple arms share the pattern `self.worklog.push(WorklogEntry::<level>(format!(...), self.session_start))`. (`VerificationStarted`/`VerificationComplete` arms removed.) | `src/tui/mod.rs:186-341` | Introduce a small helper `fn log(&mut self, level: Level, msg: String)` on `TuiApp` to eliminate the repeated `self.worklog.push(WorklogEntry::...(msg, self.session_start))` boilerplate. |
| 3 | minor | `MetricsWidget::render` iterates `self.tasks.values()` four separate times (completed, failed, in_progress, pending) to count phases. | `src/tui/metrics.rs:25-50` | Use a single pass with a fold or manual counters to compute all four counts at once. |
| 4 | minor | The metrics panel duplicates the completed-task count already computed in `render_header`. Both iterate `self.tasks.values().filter(phase == Completed).count()`. | `src/tui/mod.rs:372-376` and `src/tui/metrics.rs:28-30` | Compute task-phase counts once per render cycle in `TuiApp` and pass them into both the header and `MetricsWidget`. |
| 5 | note | Every widget file (`task_tree.rs`, `worklog.rs`, `metrics.rs`) constructs a `Block` with identical border style (`Borders::ALL`, `fg(Color::DarkGray)`). This is a minor pattern but worth noting. | `src/tui/task_tree.rs:113-116`, `src/tui/worklog.rs:56-59`, `src/tui/metrics.rs:17-20` | Extract a shared `fn panel_block(title: &str) -> Block` helper to reduce visual noise and ensure consistency if the style ever changes. |
| 6 | note | `render_body` duplicates the `render_task_tree` and `render_worklog` calls across both branches (metrics on/off), differing only in layout constraints. | `src/tui/mod.rs:398-428` | Compute the layout constraints conditionally, then call `render_task_tree` and `render_worklog` once after the branch, rendering the metrics widget only when `show_metrics` is true. |

## Notes
- The codebase is relatively small and well-structured for a monitoring TUI. None of the findings are critical or represent correctness issues; they are purely about reducing boilerplate and improving maintainability.
- The `#[allow(clippy::too_many_lines)]` on `handle_event` is a signal that the function is aware of its length. The suggested `log` helper (finding #2) would likely bring it under the threshold naturally, removing the need for the suppression.
