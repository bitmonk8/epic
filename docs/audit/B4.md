# B4: Error handling uniformity

## Summary
Agent call-sites in `orchestrator.rs` use two error strategies — fatal (`?`) and best-effort (log-and-continue) — but the boundary between them is inconsistent: structurally similar calls in adjacent code paths use different strategies, and the choice of which calls are best-effort appears driven by when each feature was added rather than by a coherent policy.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `verify()` is always fatal (`?`) even inside `leaf_fix_loop` and `branch_fix_loop` (orchestrator.rs:625, 757), yet the surrounding recovery and checkpoint calls in the same `execute_branch` method are best-effort. A transient agent error (rate limit, timeout) during re-verification after a fix kills the entire orchestrator run, while the same transient error during `checkpoint()` (line 956) or `assess_recovery()` (line 1057) is gracefully absorbed. Since verification is retried in loops, a transient failure should be survivable. | `orchestrator.rs:625,757` vs `orchestrator.rs:956,1057` | Make `verify()` failures inside fix loops best-effort: treat a transient agent error as a verification failure (retry the fix) rather than aborting the run. Alternatively, introduce a retry-with-backoff wrapper specifically for verification calls. |
| 2 | major | `design_fix_subtasks()` is fatal (`?`, orchestrator.rs:732) while the structurally analogous `design_recovery_subtasks()` is best-effort (orchestrator.rs:1092). Both are called mid-loop to generate corrective subtasks after a failure, and both have the same failure semantics (the parent should fail if no subtasks are produced). A transient error in `design_fix_subtasks` aborts the run; the same error in `design_recovery_subtasks` consumes a recovery round and continues. | `orchestrator.rs:732` vs `orchestrator.rs:1092` | Apply the same best-effort pattern to `design_fix_subtasks`: catch the error, log it, and treat it as a failed fix round (continue to next round or escalate). |
| 3 | minor | `flick.rs` treats all agent errors uniformly (propagate via `?`), giving `orchestrator.rs` no signal to distinguish transient errors (timeout, rate limit) from permanent ones (bad config, parse failure). The orchestrator's best-effort call-sites (checkpoint, recovery) discard the error category, logging the same warning for a timeout as for a schema violation. If `flick.rs` distinguished error kinds, the orchestrator could retry transient errors and fail fast on permanent ones, rather than choosing one strategy per call-site. | `flick.rs:74,103,134` (all use `anyhow::anyhow!`) and `orchestrator.rs:958,1064,1098` (all use `eprintln!`) | Have `FlickAgent` return a typed error enum (e.g., `AgentError::Transient` vs `AgentError::Permanent`) so that orchestrator call-sites can apply a single policy: retry transient errors, propagate permanent ones. This would replace the current per-call-site fatal-vs-best-effort split with a uniform strategy. |
| 4 | minor | `init.rs` propagates `explore_for_init()` errors directly to the CLI user (`run_init`, line 23) with no context about what the agent was doing, while `orchestrator.rs` best-effort sites add `"warning: checkpoint classification failed"` or `"warning: recovery assessment failed"` context. A user seeing a raw flick timeout from `init` gets `"flick call timed out after 120s"` with no indication it was the project-scanning step. Since `flick.rs` produces generic error messages, the calling module is responsible for adding context — but only `orchestrator.rs` does so (and only at best-effort sites). | `init.rs:23` vs `orchestrator.rs:959,1065,1099` | Add `.context("project exploration agent failed")` to the `explore_for_init` call in `init.rs`, and similarly to the fatal `?` sites in `orchestrator.rs` (e.g., `assess`, `execute_leaf`, `verify`) where the generic flick error message would be unhelpful in logs. |

## Notes
- The best-effort pattern in `orchestrator.rs` is intentionally applied to `checkpoint` and `assess_recovery` (comments at lines 955 and 1054 explain the rationale). The inconsistency is that the same rationale ("avoid aborting the entire run due to transient errors") applies equally to `verify` and `design_fix_subtasks` when called inside fix loops, but those calls remain fatal.
- Finding 3 is a design-level observation that spans `flick.rs` and `orchestrator.rs`: neither module alone reveals the problem, but together they show that error categorization is lost at the boundary between them, forcing the orchestrator into a binary fatal-or-silent choice.
