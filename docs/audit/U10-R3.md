# U10-R3: config / Error handling

## Summary
The config module is a lightweight set of serde data structures with no runtime logic of its own. It contains no panics, unwraps, or error-prone code in production paths; all `unwrap()` calls are confined to test code.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | note | `unwrap()` used on `toml::to_string_pretty` and `toml::from_str` in three test functions | `src/config/project.rs:118-119, 137, 146` | Acceptable in tests; no action needed |
| 2 | note | No config-loading function exists in this module; parsing is delegated to callers | `src/config/mod.rs` (entire file) | If a shared `load()` helper is added later, ensure it returns `Result` and handles missing/malformed files gracefully |
| 3 | note | `VerificationStep` requires `name` and `command` fields with no defaults, so a partial TOML `[[verification]]` entry will produce a serde deserialization error | `src/config/project.rs:58-64` | Consider providing a user-friendly error message wrapper when deserializing verification steps |

## Notes
- The module is purely declarative: structs, `Default` impls, and serde attributes. There is no file I/O, no resource acquisition, and no fallible production code, so there is nothing to clean up or degrade gracefully within this module itself.
- The caller in `src/init.rs` correctly uses `?` / `map_err` for serialization, demonstrating proper error propagation at the call site.
- All `#[serde(default)]` annotations ensure that missing optional sections parse without error, which is good defensive design.
