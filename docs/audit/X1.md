# X1: Cargo.toml & dependencies

## Summary

The manifest is well-structured with strict lints, but carries one unused tokio feature flag, a git dependency without a pinned tag/rev, and is missing several metadata fields that matter for reproducibility and discoverability.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `flick` is pulled from a bare git URL with no `tag` or `rev` pin in Cargo.toml. The lock file pins a commit hash, but any fresh `cargo update` can silently advance to HEAD, breaking reproducibility. | `Cargo.toml:14` | Pin with `rev = "8bf1d79..."` or a release tag to make builds deterministic independent of the lock file. |
| 2 | minor | Tokio feature `"signal"` is enabled but never imported (`tokio::signal`, `ctrl_c`, etc. are absent from all `.rs` files). | `Cargo.toml:8` | Remove `"signal"` from the tokio features list to reduce compile scope. |
| 3 | minor | Package metadata is sparse -- `description`, `license`, `authors`, and `repository` are all missing. | `Cargo.toml:1-5` | Add at minimum `description` and `license` (required for crates.io publish) and `repository` for traceability. Even for internal crates these fields aid tooling and audit trails. |
| 4 | note | `crossterm = "0.28"` is listed as a direct dependency and is used in `src/tui/mod.rs`. `ratatui 0.29` also depends on crossterm. The versions are compatible today, but keeping crossterm as a direct dep creates a future upgrade coordination burden. | `Cargo.toml:18` | Acceptable as-is since the TUI code uses crossterm APIs directly; just be aware both must be upgraded in lockstep with ratatui. |
| 5 | note | All other dependencies (`anyhow`, `thiserror`, `serde`, `serde_json`, `toml`, `bitflags`, `clap`, `ratatui`, `globset`, `walkdir`, `regex`) are confirmed imported in source. `tempfile` is correctly scoped under `[dev-dependencies]`. No unused crates found. | `Cargo.toml:7-21, 37-38` | No action needed. |

## Notes

- Edition 2024 with `rust-version = "1.85"` is current and correct (Rust 1.85 is the first release supporting edition 2024).
- Cargo.lock is committed and uses `version = 4`, matching the edition 2024 resolver. This is good practice for a binary crate.
- The lint configuration (`unsafe_code = "deny"`, clippy pedantic/nursery at warn) is thorough and well-tuned with sensible allowances.
- The `flick` git dependency transitively pulls in `reqwest`, `chacha20poly1305`, and other heavy crates. If `flick` stabilises, switching to a crates.io release would improve auditability via `cargo audit`.
