# U1-R2: orchestrator / Security

## Summary

The orchestrator itself has a limited attack surface since it delegates execution to the agent layer, but it spawns an unsandboxed `git` subprocess, silently swallows checkpoint-write failures that could mask state corruption, and lacks bounds on cumulative subtask creation which could lead to resource exhaustion.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Unbounded subtask accumulation leading to resource exhaustion.** `create_subtasks` appends children without any global cap on total task count. The branch fix loop (`MAX_ROOT_FIX_ROUNDS=4`), recovery loop (`MAX_RECOVERY_ROUNDS=2`), and checkpoint-escalation recovery all create new subtasks that are appended to the parent. Combined with depth-8 recursion, the theoretical worst case is exponential task-tree growth. An adversarial or degenerate agent response could fill memory before any single loop cap is hit. | `orchestrator.rs:184-238` | Add a global task-count ceiling to `EpicState` (e.g., `MAX_TOTAL_TASKS = 500`) and check it in `create_subtasks` before inserting. |
| 2 | major | **`git diff` subprocess has no timeout.** `check_scope_circuit_breaker` calls `tokio::process::Command::new("git")` with `.output().await` but sets no timeout. In a large repository or on a hung NFS mount, this blocks the orchestrator indefinitely. Unlike the bash tool (which enforces `MAX_BASH_TIMEOUT_SECS`), this code path has no deadline. | `orchestrator.rs:345-353` | Wrap the `.output().await` in `tokio::time::timeout(Duration::from_secs(30), ...)` and treat timeout as `WithinBounds` (fail-open, matching the existing error-handling pattern). |
| 3 | minor | **Checkpoint write failure is silently swallowed.** `checkpoint_save` logs to stderr but never returns an error or sets a flag. If the disk is full or the path is invalid, the orchestrator continues executing tasks whose state will not survive a crash, violating the crash-safety invariant documented in the code comments. An operator has no programmatic way to detect this. | `orchestrator.rs:137-143` | At minimum, set a `checkpoint_failed: bool` flag on the orchestrator and check it before critical state transitions. Consider failing the run after N consecutive checkpoint failures. |
| 4 | minor | **No path validation on `state_path` or `project_root`.** `with_state_path` and `with_project_root` accept arbitrary `PathBuf` values without canonicalization or existence checks. A state path pointing to a symlink outside the intended directory could be used to overwrite arbitrary files via the atomic-rename in `EpicState::save`. The agent tools layer has `safe_path` validation, but the orchestrator's own paths bypass it entirely. | `orchestrator.rs:99-107` | Canonicalize and validate both paths at construction time. For `state_path`, verify the parent directory exists and is within an expected prefix. |
| 5 | minor | **`git diff` invoked without `--no-ext-diff`.** The scope circuit breaker runs `git diff --numstat HEAD` which can invoke an external diff driver configured in `.gitattributes` or `.gitconfig`. A malicious repository could configure an external diff program that executes arbitrary code. | `orchestrator.rs:346` | Pass `--no-ext-diff` to prevent external diff drivers: `args(["diff", "--no-ext-diff", "--numstat", "HEAD"])`. |
| 6 | note | **TOCTOU in scope circuit breaker.** `check_scope_circuit_breaker` reads `git diff` output and then returns, but the working tree can change between the check and subsequent agent execution. This is inherent to the design (the check is a best-effort guardrail, not a hard gate) but worth documenting as a known limitation. | `orchestrator.rs:326-357` | Document the TOCTOU gap explicitly. If stronger guarantees are needed, run the scope check inside a git stash/lock or immediately before and after each leaf execution. |
| 7 | note | **Recursive `execute_task` uses heap-allocated `Pin<Box<dyn Future>>`.** Each level of task-tree recursion allocates a boxed future. With `MAX_DEPTH=8` and potential fan-out from recovery/fix subtasks, this is unlikely to cause issues in practice, but deep trees with wide fan-out could accumulate significant heap pressure. | `orchestrator.rs:361-363` | Acceptable at current depth limits. If `MAX_DEPTH` is ever raised, consider converting to an explicit stack-based iteration. |

## Notes

- The orchestrator does not directly handle credentials or secrets; those concerns live in the agent/flick layer. No credential-exposure issues were found in this file.
- The `AgentService` trait is the primary trust boundary. The orchestrator trusts agent responses (subtask specs, verification outcomes) without structural validation beyond empty-list checks. A compromised or buggy agent implementation could induce the resource-exhaustion scenario in finding #1.
- The `EpicState::save` method uses atomic write-then-rename, which is good practice for crash safety on POSIX systems. However, the orchestrator's error handling around it (finding #3) undermines this.
- The bash tool in `agent/tools.rs` has proper timeout enforcement and path sandboxing; the gap in finding #2 is specific to the orchestrator's own subprocess call.
- **Update (finding #1):** A global `max_total_tasks` cap (default 100, configurable via `limits.max_total_tasks` in `epic.toml`) has been added to `EpicState`, preventing unbounded subtask accumulation. Recovery subtasks now inherit the parent's `recovery_rounds` counter instead of starting at 0. `MAX_DEPTH`, `MAX_RECOVERY_ROUNDS`, and other previously hardcoded constants are now configurable via `epic.toml`.
