# U11-R3: init / Error handling

## Summary
Error handling in `init.rs` is generally solid, with proper use of `anyhow::Result` propagation and `bail!` for expected failures. A few `unwrap_or` calls silently swallow parse errors, and `read_line` silently discards I/O errors, which could mask problems in non-interactive contexts.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `read_line` (line 310-316) silently swallows I/O errors via `.and_then(Result::ok)`, converting them to empty strings. In a piped/scripted context this hides real failures. | `src/init.rs:310-316` | Return `anyhow::Result<String>` and propagate errors, matching `read_line_checked` and `read_line_or_eof`. |
| 2 | minor | `timeout_str.parse().unwrap_or(default_timeout)` on lines 182, 212, 271-273, 276-278, 281-283 silently falls back to the default when the user enters a non-numeric value with no feedback. | `src/init.rs:182, 212, 271-283` | Print a warning (e.g., "Invalid number, using default") so the user knows their input was ignored. |
| 3 | minor | Temp file `epic.toml.tmp` (line 58) is not cleaned up if `std::fs::rename` on line 59 fails. | `src/init.rs:57-59` | Wrap the rename in a match or use a scopeguard/drop-guard to remove the temp file on error. |
| 4 | note | `let _ = write!(...)` on line 48 discards the `fmt::Write` result. `fmt::Write` on `String` is infallible so this is safe, but the explicit discard could confuse readers. | `src/init.rs:48-53` | Remove `let _ =` or add a comment explaining infallibility. |
| 5 | note | `prompt_models` and `prompt_limits` each acquire a new `stdin().lock()` (lines 228-229, 260-261), which will conflict if stdin is already locked by a caller. Currently safe because `present_and_confirm` drops its lock before these are called, but the pattern is fragile under refactoring. | `src/init.rs:228-229, 260-261` | Consider threading a single `lines` iterator through all prompt functions to avoid multiple lock acquisitions. |

## Notes
- No `unwrap()` or `expect()` calls exist anywhere in the file; all fallible operations either use `?` or `unwrap_or` with a default.
- The atomic-write pattern (write tmp then rename) is a good practice; the only gap is cleanup on rename failure (finding #3).
- The `bail!` on pre-existing `epic.toml` (line 15) is appropriate and gives a clear message.
