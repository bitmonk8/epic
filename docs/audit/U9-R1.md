# U9-R1: events / Correctness

## Summary

The events module is a straightforward enum definition plus an unbounded channel constructor. It has no logic errors or missed error paths. Two minor observations relate to the silent send-error suppression in the caller and the unbounded channel choice.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | note | `emit()` silently discards the `SendError` via `let _ = self.events.send(event)` (orchestrator.rs:146). If the TUI receiver is dropped early (e.g. user quits), all subsequent events are silently lost. This is likely intentional for a fire-and-forget UI channel, but means the orchestrator has no feedback that its observer is gone. | `src/orchestrator.rs:146` | Acceptable for a TUI channel; document the invariant or log at debug level on first send failure if observability is desired. |
| 2 | note | `event_channel()` returns an unbounded channel. A pathological burst of events (e.g. thousands of subtasks created faster than the TUI can drain at its 50 ms poll interval) could cause unbounded memory growth. In practice the TUI drains all pending events each tick, and task counts are bounded by the decomposition depth/width limits, so this is unlikely to be a real issue. | `src/events.rs:99-101` | No action needed unless the event rate grows significantly; a bounded channel with `try_send` would add backpressure at the cost of complexity. |

## Notes

- The `Event` enum is exhaustively matched in `handle_event` (tui/mod.rs:132-342) with no wildcard arm, so the compiler enforces that every new variant is handled. This is a good correctness property.
- All fields in the enum variants use owned types (`String`, `Vec<TaskId>`, cloneable newtypes), avoiding any lifetime or borrowing issues.
- The `#[derive(Clone)]` on `Event` is required because `TaskOutcome` is cloned at the emit site (orchestrator.rs:164) and because the TUI consumer takes events by value. This is correct.
- No off-by-one or edge-case issues exist in this file; the module is purely declarative (type definitions and a single constructor).
