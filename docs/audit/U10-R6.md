# U10-R6: config / Testability

## Summary

The config structs in `project.rs` are well-suited for unit testing thanks to `Default` derives and serde round-trip coverage, but the module lacks validation logic tests, `PartialEq` derives needed for assertion ergonomics, and the `mod.rs` stub advertises config-loading behaviour that does not yet exist (and therefore has zero coverage). The interactive `init.rs` consumers of these structs are tightly coupled to real stdin/stdout, creating significant test friction.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | No `PartialEq` derive on any config struct. Tests must compare field-by-field; snapshot/equality assertions against whole structs are impossible, making future regression tests fragile and verbose. | `project.rs:6,24,32,42,52,58` | Add `#[derive(PartialEq)]` to all config structs. |
| 2 | major | No validation logic and therefore no validation tests. `max_depth = 0`, `retry_budget = 0`, an empty `command` vec in `VerificationStep`, or a blank model name are all silently accepted. Any consumer must defensively check these invariants itself, and there is no centralised test proving invalid configs are rejected. | `project.rs` (all structs) | Add a `validate(&self) -> Result<()>` method on `EpicConfig` (or use the `validator` crate) with corresponding test cases for boundary and invalid values. |
| 3 | major | `mod.rs` comment declares it handles loading from `~/.config/epic/config.toml` and project `epic.toml`, but the file contains only `pub mod project;`. The loading/merging logic is absent, so there are no tests for file-system config loading, path resolution, merge precedence, or missing-file fallback. | `mod.rs:1` | Implement a `load(project_root: &Path) -> Result<EpicConfig>` function with filesystem abstraction (or accept a reader) and add tests for missing file, partial file, and merge behaviour. |
| 4 | major | `init.rs` functions (`present_and_confirm`, `prompt_models`, `prompt_limits`) read directly from `io::stdin()`. There is no way to inject a mock reader in tests, so the entire interactive init flow is untestable without spawning a subprocess or redirecting file descriptors. | `init.rs:90-91,227-229,260-261` | Accept a generic `BufRead` parameter (or a `&mut dyn BufRead`) instead of calling `io::stdin()` internally. The existing `edit_step` and `prompt_custom_step` already accept `impl Iterator<Item = io::Result<String>>` -- extend that pattern to all prompt functions. |
| 5 | minor | `run_init` performs filesystem I/O (existence check, atomic write+rename) with no abstraction boundary. Testing it requires a real temp directory and cannot be intercepted or mocked. | `init.rs:13-59` | Extract file operations behind a trait or accept a writer/path-provider, or at minimum add integration tests using `tempfile`. |
| 6 | minor | Existing tests cover round-trip, verification-step parsing, and minimal config, but do not test individual section overrides (e.g., setting only `[models]` while relying on defaults for everything else), or partially-filled structs. Edge cases like unknown keys (forward-compat) are also untested. | `project.rs:111-150` | Add test cases for partial configs (`[models]\nfast = "custom"` with all else defaulted), unknown-key tolerance (`#[serde(deny_unknown_fields)]` or explicitly testing that unknown keys are ignored), and zero/boundary numeric values. |
| 7 | note | Default functions (`default_root`, `default_fast_model`, etc.) are private, which is fine, but they duplicate the logic in the `Default` impls. If one changes without the other, serde defaults and programmatic defaults diverge silently. | `project.rs:68-77,79-109` | Consider using a single source of truth: either call the freestanding functions from the `Default` impl (already done) or use a const/static and reference it in both places. Currently correct but fragile under maintenance. |

## Notes

- The scope of this review is limited to `src/config/mod.rs` and `src/config/project.rs`, but testability findings in `src/init.rs` are included because it is the primary consumer of the config structs and its coupling directly impacts how config logic can be tested end-to-end.
- The existing three tests in `project.rs` are well-written and cover the happy path adequately; the gaps are in negative/edge-case coverage and in the absence of the advertised config-loading layer.
