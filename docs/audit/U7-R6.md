# U7-R6: task / Testability

## Summary
The task module is a pure data-model layer with no logic to test directly; testability concerns center on missing trait derives that create test friction, the absence of a validated state machine for phase transitions, and the lack of any unit tests within the module itself.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `TaskPhase` transitions are unchecked -- any code can set `task.phase` to any variant. The `transition()` method in the orchestrator performs no validation of legal state changes (e.g. `Pending -> Completed` is silently allowed). This makes it impossible to write tests that assert the state machine is upheld without inspecting every call site. | `src/task/mod.rs:27-34`, `src/orchestrator.rs:149-157` | Add a `TaskPhase::try_advance(from, to) -> Result` or `Task::transition(to) -> Result` method that encodes the legal edges, and unit-test the state machine in `task/mod.rs`. |
| 2 | major | `LeafResult` and `RecoveryPlan` lack `PartialEq` (and `Serialize`/`Deserialize`). Every other domain type in the module derives both. Without `PartialEq`, test assertions require field-by-field destructuring instead of `assert_eq!`, adding friction and fragility. | `src/task/mod.rs:83-87` (`LeafResult`), `src/task/mod.rs:92-97` (`RecoveryPlan`) | Derive `PartialEq, Eq, Serialize, Deserialize` on both types (and add `PartialEq` to `TaskOutcome`-containing fields that already have it). |
| 3 | major | Zero unit tests exist in the `task` module (no `#[cfg(test)]` block in any of the five files). All coverage of these types comes indirectly through orchestrator integration tests, which means regressions in data-model invariants (e.g. `Task::new` defaults, `Model::escalate` chain) are caught late. | `src/task/mod.rs`, `src/task/assess.rs`, `src/task/leaf.rs`, `src/task/branch.rs`, `src/task/verify.rs` | Add a `#[cfg(test)] mod tests` block in `mod.rs` covering at minimum: `Model::escalate` round-trip, `Task::new` default field values, `TaskId` display formatting, and serde round-trips for all public types. |
| 4 | minor | `Task` has 20 fields, all public, and is constructed only via `Task::new` which sets many to default/empty values. Tests in the orchestrator must set numerous fields manually (e.g. `root.phase = TaskPhase::Executing`) to arrange state, creating verbose and brittle test setup. | `src/task/mod.rs:101-122` | Introduce a `TaskBuilder` or `Task::test_builder()` (cfg-gated) with a fluent API, or at least a `Task::with_phase(self, phase) -> Self` chain to reduce test boilerplate. |
| 5 | minor | `Attempt` lacks `PartialEq`, preventing direct equality assertions in tests. The orchestrator tests that push `Attempt` values into `task.attempts` or `task.fix_attempts` cannot verify the contents with `assert_eq!`. | `src/task/mod.rs:54-59` | Derive `PartialEq, Eq` on `Attempt`. |
| 6 | minor | `Magnitude` lacks `PartialEq`, inconsistent with `MagnitudeEstimate` which does derive it. Same test-friction issue as above. | `src/task/mod.rs:68-74` | Derive `PartialEq, Eq` on `Magnitude`. |
| 7 | note | The `MockAgentService` in `orchestrator.rs` tests is ~160 lines of hand-rolled queue-based mock. Because the task types are the contract surface of `AgentService`, any new field or variant added to task types requires updating the mock manually. This is workable at current scale but will become friction as the trait grows. | `src/orchestrator.rs:1183-1320` | Consider generating mocks via `mockall` or extracting `MockAgentService` into a shared `testutil` module so multiple test files can reuse it. |

## Notes
- The `task` module is intentionally a thin data-model layer; business logic lives in `orchestrator.rs`. The audit therefore focuses on whether the data types are structured to support easy, reliable testing of the orchestrator and any future task-level logic.
- The `leaf.rs` file contains only a two-line comment and no code, meaning the `LeafResult` type that represents leaf execution lives in `mod.rs`. This split is slightly misleading for developers looking for leaf-related types.
- The `AgentService` trait uses `impl Future` returns (not `dyn`), which is good for testability since concrete mock structs work without boxing overhead.
