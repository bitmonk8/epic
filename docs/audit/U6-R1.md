# U6-R1: agent/models / Correctness

## Summary

The `models.rs` file is small and correct. The `mod.rs` file defines the `AgentService` trait and supporting types with no logic errors. One substantive correctness issue exists in `flick.rs` (the primary `AgentService` implementation reachable from `mod.rs`): the `assess` method uses `run_structured` (no tool loop) despite being granted tools, which would fail if the LLM tries to use them. Several minor edge-case and robustness issues were found across the agent module.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `assess()` calls `run_structured` (single-shot, no tool loop) but passes `ToolGrant::READ` tools into the config. If the LLM issues a tool call, the result will have `ToolCallsPending` status, `run_structured` will not handle it, and `extract_text` will likely fail or return incomplete output. All other tool-granted methods correctly use `run_with_tools`. | `flick.rs:194-206` | Either use `run_with_tools` like the other tool-granted methods, or remove tools from the assess config (pass `&[]` instead of `grant`). |
| 2 | minor | `run_with_tools` loop runs `1..=MAX_TOOL_ROUNDS` (50 iterations) but the initial call is outside the loop, so the actual maximum total API calls is 51. The post-loop check correctly catches the overshoot case, but the effective limit is one higher than `MAX_TOOL_ROUNDS` suggests. This is not a bug per se, but violates the principle of least surprise. | `flick.rs:98-139` | Rename constant to `MAX_RESUME_ROUNDS` or adjust the comment/loop to make the semantics clear. The current comment on line 97 documents this, so the impact is low. |
| 3 | minor | `CheckpointWire` with `decision: "adjust"` and `guidance: None` produces `CheckpointDecision::Adjust { guidance: "" }` (empty string). Downstream code consuming the guidance may not distinguish between "no guidance given" and "empty guidance string", potentially injecting an empty adjustment instruction into prompts. | `config_gen.rs:155-157` | Consider using `Option<String>` in `CheckpointDecision::Adjust` instead of defaulting to empty string, or bail if `adjust` is chosen without guidance. |
| 4 | minor | `RecoveryWire::into_strategy` returns `Some(String::new())` when `recoverable: true` but `strategy: None`. This empty strategy string is then passed verbatim to `design_recovery_subtasks` as the `strategy: &str` parameter, producing a prompt with an empty strategy field. The LLM may produce poor results with no strategy guidance. | `config_gen.rs:171-173` | Bail or return a default strategy description when `recoverable` is true but `strategy` is absent. |
| 5 | note | `TaskOutcomeWire` allows `outcome: "success"` with a `reason` field present, which is silently ignored. A success with a reason attached could indicate an LLM formatting error that goes undetected. | `config_gen.rs:106-111` | Consider logging a warning or bailing if `reason` is `Some` when outcome is `"success"`. |
| 6 | note | `extract_text` returns the *last* text block from the response. If the LLM emits structured JSON in a non-final text block (e.g., followed by a trailing conversational text block), the wrong block will be parsed, causing a deserialization error. This is an assumption about LLM output ordering that may not always hold. | `flick.rs:377-388` | Consider looking for a text block that successfully parses as JSON, or using the output schema enforcement to guarantee a single text block. |

## Notes

- `models.rs` itself is 39 lines of pure `const fn` mapping with exhaustive matches -- no correctness issues.
- `mod.rs` is purely a trait definition and struct declarations with no logic to audit beyond type soundness, which Rust enforces at compile time.
- The bulk of the findings come from `flick.rs` and `config_gen.rs`, which are the implementation files directly supporting `mod.rs` and `models.rs`.
- Finding #1 is the most impactful: if the assess model ever decides to call a tool (which is plausible given tools are offered in the config), the call will fail at runtime.
