# U2-R1: agent/flick / Correctness

## Summary

The `FlickAgent` implementation is generally sound, with correct tool-loop termination and error propagation. The main issues are: `run_structured` silently ignores a `ToolCallsPending` status (masking model misbehaviour), and there is no aggregate timeout bounding the total wall-clock cost of a tool-loop session.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `run_structured` does not guard against `ToolCallsPending` status. If the model unexpectedly requests tool calls in a no-tool config, `check_error` passes (it only matches `Error`), and `extract_text` may return partial/non-JSON text or fail with a misleading "no text block" error instead of diagnosing the real cause. | `flick.rs:76-81` | After `check_error`, add an explicit check: `if matches!(result.status, ResultStatus::ToolCallsPending) { bail!("model requested tool calls in structured-only (no-tool) context"); }` |
| 2 | minor | No aggregate session timeout in `run_with_tools`. Each `client.run`/`client.resume` call gets `call_timeout`, but with `MAX_TOOL_ROUNDS = 50` rounds plus unbounded tool execution time, total wall-clock time can far exceed any single timeout. A runaway tool loop (e.g., model keeps requesting tools that each succeed quickly) could run for a very long time. | `flick.rs:98-136` | Wrap the entire tool loop in a single `tokio::time::timeout` with a separate `session_timeout`, or track elapsed time and bail when a cumulative limit is exceeded. |
| 3 | minor | Tool calls within a single round are executed sequentially. When the model returns multiple independent tool calls (e.g., reading several files), they are awaited one at a time, missing an opportunity for concurrency. More importantly, if a later tool call depends on an earlier one's side effects within the same round, the sequential order is coincidentally correct but undocumented -- a future refactor to parallel execution could break that assumption. | `flick.rs:113-127` | Document the sequential execution guarantee, or use `futures::future::join_all` for parallel execution and note that intra-round tool ordering is not guaranteed. |
| 4 | note | `extract_text` returns only the last `Text` block, discarding earlier ones. This relies on the convention that the model places structured JSON in its final text block. If the model emits JSON followed by a trailing remark (e.g., "Here is the JSON: ... Hope that helps!"), the wrong block is parsed and deserialization fails with a confusing error. | `flick.rs:377-388` | Consider trying to parse each text block as JSON (last-to-first) and returning the first successful parse, or stripping markdown fences/surrounding prose before parsing. |
| 5 | note | Timeout cancellation during `client.resume` (line 129-135) drops the model response but tool side effects (file writes, bash commands) from lines 113-127 have already occurred. The error message says "timed out" but the workspace may be in a partially-modified state. This is inherent to the architecture but worth documenting. | `flick.rs:129-135` | Document that timeout during resume leaves tool side effects intact. Callers performing retries should be aware of partial mutations. |

## Notes

- `ResultStatus` has exactly three variants (`Complete`, `ToolCallsPending`, `Error`), confirmed from the flick crate source. The match arms in `check_error` and the tool loop cover all three correctly, with no risk of an unhandled fourth variant today. However, `check_error` uses a positive match on `Error` rather than a negative match, so a future fourth variant (e.g., `Cancelled`) would silently pass through -- something to watch if the flick crate evolves.
- `ToolGrant` is a `bitflags` type (inherently `Copy`), so passing it by value in the loop is correct.
- The `const fn new` on `FlickAgent` taking `PathBuf`/`String` compiles on Rust 1.85+ (edition 2024) due to const-eval improvements for non-Copy types in simple constructors.
