# U6-R7: agent/models + agent/mod / Design intent

## Summary

The agent layer is well-structured with clean trait boundaries and correct model-per-phase wiring for most activities. However, several model assignments in `flick.rs` diverge from the AGENT_DESIGN spec, the `models.rs` module is too thin to enforce cost control, and ARCHITECTURE.md contains stale references contradicting the current Flick-as-library integration.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Assessment hardcoded to Sonnet, design spec says Haiku (escalate to Sonnet if uncertain). The spec's cost-saving intent is lost -- every assessment pays Sonnet prices. | `flick.rs:199` | Use `Model::Haiku` for `assess()`, implement escalation logic (re-call with Sonnet when Haiku returns uncertain). |
| 2 | major | `design_and_decompose` hardcoded to Sonnet, but AGENT_DESIGN says "Assessment-selected" model. The assessment result's model choice is ignored for branch decomposition. | `flick.rs:258` | Thread the assessment-selected model into `design_and_decompose`, or accept it as a parameter like `execute_leaf` does. |
| 3 | major | `verify()` hardcoded to Sonnet for all tasks. AGENT_DESIGN specifies leaf verification as "max(Haiku, implementing model) capped at Sonnet" and branch verification as Sonnet. The trait signature `fn verify(&self, ctx: &TaskContext)` carries no model parameter, making it impossible to differentiate leaf vs branch verification model selection. | `flick.rs:296`, `mod.rs:63` | Add a `model: Model` parameter to `verify()` (or a verification-kind enum), and compute the correct model in the orchestrator. |
| 4 | major | `assess_recovery` uses Sonnet, but AGENT_DESIGN specifies Opus ("Requires strongest reasoning"). Recovery assessment is one of the highest-stakes decisions and using a weaker model undermines design intent. | `flick.rs:331` | Change to `Model::Opus`. |
| 5 | minor | `models.rs` declares an escalation path in its comment (line 1: "Haiku -> Sonnet -> Opus") but contains no escalation logic -- no `next_model()`, `escalate()`, or ordering function. Escalation is handled ad-hoc in `orchestrator.rs` with hardcoded model constants. | `models.rs:1` | Add a `fn escalate(model: Model) -> Option<Model>` and `impl PartialOrd for Model` to centralize the escalation ladder, then use it from the orchestrator. |
| 6 | minor | No per-call or per-task cost tracking or budget enforcement in the agent layer. `default_max_tokens` sets output token limits but there is no mechanism to accumulate spend, enforce a session-wide budget, or alert on runaway costs. The 50-round tool loop (`MAX_TOOL_ROUNDS`) is the only cost guardrail. | `models.rs`, `flick.rs:21` | Add token/cost accumulation (the Flick `usage` field is already available on `FlickResult`) and a configurable budget cap that can abort execution. |
| 7 | minor | `AgentService::verify` grants `AgentMethod::Analyze` (READ-only tools). AGENT_DESIGN says branch verification gets TASK tools ("May spawn sub-agents for large diffs"). This limits branch verification capability. | `flick.rs:293`, `tools.rs:47` | Differentiate leaf vs branch verification tool grants; grant TASK for branch verification. |
| 8 | minor | `AgentService::design_and_decompose` grants `AgentMethod::Decompose` which maps to READ-only. AGENT_DESIGN specifies EXPLORE (READ + EXECUTE + WEB) for branch design+decompose. The agent cannot run shell commands or fetch web resources during decomposition research. | `flick.rs:255`, `tools.rs:47` | Add an `AgentMethod::Explore` variant mapped to `READ | BASH | WEB` and use it for decompose. |
| 9 | note | ARCHITECTURE.md (line 20, line 39, line 113) still says "Flick CLI", "subprocess invocation", "No crate dependency", and "spawns a new Flick process". The codebase uses Flick as a library crate. This was partially noted in commit d18df75 but the doc was not fully updated. | `ARCHITECTURE.md:20,39,113` | Update ARCHITECTURE.md to reflect the library integration pattern throughout. |
| 10 | note | `mod.rs` trait uses `impl Future` return types (RPITIT) and notes "async fn in trait works natively in edition 2024". This is correct for Rust edition 2024/nightly, but worth verifying the project's edition setting to avoid surprises on stable toolchains. | `mod.rs:39-45` | Confirm `Cargo.toml` specifies `edition = "2024"` or that nightly is pinned in CI. |

## Notes

- The `AgentService` trait in `mod.rs` is cleanly designed with appropriate separation of concerns. The issues above are predominantly about the concrete `FlickAgent` implementation in `flick.rs` not honoring the model and tool assignments laid out in AGENT_DESIGN.md.
- Findings 1-4 represent systematic drift where the implementation takes shortcuts (hardcoding Sonnet) rather than implementing the design's nuanced model selection. This inflates cost (assessment, checkpoint) or weakens reasoning (recovery assessment) relative to design intent.
- The `models.rs` module is minimal by design, but its lack of escalation primitives has pushed that logic into the orchestrator in a scattered fashion (see `orchestrator.rs:530`, `orchestrator.rs:711`, `orchestrator.rs:783`).
