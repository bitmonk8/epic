# U7-R3: task / Error handling

## Summary

The task module is a pure data-model layer with no I/O, no panics, and no `Result`-returning functions. Error-handling concerns are minimal within the module itself, but several `Option` fields on `Task` create implicit precondition contracts that downstream code fulfils with `unwrap()`, and two domain result types lack `Serialize`/`Deserialize`, preventing checkpoint persistence.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `Task::path` is `Option<TaskPath>` and `Task::current_model` is `Option<Model>`, but callers in the orchestrator unwrap these without guards (e.g., `task.path.clone().unwrap()` at `orchestrator.rs:400`). The task module provides no accessor that returns `Result` or enforces the state-machine invariant that these fields are `Some` after assessment. | `task/mod.rs:106-109` | Add typed accessor methods (e.g., `fn path(&self) -> Result<&TaskPath, TaskError>`) that return a descriptive error instead of relying on callers to unwrap raw `Option`s. Alternatively, encode the state machine in types so that post-assessment tasks carry non-optional `path` and `model` fields. |
| 2 | minor | `LeafResult` and `RecoveryPlan` do not derive `Serialize`/`Deserialize`, unlike every other result struct in the module (`AssessmentResult`, `DecompositionResult`, `VerificationResult`). This means they cannot be persisted to the checkpoint file, reducing crash-recovery fidelity for in-flight leaf executions and recovery plans. | `task/mod.rs:83-97` | Add `#[derive(Serialize, Deserialize)]` to `LeafResult` and `RecoveryPlan` (and ensure `TaskOutcome`, which they embed, also derives both -- it already does). |
| 3 | minor | `TaskOutcome::Failed { reason }` and `VerificationOutcome::Fail { reason }` duplicate the concept of a failure reason string. If both are constructed from the same event, mismatches are possible and there is no single error type that flows through the pipeline. | `task/mod.rs:77-80`, `task/verify.rs:6-8` | Consider a shared `FailureReason` newtype or unify the two enums so verification failures map cleanly to task outcomes without ad-hoc string forwarding. |
| 4 | note | `Task::new` is declared `const fn` but constructs `Vec::new()` and `String` (allowed since Rust 1.79 const-eval changes). If the MSRV is below 1.79 this will fail to compile. No issue on current stable, but worth documenting the MSRV requirement. | `task/mod.rs:125` | Document MSRV >= 1.79 or remove `const` qualifier if broader compatibility is needed. |

## Notes

- The task module contains no `unwrap`, `expect`, `panic!`, `todo!`, or `unreachable!` calls. All error-handling risk stems from the Option-heavy design that pushes validation responsibility to callers.
- Resource cleanup is not applicable here; the module is purely data types with no file handles, network connections, or other resources requiring `Drop`.
- The `Attempt` struct records errors as `Option<String>` which is adequate for diagnostics but loses structured error information (e.g., exit codes, affected file paths). This is a design trade-off rather than a defect.
