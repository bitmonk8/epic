# U1-R7: orchestrator / Design intent

## Summary

The orchestrator faithfully implements the core EPIC_DESIGN2 recursive task execution model -- DFS traversal, assessment-driven path selection, fix loops, recovery, checkpoint/discovery flow, and scope circuit breakers are all present and structurally correct. However, several model-selection decisions in the agent layer diverge from the design spec, and the context propagation omits parent decomposition rationale.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | Assessment uses `Model::Sonnet` instead of `Model::Haiku`. AGENT_DESIGN table specifies "Assessment: Haiku (escalate to Sonnet if uncertain)". Using Sonnet for every assessment inflates cost significantly across the task tree. | `agent/flick.rs:199` | Use `Model::Haiku` for assessment; implement conditional escalation to Sonnet when the assessment result indicates uncertainty. |
| 2 | major | `design_and_decompose` hardcodes `Model::Sonnet` instead of using the assessment-selected model. AGENT_DESIGN says "Branch design + decompose: Assessment-selected". A branch assessed as needing Opus will still decompose with Sonnet, potentially producing weaker plans for complex problems. | `agent/flick.rs:258` | Pass the task's `current_model` (from assessment) through the `design_and_decompose` trait method and use it for config generation. |
| 3 | major | `verify` hardcodes `Model::Sonnet` for all tasks. AGENT_DESIGN specifies leaf verification should use `max(Haiku, implementing_model)` capped at Sonnet. Branch verification at Sonnet is correct, but leaf verification should vary -- a Haiku-implemented leaf should verify at Haiku, not Sonnet, wasting cost. | `agent/flick.rs:296` | Have the orchestrator pass the verification model to `verify()`, computed as `max(Haiku, implementing_model).min(Sonnet)` for leaves and `Sonnet` for branches. |
| 4 | major | `assess_recovery` uses `Model::Sonnet` instead of `Model::Opus`. AGENT_DESIGN table specifies "Recovery assessment: Opus -- Requires strongest reasoning". Using Sonnet for recovery assessment risks underperforming on the hardest failure-analysis decisions. | `agent/flick.rs:331` | Change to `Model::Opus` to match the design specification. |
| 5 | major | `build_context` does not include the parent's `decomposition_rationale` or parent-level `discoveries` in `TaskContext`. TASK_MODEL specifies the structural map must inject "Parent: goal, decomposition rationale, discoveries". Agents lack critical framing about why work was decomposed as it was. | `orchestrator.rs:240-324` | Add `parent_decomposition_rationale: Option<String>` and `parent_discoveries: Vec<String>` fields to `TaskContext` and populate them in `build_context`. |
| 6 | minor | No orchestrator-level enforcement of the 2-5 subtask count from decomposition. The limit exists only as a prompt guideline (`prompts.rs:147`). A misbehaving model could return 20 subtasks, causing cost explosion. | `orchestrator.rs:900-917` | Add a guard in `execute_branch` (and `branch_fix_loop`) that caps or rejects decompositions outside a configurable range (e.g., 1-8 with a warning, reject >10). |
| 7 | minor | No commit or rollback operations in the orchestrator. TASK_MODEL specifies "Commit on success, full rollback on terminal failure" for leaves. The orchestrator relies entirely on the agent to handle git operations, with no enforcement or fallback. | `orchestrator.rs` (entire file) | Consider adding explicit git commit after `complete_task_verified` for leaves, and git rollback in `fail_task` for leaves, either in the orchestrator or as a documented delegation to a verification service. |
| 8 | minor | Scope circuit breaker diffs against `HEAD` (`git diff --numstat HEAD`), which measures uncommitted changes only. If sibling tasks have already committed, their changes are invisible to the breaker. This is correct for leaf-scoped work, but may under-count for branch-level scope checks if subtask commits are included in `HEAD`. | `orchestrator.rs:346` | Document the semantic assumption (scope check is per-leaf uncommitted diff). If branch-level scope checks are intended to cover aggregate changes, diff against the branch base instead of HEAD. |
| 9 | note | `checkpoint_save` silently swallows errors (`eprintln` only). While the comment explains the rationale (avoid aborting the run), repeated checkpoint failures could lead to significant state loss on crash with no visibility beyond stderr. | `orchestrator.rs:137-143` | Consider emitting an `Event::CheckpointFailed` so the TUI can surface persistent save failures to the user. |
| 10 | note | The depth-cap forced-leaf assessment uses `Model::Sonnet` rather than running the normal Haiku assessment. This is a reasonable conservative choice but diverges from the design flow (assessment determines model). The forced leaf may not need Sonnet. | `orchestrator.rs:427-431` | Consider running normal assessment even at max depth (just force `path = Leaf`), so the model is still right-sized by the assessor. |

## Notes

- Findings 1-4 are in `agent/flick.rs` (the concrete `AgentService` implementation) rather than `orchestrator.rs` itself. The orchestrator correctly delegates model decisions to the agent layer, but the agent layer does not implement the model-selection rules from AGENT_DESIGN. These are reported here because they represent design-intent violations visible in the orchestration flow.
- The orchestrator's fix loop implementation (both leaf and branch) closely matches FIX_LOOP_SPEC, including resume-safe retry counting, scope circuit breaker placement, and the fix-task recursion guard.
- Recovery machinery (assessment, re-decomposition, round budgets, full vs incremental strategies) aligns well with TASK_MODEL's recovery section.
- The checkpoint/discovery/adjust/escalate flow is implemented correctly per TASK_MODEL's discovery flow section.
- Test coverage in the file is extensive, covering happy paths, retry/escalation, terminal failure, resume, discoveries, checkpoint decisions, recovery, and scope checks.
