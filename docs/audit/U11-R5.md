# U11-R5: init / Simplification

## Summary
The init module contains duplicated stdin-locking patterns across `prompt_models` and `prompt_limits`, and the interactive prompt helpers could be consolidated with a small macro or helper to reduce the repetitive print-flush-read boilerplate.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `prompt_models` and `prompt_limits` duplicate the same structure: lock stdin, show defaults, ask Y/n, optionally prompt each field, drop lock. These two functions share identical scaffolding and differ only in which fields they collect. | `prompt_models` (L221-248), `prompt_limits` (L250-286) | Extract a generic `prompt_defaults_or_edit` helper that accepts a default value, a display closure, and a field-prompting closure, eliminating the duplicated accept-defaults gate. |
| 2 | minor | `prompt_models` and `prompt_limits` each independently lock stdin (`io::stdin().lock().lines()`), while `present_and_confirm` also locks stdin separately. If these are always called sequentially from `run_init`, the lock could be acquired once and threaded through. | L90, L228-229, L260-261 | Acquire the stdin lock once in `run_init` and pass it to all three prompt functions, removing three separate lock/drop pairs. |
| 3 | minor | Repetitive print-flush-read triplet (`print!(...); io::stdout().flush()?; read_line_or_default(...)`) appears 9 times across `edit_step`, `prompt_custom_step`, `prompt_models`, and `prompt_limits`. | L163-165, L169-171, L179-181, L194-196, L201-203, L209-211, L237-239, L241-243, L269-271 (and more) | Introduce a small `prompt(label, default, lines) -> String` helper that encapsulates the print, flush, and read-with-default pattern. |
| 4 | note | `read_line_checked` and `read_line_or_eof` are nearly identical -- they differ only in the `None` arm (bail vs empty string). | L289-308 | Consider merging into a single function with a parameter or using `read_line_or_eof` everywhere with an explicit empty-check at the one call site that needs the error. |
| 5 | note | `edit_step` and `prompt_custom_step` build a `VerificationStep` with the same three fields and share the same command-parsing logic (`split_whitespace().map(String::from).collect()`). | L156-189, L191-219 | Extract the command-parsing and step-construction into a shared builder, or unify the two functions with an `Option<&DetectedStepWire>` parameter for defaults. |
| 6 | note | The explicit `drop(lines)` calls (L137, L234, L246, L284) are unnecessary since the lock would be dropped at end of scope anyway; `present_and_confirm` already wraps its lock in a block scope. | L137, L234, L246, L284 | Remove the explicit `drop` calls and, if desired, wrap the relevant sections in block scopes for clarity, as `present_and_confirm` already does. |

## Notes
The file is only ~325 lines and each function is individually clear, so none of these findings are severe. The main opportunity is reducing the copy-paste interactive-prompt scaffolding -- a single `prompt` helper plus one shared stdin lock would cut roughly 30-40 lines and make it easier to add new prompted config sections in the future.
