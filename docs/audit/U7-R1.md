# U7-R1: task / Correctness

## Summary

The task module types are structurally sound and the data model is well-designed for crash-safe resume. The primary correctness issues are in how the orchestrator consumes these types: hardcoded constants shadow user-configurable limits, phase transitions lack legality checks, and the retry event emits a potentially misleading attempt number on resume.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `MAX_DEPTH` (8), `MAX_RECOVERY_ROUNDS` (2), and `RETRIES_PER_TIER` (3) are hardcoded constants in the orchestrator, but `LimitsConfig` exposes user-configurable `max_depth`, `max_recovery_rounds`, and `retry_budget` fields that are collected during `init` and serialized to project config yet never read back by the orchestrator. Users who change these settings get no effect. | `orchestrator.rs:15-19`, `config/project.rs:44-49` | Have `Orchestrator` accept `LimitsConfig` and use its values instead of the module-level constants. |
| 2 | minor | `transition()` performs no validation of legal phase changes -- any `TaskPhase` can transition to any other, including nonsensical paths like `Completed -> Pending` or `Failed -> Assessing`. A bug in a caller could silently produce an invalid state machine trajectory that persists through checkpoint. | `orchestrator.rs:149-157`, `task/mod.rs:27-34` | Add a `TaskPhase::can_transition_to(&self, next: TaskPhase) -> bool` method and assert/error in `transition()`. |
| 3 | minor | In `execute_leaf`, when resuming with pre-existing attempts, the `RetryAttempt` event's `attempt` field is set to the local `retries_at_tier` counter, which counts only consecutive trailing attempts at the current tier. On a fresh run this means the first retry emits `attempt: 1` (correct), but on resume after e.g. 2 prior Haiku failures, `retries_at_tier` starts at 2 and the event emits `attempt: 2` with no `attempt: 0` or `attempt: 1` ever emitted in that session. This is cosmetic but could confuse TUI/log consumers. | `orchestrator.rs:858-865` | Use the total `task.attempts.len()` for the event field, or document that the counter is tier-local and may skip values on resume. |
| 4 | note | `Task::model` (the originally assessed model) is written once during assessment but never read anywhere in the orchestrator; only `current_model` is used for execution. The field is not dead -- it serves as an audit trail in serialized state -- but if it is truly write-only in logic, consider documenting that intent or removing it to avoid confusion with `current_model`. | `task/mod.rs:108-109`, `orchestrator.rs:445` | Add a doc comment clarifying `model` is the initial assessment record vs `current_model` which tracks escalation, or remove if unneeded. |
| 5 | note | `LeafResult` does not derive `Serialize`/`Deserialize` (unlike nearly every other type in the module). Since leaf results are consumed inline and not persisted, this is intentional, but it means a future change to checkpoint mid-leaf-execution would require adding the derives. | `task/mod.rs:83-87` | No action needed unless checkpoint granularity changes; note for future reference. |

## Notes

- The task module files themselves (`mod.rs`, `assess.rs`, `leaf.rs`, `branch.rs`, `verify.rs`) are primarily type definitions with no business logic beyond `Model::escalate()`. The escalation ladder (`Haiku -> Sonnet -> Opus -> None`) is correct and exhaustive. The `const fn new` constructor correctly initializes all fields to safe defaults.
- Rust 2024 edition (1.85) supports `const fn` with `String` and `Vec` parameters, so `Task::new` being `const` is valid.
- Finding #1 is the most impactful: users can configure limits via `epic.toml` but the orchestrator ignores them entirely, using hardcoded values instead. This is a silent misconfiguration bug.
