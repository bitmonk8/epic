# U7-R5: task / Simplification

## Summary
The task module is reasonably lean for a set of domain types, but contains structural duplication between `TaskOutcome` and `VerificationOutcome`, redundant fields on the `Task` struct, and a `rationale` pattern repeated across multiple structs that could benefit from a shared trait or newtype.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `TaskOutcome` and `VerificationOutcome` are structurally identical enums (`Success`/`Pass` vs `Failed{reason}`/`Fail{reason}`). They carry the same data shape, and consumer code in `orchestrator.rs` frequently converts between the two semantically (e.g., `VerificationOutcome::Pass` leads to `TaskOutcome::Success`). | `mod.rs:77-80`, `verify.rs:5-9` | Unify into a single `Outcome` enum (or make `VerificationOutcome` a type alias). If distinct naming is truly needed, a single generic `Outcome<S, F>` or a shared trait would remove the duplication and the manual mapping code in the orchestrator. |
| 2 | minor | `Task` has both `model: Option<Model>` and `current_model: Option<Model>`. The `model` field is set during assessment and never mutated afterward, while `current_model` tracks the escalation state. Having two nullable model fields is confusing and easy to misuse. | `mod.rs:108-109` | Rename `model` to `assessed_model` (or `initial_model`) to make the distinction explicit, or merge them: store only `current_model` and derive the assessed model from the first `Attempt` entry or from `AssessmentResult` metadata. |
| 3 | minor | `Task` carries both `attempts: Vec<Attempt>` and `fix_attempts: Vec<Attempt>` with the same `Attempt` type. The split exists only to distinguish execution retries from verification-fix retries. | `mod.rs:110, 116` | Consider a single `Vec<Attempt>` with an added `AttemptKind { Execute, Fix }` discriminator on `Attempt`, or group them into a nested `RetryHistory` struct. This removes the parallel logic in the orchestrator that counts trailing attempts identically across both vectors. |
| 4 | note | The `rationale: String` field appears in `AssessmentResult`, `DecompositionResult`, `RecoveryPlan`, and multiple wire types in `config_gen.rs`. While not strictly duplicated logic, a `Rationale` newtype (`pub struct Rationale(pub String)`) would add type safety and make grep/refactoring easier. | `assess.rs:10`, `branch.rs:16`, `mod.rs:96` | Introduce `pub struct Rationale(pub String)` in `mod.rs` and use it across all result types. |
| 5 | note | `LeafResult` in `mod.rs` does not derive `Serialize`/`Deserialize` even though every other struct in the module does. It also does not derive `PartialEq`/`Eq`, unlike `TaskOutcome` which it wraps. This inconsistency may cause friction if persistence or testing needs expand. | `mod.rs:83-87` | Add `#[derive(Serialize, Deserialize, PartialEq)]` to `LeafResult` for consistency with the rest of the module. |
| 6 | note | `Task::new` is declared `const fn` but takes `String` and `Vec<String>` parameters. Const constructors with heap-allocated types cannot actually be called in const contexts today, making the `const` annotation misleading. | `mod.rs:125` | Remove the `const` qualifier or document why it is kept (e.g., future Rust const-heap support). |
| 7 | note | `leaf.rs` is effectively empty (two comment lines, no types or logic). Its existence as a separate file adds navigational overhead without contributing any code. | `leaf.rs:1-2` | Either add leaf-specific types here (e.g., move `LeafResult` from `mod.rs`) or remove the file and the `pub mod leaf;` declaration until it is needed. |

## Notes
- The task module is primarily a data-model crate; the real complexity lives in `orchestrator.rs` which consumes these types. Several findings (especially #1 and #3) would yield larger simplification wins in the orchestrator than in the task module itself.
- The `#[allow(clippy::struct_field_names)]` suppression on `Magnitude` (mod.rs:69) is a minor code smell -- all three fields share the `max_lines_` prefix, which is what Clippy flags. Renaming to `added`/`modified`/`deleted` inside the struct (the struct name already conveys "magnitude") would let the allow be removed.
