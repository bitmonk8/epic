# U4-R6: agent/prompts / Testability

## Summary

The module is a pure function library with no side effects, making it inherently testable. Existing tests cover every public builder but rely on shallow substring assertions and leave `format_context` edge cases and structural prompt guarantees under-tested.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | Tests use only `contains()` substring checks, which are fragile and can pass even if prompt structure is malformed (e.g., section headers missing, fields in wrong order). No test validates the overall shape of the rendered prompt. | `prompts.rs:386-494` | Add at least one snapshot test (e.g., via `insta`) or structured assertion that checks section ordering and header presence for a representative prompt. |
| 2 | minor | `format_context` is `fn` (private), so it can only be tested indirectly through the public builders. The two dedicated tests (`context_format_includes_checkpoint_guidance`, `context_format_with_no_siblings`) call it directly only because they are inside the same module. If the function were ever extracted to a shared utility crate, these tests would break. | `prompts.rs:12-77` | Consider making `format_context` `pub(crate)` so tests in sibling modules can exercise it directly, and document this as the canonical unit-test surface for context rendering. |
| 3 | minor | No test exercises `format_context` when `completed_siblings` contains a `TaskOutcome::Failed` variant. The match arm at line 39 that renders `"FAILED"` is never covered by any test assertion. | `prompts.rs:37-40` | Add a `test_context` variant (or a dedicated test) that includes a failed sibling and assert the output contains `[FAILED]`. |
| 4 | minor | `test_context()` helper is duplicated inline; any change to `TaskContext` fields requires updating this one fixture. There is no builder or factory pattern, increasing maintenance cost as fields are added. | `prompts.rs:364-384` | Introduce a small `TaskContextBuilder` (or a `Default` impl gated behind `#[cfg(test)]`) to reduce boilerplate and make new-field additions a single-point change. |
| 5 | note | All prompt text is inlined as string literals with no mechanism for override or injection. This is fine for the current architecture, but if prompt A/B testing or runtime configuration is ever needed, the lack of an indirection layer (trait, template engine, config file) will require touching every builder. | `prompts.rs:80-93, 109-122, ...` | No action needed now. If prompt experimentation becomes a goal, extract prompt text into a `PromptTemplates` struct that builders accept as a parameter. |
| 6 | note | The `build_fix_leaf` test at line 443 asserts `pair.query.contains("2")` for the attempt number, but `"2"` is a single character that could match other content accidentally. Similarly, `build_design_fix_subtasks` asserts `contains("2")` for round. | `prompts.rs:443, 491` | Assert a more specific substring like `"Attempt: 2"` or `"Round: 2"` to avoid false positives. |

## Notes

- The module has zero external dependencies beyond plain data structs (`TaskContext`, `Task`, `TaskOutcome`), meaning there are no mock boundaries to worry about -- every function is a pure data-in, data-out transformation. This is an ideal testability posture.
- The existing 11 tests provide good breadth coverage of every public function. The gaps are in depth: edge-case variants, structural correctness, and assertion precision.
- No state machines or mutable state exist in this module; all functions are stateless builders.
