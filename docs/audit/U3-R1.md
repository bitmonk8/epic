# U3-R1: agent/config_gen / Correctness

## Summary

The wire-format conversion layer is generally sound. The main issues are missing validation of empty collections that would produce semantically invalid domain objects, silently discarded data on certain variant combinations, and JSON schemas that lack `additionalProperties: false` (which may cause the LLM provider to accept malformed payloads without error).

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `DecompositionWire::try_from` accepts an empty `subtasks` vec without error. Every downstream consumer (`orchestrator.rs:735`, `:912`) must guard against this separately; the branch at `:735` retries in a loop, but the decompose path at `:912` proceeds to `create_subtasks` with an empty vec. Validation should be in the conversion so callers cannot receive an invalid `DecompositionResult`. | `config_gen.rs:81-94` | Add `if subtasks.is_empty() { bail!("decomposition must contain at least one subtask"); }` before constructing `DecompositionResult`. |
| 2 | major | `RecoveryPlanWire::try_from` similarly accepts an empty `subtasks` vec. The orchestrator checks this later (`orchestrator.rs:1106`), but the domain type `RecoveryPlan` is conceptually invalid with zero subtasks. Pushing the invariant into the conversion prevents future callers from forgetting the check. | `config_gen.rs:183-202` | Add an empty-subtasks guard in `try_from`. |
| 3 | minor | `TaskOutcomeWire::try_from` silently discards `reason` when `outcome == "success"`. If the LLM returns `{ "outcome": "success", "reason": "..." }`, the reason is lost with no warning. This is unlikely to cause runtime failure but masks potentially useful diagnostic data. | `config_gen.rs:106-112` | Log a warning (or return an error) if `reason` is `Some` when `outcome` is `"success"`. |
| 4 | minor | `DetectedStepWire::From` does not validate that `command` is non-empty. An empty command vec will be accepted and later cause a panic or confusing error when the command is spawned. The JSON schema also does not set `minItems: 1` on the `command` array. | `config_gen.rs:461-469` | Validate `!w.command.is_empty()` (change `From` to `TryFrom`) and add `"minItems": 1` to `init_findings_schema`. |
| 5 | minor | None of the JSON schemas set `"additionalProperties": false`. Depending on the LLM provider's structured-output implementation, extra keys in the response may pass schema validation silently and then be ignored by serde deserialization (with default settings). While not a logic error per se, it weakens the contract and can hide LLM hallucinated fields. | `config_gen.rs:208-321`, `479-520` | Add `"additionalProperties": false` to each object schema, including nested subtask objects. |
| 6 | note | `RecoveryWire::into_strategy` returns `Some(String::new())` when `recoverable == true` and `strategy` is `None`. Callers receiving `Some("")` must treat an empty strategy string as "no specific strategy" rather than a meaningful value. This is tested and intentional, but the empty-string sentinel is fragile. | `config_gen.rs:171-173` | Consider returning `Some(None)` (i.e., `Option<Option<String>>`) or requiring `strategy` when `recoverable` is true in the schema. |
| 7 | note | All string matching for enums (`"leaf"`, `"success"`, `"pass"`, etc.) is case-sensitive. If the LLM returns `"Success"` or `"PASS"`, the conversion fails. The JSON schemas use `enum` constraints which should prevent this at the provider level, but not all providers enforce `enum` strictly. | `config_gen.rs:31-34`, `106-111`, `130-135`, `153-159`, `186-189`, `537-541` | Consider normalising with `.to_lowercase()` before matching, or document that case-sensitivity is intentional and enforced by the schema. |

## Notes

- The orchestrator already has defensive checks for empty subtask vecs in several places, so findings 1 and 2 do not represent live bugs today. They are architectural: the domain types permit invalid states that callers must remember to guard against.
- Finding 5 depends on how the Flick library and the Anthropic API handle `additionalProperties`. If the provider already strips extra keys, the risk is minimal.
- The test suite covers the happy paths and several error cases well, but does not test empty-subtask decompositions or empty-command verification steps.
