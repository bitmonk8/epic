# U8-R3: state / Error handling

## Summary
The `state.rs` module has clean error handling in production code, using `anyhow::Result` with proper propagation via `?`. The atomic-write pattern in `save()` includes resource cleanup. One minor issue exists around `save()` not fsyncing before rename, and the `dfs_order` method silently swallows missing task IDs rather than signalling an inconsistency.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | `save()` does not call `fsync` on the temp file before renaming. On some filesystems/OS crash scenarios the renamed file may be empty or truncated. | `src/state.rs:64-71` | Open the file via `File::create`, write, call `file.sync_all()?`, then rename. |
| 2 | minor | `dfs_order` silently ignores missing task IDs (line 45: `if let Some(task)`). If the task map is inconsistent (a `subtask_ids` entry points to a non-existent task), this is silently dropped with no warning, which could mask data-corruption bugs. | `src/state.rs:40-53` | Log a warning or return a `Result` when a referenced task ID is missing from the map. |
| 3 | note | `load()` provides no context on deserialization failure (e.g., which field was unexpected). `anyhow` propagates the `serde_json` error, but an `.context("loading epic state from ...")` would aid debugging. | `src/state.rs:74-78` | Add `.context(format!("loading state from {}", path.display()))` to the `serde_json::from_str` call. |
| 4 | note | `save()` cleans up the temp file on rename failure (`let _ = std::fs::remove_file`) but silently ignores cleanup failure. Acceptable in practice, but worth a debug-level log. | `src/state.rs:68` | Add a `tracing::debug!` or `eprintln!` if temp-file cleanup fails. |

## Notes
- All `unwrap()` calls in this file are confined to the `#[cfg(test)]` module and are appropriate for test code.
- The production API surface (`get`, `get_mut`, `insert`, `save`, `load`) correctly returns `Option` or `Result` and never panics.
- The atomic-write pattern (write to `.tmp`, rename) is a solid choice for crash safety; the fsync gap (finding 1) is the only weakness in that strategy.
