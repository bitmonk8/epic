# U11-R6: init / Testability

## Summary

The `init` module has zero test coverage and its core logic is tightly coupled to concrete I/O (stdin/stdout, filesystem, and the `FlickAgent` struct), making it effectively untestable without integration-level setup. Separating pure logic from side effects and introducing trait-based seams would unlock unit testing for the majority of the module.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|

All flows (accept, decline, edit, custom step, model prompts, limits prompts) are unverified. | `src/init.rs` (entire file) | Add unit tests for each user-interaction path using an injected line iterator (see #3). |

## Notes

- The helper functions `edit_step` and `prompt_custom_step` already accept generic iterators, which is good design -- it just is not leveraged by their callers or by any tests.
- The atomic write pattern (write-to-tmp then rename) is a solid choice, but verifying atomicity requires filesystem-level tests that do not yet exist.
- A recommended first step would be to refactor `present_and_confirm` to accept an iterator, then write table-driven tests covering the accept/decline/edit matrix. This single change would cover the majority of the module's branching logic.
