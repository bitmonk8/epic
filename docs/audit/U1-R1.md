# U1-R1: orchestrator / Correctness

## Summary

The orchestrator is well-structured with careful resume-safety handling. Most logic is sound, but there are a few edge cases around the `execute_branch` loop control flow, scope checking semantics, and a missed `PhaseTransition` event on recovery-superseded children that could lead to subtle misbehavior.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `execute_branch` outer loop can silently declare success when a child fails and recovery creates new subtasks, but those new subtasks also fail. After `attempt_recovery` returns `None` (recovery subtasks created), the inner `for` loop `break`s to restart the outer loop. On the next iteration, the failed original child and the failed recovery child are both skipped (`Completed \| Failed => continue`), and if no pending children remain, `all_done` stays `true` and the branch returns `Ok(TaskOutcome::Success)` even though every child failed. The branch-level verification may catch this, but the orchestrator reports `Success` from `execute_branch` before verification runs. | `orchestrator.rs:928-1018` | After the outer loop exits, check that at least one non-fix child completed successfully, or rely solely on verification but document the invariant clearly. |
| 2 | minor | `check_scope_circuit_breaker` runs `git diff --numstat HEAD` which measures uncommitted working-tree changes against the last commit. If a leaf task commits its changes (e.g., via the agent), the diff resets to zero and the scope check becomes ineffective. The circuit breaker only works if changes are left uncommitted. | `orchestrator.rs:345-356` | Consider diffing against a baseline ref (e.g., the branch point or a tag set at task start) rather than `HEAD`, so committed changes are also counted. |
| 3 | minor | When `full_redecomposition` marks pending children as `Failed`, only a `TaskCompleted` event is emitted. The `PhaseTransition` event to `Failed` is skipped because the phase is set directly on the task struct rather than going through `self.transition()`. Any TUI or event consumer tracking phase state will be out of sync. | `orchestrator.rs:1131-1145` | Use `self.transition(child_id, TaskPhase::Failed)?` (or emit `PhaseTransition` manually) before emitting `TaskCompleted`. |
| 4 | minor | `evaluate_scope` uses a hardcoded `multiplier = 3`. If `Magnitude` fields are zero (which is a valid value for the type), the limit is `0 * 3 = 0`, so any single line of change triggers `Exceeded`. A magnitude of zero lines added/modified/deleted is unlikely to be intentional and would cause immediate failure on the first scope check. | `orchestrator.rs:54-75` | Either validate that `Magnitude` fields are non-zero when constructing them, or treat a zero limit as "unlimited" in `evaluate_scope`. |
| 5 | minor | In `leaf_fix_loop`, the `attempt_number` counter (line 574-580) is derived from `fix_attempts.len() + 1` and passed to the agent, but it only counts fix attempts, not the original execution attempts. If the agent uses this number to decide strategy, it sees attempt 1 even when the task has already been tried 9 times in `execute_leaf`. This is potentially confusing but may be intentional. | `orchestrator.rs:573-580` | Clarify whether `attempt_number` should reflect total attempts or just fix-loop attempts. If fix-only, add a doc comment. |
| 6 | note | The `execute_branch` checkpoint discovery check (lines 946-996) runs the checkpoint agent call for every child that has discoveries, even children that were already completed on a previous run (resume scenario). On resume, completed children are skipped by the `Completed => continue` match arm (line 938), so this is actually safe. However, if a child finishes and the process crashes between `execute_task` returning and the checkpoint call, on resume the child will be skipped as `Completed` and its discoveries will never be evaluated by the checkpoint agent. | `orchestrator.rs:938-996` | For crash-safety, consider persisting a "checkpoint evaluated" flag per child, or re-evaluating discoveries for children that completed since the last checkpoint. |
| 7 | note | `attempt_recovery` increments `recovery_rounds` and checkpoints (lines 1080-1087) before designing subtasks. If `design_recovery_subtasks` then fails (returns `Err` wrapped as `Ok(Some(Failed))`), the round is consumed but no subtasks were created. This is documented as intentional ("round already consumed"), but it means a transient agent error permanently costs a recovery round. | `orchestrator.rs:1080-1104` | Acceptable given the design goal of bounded retries, but worth noting that a transient LLM error during planning wastes a recovery slot. |

## Notes

- The code has thorough resume-safety handling throughout, with careful attention to counting persisted attempts to avoid granting extra retries after crashes. This is a notable strength.
- The recursive `execute_task -> execute_branch -> execute_task` pattern via `Pin<Box<dyn Future>>` is correctly structured for async recursion.
- Finding #1 is the most consequential: a branch can report success when all children have failed, relying entirely on the downstream verification step to catch the problem. If the verification agent is lenient or buggy, incorrect results could propagate.
- The test suite is extensive and covers most of the identified edge cases except for finding #1 (all-children-failed-after-recovery scenario) and finding #3 (missing PhaseTransition event).
