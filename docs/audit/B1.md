# B1: Duplicated patterns across modules

## Summary

Cross-module review of `orchestrator.rs`, `agent/flick.rs`, `agent/config_gen.rs`, and `init.rs` reveals three duplicated patterns where shared abstractions would reduce code without harming readability, plus one structural duplication worth noting.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | **Retry-then-escalate loop duplicated between `execute_leaf` and `leaf_fix_loop`**. Both methods in `orchestrator.rs` implement an identical state-machine: resume tier count from persisted attempts, drain stale tiers via `while retries_at_tier >= RETRIES_PER_TIER`, loop with `checkpoint_save` after each attempt, escalate via `model.escalate()`, and terminate on tier exhaustion. The two copies differ only in which agent method is called (`execute_leaf` vs `fix_leaf`) and which attempt vec is used (`attempts` vs `fix_attempts`). This is ~120 lines of duplicated control flow. | `orchestrator.rs` lines 519-671 (`leaf_fix_loop`) and 776-888 (`execute_leaf`) | Extract a generic `retry_with_escalation` helper parameterized by a closure for the agent call and an accessor for the attempt list. The helper owns the tier-counting, escalation events, and checkpoint saves. Both call sites become ~10 lines each. |
| 2 | minor | **Verify-then-branch-on-outcome sequence duplicated across `finalize_task`, `leaf_fix_loop`, and `branch_fix_loop`**. Three sites in `orchestrator.rs` perform the same sequence: emit `VerificationStarted`, call `agent.verify`, match on `Pass`/`Fail`, emit `VerificationComplete` on failure, and then branch. The post-verification handling differs, but the verify-and-dispatch preamble (~10 lines) is identical each time. | `orchestrator.rs` lines 477-506, 622-638, 755-770 | Extract a small `run_verification(&mut self, id) -> Result<VerificationOutcome, OrchestratorError>` that handles the emit-verify-emit sequence and returns the outcome enum. Each call site then `match`es only on the result. |
| 3 | minor | **Agent-call-then-parse sequence is structurally identical across all `AgentService` impl methods in `flick.rs`**. Every method follows the same 5-step pattern: (a) build prompt pair via `prompts::build_*`, (b) resolve `ToolGrant`, (c) build config via `config_gen::build_*_config`, (d) call `run_structured` or `run_with_tools`, (e) convert wire type via `try_from`. The only varying inputs are the prompt builder, config builder, wire type, domain type, and whether tools are used. This is not high-severity because each method is short (~10 lines), but the rigid pattern means adding a new agent method requires duplicating the same boilerplate. | `flick.rs` lines 194-359 (9 methods) coupled with `config_gen.rs` lines 375-446 (8 `build_*_config` functions) | Consider a table-driven or macro approach only if method count grows beyond ~12. Currently the duplication is tolerable because each method is concise and the indirection cost of a generic dispatcher would hurt readability. Flag for revisit. |
| 4 | note | **`error-to-string` formatting via `anyhow::anyhow!("...: {e}")` vs `format!`/`.context()`**. `init.rs` uses `anyhow::anyhow!("failed to serialize config: {e}")` (line 36), `flick.rs` uses `anyhow::anyhow!("failed to resolve provider: {e}")` and `anyhow::anyhow!("flick call failed: {e}")` (lines 53, 75), and `config_gen.rs` uses `.context()` and `anyhow::anyhow!("failed to parse flick config: {e}")` (lines 81, 372). The inconsistency between `.context()` (which preserves the error chain) and `anyhow!("...: {e}")` (which flattens it) is a cross-module style divergence. It is not a bug today because these errors are terminal, but it makes structured error reporting harder later. | `init.rs:36`, `flick.rs:53,74-75,103-104,134-135`, `config_gen.rs:370-372` | Standardize on `.context("failed to ...")` throughout to preserve error chains. The `map_err(\|e\| anyhow!(...))` pattern should only be used when the source error does not implement `std::error::Error`. |

## Notes

- Finding 1 is the highest-value refactor target. The two retry-escalation loops are long, structurally identical, and both need to stay in sync for correctness (e.g., checkpoint placement, tier-drain logic). A divergence bug between them would be difficult to catch in review.
- Finding 3 is deliberately rated low because the current method count (9) is manageable and each method body is short. The coupling between `flick.rs` and `config_gen.rs` (1:1 method-to-config-builder mapping) is the real concern -- if a new agent method is added to the trait, both files must be updated in lockstep with no compiler enforcement beyond the trait itself.
- `init.rs` shares minimal structural patterns with the other three modules. Its only cross-module touchpoint is the `FlickAgent::explore_for_init` call, which already follows the same `run_with_tools` path as the `AgentService` methods. No actionable duplication was found between `init.rs` and the other modules beyond Finding 4.
