# U12-R6: cli + main / Testability

## Summary

The `main` function is a 200-line monolith that tangles argument parsing, filesystem I/O, state validation, agent construction, orchestrator wiring, and TUI lifecycle into a single untestable `async fn main`. There are no tests for any main.rs logic, and 10 `std::process::exit` calls make the function hostile to in-process testing.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | critical | `main()` has zero test coverage -- all command dispatch, state-loading validation, resume-vs-fresh logic, and TUI/headless branching is untested | `src/main.rs:27-207` | Extract an `async fn run(cli: Cli, project_root: PathBuf) -> anyhow::Result<()>` (or similar) that takes injected paths and an `AgentService` impl, returning `Result` instead of calling `process::exit`. Test that function directly. |
| 2 | major | 10 calls to `std::process::exit()` bypass destructors and make it impossible to assert error paths in tests | `src/main.rs` lines 47, 73, 83, 93, 101, 123, 131, 141, 151, and `print_status` at 220 | Return typed errors (e.g., `CliError::CorruptState`, `CliError::GoalMismatch`) and let the top-level `main` map them to exit codes. |
| 3 | major | `FlickAgent` is constructed directly in `main` with no seam for injection; integration tests must either hit the real agent or duplicate all the setup logic | `src/main.rs:53-57` | Accept `A: AgentService` generically (the trait already exists in `agent/mod.rs`) so tests can supply `MockAgentService`. |
| 4 | major | State-loading and validation logic is duplicated between `Command::Run` (lines 64-118) and `Command::Resume` (lines 120-157) with near-identical error handling branches | `src/main.rs:64-157` | Extract a shared `fn load_and_validate_state(path) -> Result<(EpicState, TaskId, String)>` to eliminate duplication and allow unit testing of each validation branch. |
| 5 | minor | `print_status` uses `process::exit(1)` on load failure instead of returning an error, making its error path untestable | `src/main.rs:209-257` | Return `anyhow::Result<()>` for the error case instead of exiting, which is already the function's return type. |
| 6 | minor | Hard-coded `Duration::from_secs(300)` timeout and `Duration::from_secs(2)` TUI-shutdown timeout are not configurable, making time-sensitive behavior untestable | `src/main.rs:51,185` | Accept timeouts via `Cli` args or a config struct so tests can use short/zero durations. |
| 7 | minor | `std::env::current_dir()` is called at the top of `main` with no override mechanism; tests that need different project roots must manipulate the process working directory (global mutable state) | `src/main.rs:30` | Pass `project_root` as a parameter to the extracted `run` function. |
| 8 | note | `cli.rs` itself is clean and fully derives `Parser`/`Subcommand`; clap's own test utilities (`Cli::try_parse_from`) can exercise it, but no such tests exist | `src/cli.rs:1-31` | Add a small `#[cfg(test)]` module with round-trip parse tests for each subcommand variant and env-var defaults. |
| 9 | note | The TUI branch spawns `tokio::spawn` + `spawn_blocking` and races them with `tokio::time::timeout`; this lifecycle logic has no test and is difficult to test because it is inlined in `main` | `src/main.rs:172-203` | Move TUI-orchestrator lifecycle management into a dedicated function that accepts channels/handles, enabling tests with mock channels. |

## Notes

- The `AgentService` trait in `src/agent/mod.rs` and the `MockAgentService` already present in `src/orchestrator.rs` tests demonstrate the codebase has the right abstraction for testability -- it just is not wired through `main`.
- `cli.rs` is small and well-structured; the testability problems are concentrated entirely in `main.rs`.
- The pattern of `eprintln!` + `process::exit(1)` appears 10 times; a single refactor to return `CliError` variants would fix findings 2 and 5 simultaneously and unlock test coverage for every error branch.
