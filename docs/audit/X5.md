# X5: Global patterns

## Summary
The codebase exhibits strong naming consistency and well-organized module boundaries overall. Key issues are: three stub modules with no code, hardcoded orchestrator constants that duplicate configurable values, and an overly large `config_gen.rs` that bundles wire types, schemas, and builder functions in a single 500+ line file.

## Findings
| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | major | `services/document_store.rs`, `services/research.rs`, and `services/verification.rs` are empty stubs (comment-only). The `services` module is declared in `main.rs` but never imported anywhere. Dead module tree ships in the binary. | `src/services/mod.rs`, `src/services/*.rs` | Remove the `services` module entirely or gate it behind a feature flag until implementation begins. |
| 2 | major | `src/git.rs` and `src/metrics.rs` are also empty stubs (comment-only) declared in `main.rs` but never used. | `src/git.rs`, `src/metrics.rs` | Remove or feature-gate until implemented. |
| 3 | major | Orchestrator hardcodes `MAX_DEPTH = 8`, `RETRIES_PER_TIER = 3`, `MAX_RECOVERY_ROUNDS = 2` as private constants, while `config/project.rs` exposes identical values via `LimitsConfig` (with the same defaults). The config is never loaded by the orchestrator, so user-supplied `epic.toml` limits are silently ignored at runtime. | `src/orchestrator.rs:15-19`, `src/config/project.rs:43-50` | Have the orchestrator accept a `LimitsConfig` (or read it from the loaded `EpicConfig`) so that the TOML values actually take effect. |
| 4 | minor | `config_gen.rs` is 550+ lines mixing ten `*Wire` structs, seven `*_schema()` functions, seven `build_*_config()` functions, and `From`/`TryFrom` impls. This makes the file hard to navigate and review. | `src/agent/config_gen.rs` | Split into submodules: `wire_types.rs` for serde structs + conversions, `schemas.rs` for JSON schema generators, `builders.rs` for config builders. |
| 5 | minor | `Task::new` is declared `const fn` but allocates `Vec::new()` nine times. This compiles on nightly/edition 2024 but is fragile and will not compile on stable Rust < 1.83. No `rust-version` is declared in `Cargo.toml` to enforce the MSRV. | `src/task/mod.rs:125` | Either drop `const` from `Task::new` or add `rust-version = "1.83"` (or later) to `Cargo.toml`. |
| 6 | minor | `Orchestrator::new` is `const fn` but takes a generic `A: AgentService`. This works only because no trait methods are called in the constructor, but it prevents adding any non-const initialization later and is unusual for a struct that owns a `PathBuf` option. | `src/orchestrator.rs:89` | Remove the `const` qualifier; the constructor is never called in a const context. |
| 7 | minor | `EpicConfig` is loaded in `init.rs` but never in `main.rs` for the `Run`/`Resume` paths. The entire config subsystem (`config/project.rs`) is effectively write-only at runtime. | `src/main.rs`, `src/config/project.rs` | Load `epic.toml` during `Run`/`Resume` and pass the resulting `EpicConfig` into the orchestrator and agent. |
| 8 | note | Five `#[allow(clippy::too_many_lines)]` annotations appear in `orchestrator.rs` and `tui/mod.rs`. While the justifications are reasonable, the orchestrator file is ~1170 lines of non-test code with deeply nested async recursion via boxed futures. | `src/orchestrator.rs` | Consider extracting leaf-fix, branch-fix, and recovery logic into separate methods or a sub-module to reduce cognitive load. |
| 9 | note | `agent/mod.rs` re-exports `pub mod config_gen`, `pub mod flick`, and `pub mod tools`, but keeps `models` and `prompts` private. This is intentional (only `flick.rs` uses them), but it means `config_gen` wire types are publicly reachable from outside the crate despite the binary-only crate boundary. | `src/agent/mod.rs` | If this is intended as a library in the future, tighten visibility to `pub(crate)` for `config_gen` and `tools`. |
| 10 | note | Naming is consistent throughout: snake_case for functions/modules, CamelCase for types, `*Wire` suffix for serde DTOs, `*Widget` for TUI components, `*Result`/`*Outcome` for domain results. No naming anomalies detected. | project-wide | No action needed. |
| 11 | note | All `mod` declarations in `main.rs` are private (no `pub`), which is correct for a binary crate. Sub-module re-exports inside `task/mod.rs`, `agent/mod.rs`, `tui/mod.rs`, and `config/mod.rs` are appropriately scoped. | `src/main.rs`, `src/*/mod.rs` | No action needed. |

## Notes
- The codebase compiles as a binary crate (`main.rs`), so `pub` visibility on items inside modules is effectively crate-internal. The visibility annotations still matter for intra-crate encapsulation and for any future library extraction.
- Empty stub files (`git.rs`, `metrics.rs`, `services/*.rs`) suggest planned features. If these are tracked elsewhere, the dead-module finding can be downgraded to `note`, but they currently add noise to the module tree and will trigger `dead_code` warnings without `#[allow]` attributes.
- The config-not-loaded issue (#3 and #7) is the most impactful finding: users who customize `epic.toml` limits will see no effect on orchestrator behavior.
