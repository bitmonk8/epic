# U4-R1: agent/prompts / Correctness

## Summary

The prompt assembly module is largely straightforward string formatting with no complex control flow. Two minor correctness issues were identified: an empty verification-criteria list produces a malformed bullet, and the `build_fix_leaf` system prompt is slightly misleading about the prior execution's outcome.

## Findings

| # | Severity | Finding | Location | Suggestion |
|---|----------|---------|----------|------------|
| 1 | minor | When `verification_criteria` is empty, `join("\n- ")` returns `""`, so the rendered text becomes `"Verification criteria:\n- "` -- a dangling bullet with no content. Every other list in `format_context` guards against the empty case with an `if is_empty()` check, but criteria does not. | `prompts.rs:13` | Add an empty guard: `if ctx.task.verification_criteria.is_empty() { "None".into() } else { format!("- {}", ctx.task.verification_criteria.join("\n- ")) }` |
| 2 | minor | `build_fix_leaf` system prompt says "A previous execution of this task passed but failed verification." The word "passed" is misleading -- the execution itself may have reported success, but the phrasing conflates "execution passed" with the task outcome. If the executor reported `Failed` but verification also failed independently, this sentence is inaccurate. | `prompts.rs:169` | Reword to "A previous execution of this task has been completed, but it failed verification." to avoid implying the executor claimed success. |
| 3 | note | `build_checkpoint` does not guard against an empty `discoveries` slice. If called with `&[]`, the prompt will contain `"## Recent Discoveries\n\n"` with no items. This is not a logic error (the LLM can handle it), but it is inconsistent with the purpose of the checkpoint prompt, which should only be invoked when discoveries exist. | `prompts.rs:271-275` | Consider adding a `debug_assert!(!discoveries.is_empty())` or an early-return / guard so callers cannot misuse the function. |
| 4 | note | `build_design_recovery_subtasks` accepts a `strategy` parameter that is inserted verbatim into the prompt, yet the system prompt tells the LLM to "Choose one of two approaches: incremental or full." If the caller passes a strategy value that is neither "incremental" nor "full", the user message and system message will contradict each other. There is no validation. | `prompts.rs:290-328` | Either validate `strategy` against an enum/known set, or adjust the system prompt to say "The suggested recovery strategy is provided below; you may override it." |

## Notes

- The file has no `unsafe` code, no integer arithmetic susceptible to overflow, and no fallible operations (no `Result`/`unwrap`/`expect`). The primary risk surface is prompt quality rather than runtime panics.
- Finding 1 is unlikely to trigger in practice because `Task::new` callers typically supply at least one criterion, but the type system (`Vec<String>`) does not enforce this.
- All `format!` calls use named arguments that match local bindings; no positional-argument mismatch risk.
